# 모노레포 + 변경기반 CI 구조의 시스템 추상화

## STEP 0. 모드 판정

- **모드**: 외부 모드
- **근거**: 분석 대상이 사용자 회사의 코드·조직 구조이며, "지금 이 대화" 자체가 아님.

## STEP 1. 경계·환경·관찰자

- **경계**: 공유 타입 정의층 · 소비 표면들(앱·서비스) · 변경기반 빌드 조정자를 하나의 시스템으로 묶는다. 내부 비즈니스 로직·런타임 인프라·사용자 트래픽은 바깥.
  - 절단 근거: "타입 변경이 어떻게 전파·수렴하는가"를 보기 위한 목적이므로, 전파 경로를 품는 최소 범위로 자름.
- **환경 (시스템이 의존하는 바깥)**:
  1. 외부 API 수요(새 엔드포인트를 요구하는 제품 요구 흐름)
  2. 각 소비 표면을 소유하는 **복수의 인간 팀**(합의·할당·우선순위 조정의 외부 주체)
  3. 버전 관리 기반 시설(리비전 추적·머지 정책)
- **관찰자 위치**: 이 구조가 왜 특정 방식으로 굴러가는지(지연·병목·전파 비용)를 구조 수준에서 이해하려는 내부 설계자.

## STEP 1.5. 제약 (위반 불가)

1. **단일 진실 원천 강제**: 공유 타입 정의층 밖에서 독립적으로 재정의된 타입은 시스템 내 계약으로 인정되지 않음(컴파일 단계가 금지).
2. **상향 수정 불가**: 소비 표면은 공유 타입 정의층을 직접 덮어쓸 권한이 없고, 합의된 개정 채널로만 수정 가능(상위→하위 단방향).
3. **부분 빌드 규약**: 변경기반 조정자는 "직접 변경 + 의존 그래프상 영향권"만 빌드·테스트 대상으로 삼는다(전체 빌드 선택지를 닫음).

## STEP 2. 요소 (역할어)

| 요소 (역할어) | 유형 | 역할 정의 |
|---|---|---|
| 계약 정의층 | stock | 소비 표면들이 공유해야 할 구조적 약속을 누적·보관 |
| 소비 표면 | stock | 계약을 참조하여 자기 상태(코드·빌드 산출물)를 유지하는 다수의 종단 |
| 영향권 산출 조정자 | flow | 변경 신호를 받아 재빌드·재검증 대상 집합을 계산·파견 |
| 불일치 신호 | flow | 계약과 소비 표면 사이의 어긋남을 타입 오류 형태로 운반하는 전달물 |
| 적응 노동 | flow | 불일치 신호를 받은 소비 표면이 수행하는 수정·반영 작업 |

## STEP 3. 관계 — 피드백 루프

**L1 (강화 +): 계약 확장 → 불일치 파급 → 적응 압력 → 추가 계약 확장**
```
계약 정의층 →[개정 커밋, 즉시]→ 영향권 산출 조정자
영향권 산출 조정자 →[빌드·검증 파견, 턴내]→ 소비 표면
소비 표면 →[불일치 신호, 즉시]→ 적응 노동
적응 노동 →[반영 커밋, 지연]→ 계약 정의층 (후속 개정 수요 누적)
```
- 왜 강화인가: 한 번의 계약 개정이 다수 소비 표면에서 동시 오류를 일으키고, 이 오류들이 추가 보조 타입·어댑터 요구를 누적시켜 다음 개정 압력으로 되돌아온다. 개정→오류→어댑터→개정이 증폭 방향.

**L2 (상쇄 −): 영향권 산출 조정자가 전체 부하를 제한**
```
계약 정의층 →[변경 메타데이터, 즉시]→ 영향권 산출 조정자
영향권 산출 조정자 →[대상 축소, 즉시]→ 소비 표면 (전체 중 영향권만)
소비 표면 →[빌드 성공 비율, 턴내]→ 영향권 산출 조정자 (그래프 신뢰 유지)
```
- 왜 상쇄인가: 변경량이 커져도 조정자가 영향권을 잘라내 실행 부하를 제한하므로, 시스템의 검증 비용이 무한히 팽창하지 않고 일정 수준에서 억제된다.

**L3 (상쇄 −, 지연): 적응 노동의 팀별 분산이 확산 속도를 감쇠**
```
불일치 신호 →[할당, 지연]→ 각 소비 표면의 소유 팀
소유 팀 →[반영 커밋, 지연]→ 소비 표면 → 계약 정합성 복원
```
- 왜 상쇄·지연인가: 적응 노동이 단일 채널이 아니라 다수 팀의 일정에 분산되므로, 전파는 즉시지만 수렴은 팀 스케줄 지연만큼 천천히 일어나 계약 변경의 체감 속도를 완화한다.

## STEP 4. 기능 — 창발

**정합성 압력의 동기화된 분배**: 한 지점의 구조 변경이 시스템 전 표면에 *동일 타이밍의* 검증 의무로 분포되는 현상.

- 제거 테스트:
  - 영향권 산출 조정자를 빼면 → 변경은 있으나 "누가 어디를 다시 검증해야 하는지" 선택적 파견이 사라져, 분배가 모 아니면 도(전체 혹은 무검증)로 붕괴. 기능 소실.
  - 계약 정의층을 빼면 → 각 표면이 독립 정의로 회귀, 동기화된 압력 자체가 사라짐. 기능 소실.
- 비창발 대비:
  - 단순 합: "타입 시스템이 오류를 보고함" + "CI가 빌드함" + "팀이 코드를 고침"
  - 창발과의 차이: 단순 합은 각자 독립 사건인데, 이 구조에서는 *한 번의 개정이 다수 지점의 동기화된 수정 의무를 생성·분배*한다. 발생 주체가 특정 팀이 아니라 구조 자체.

## STEP 5. 목적 — 명목 vs 실질

- **명목 목적**: 공유 타입으로 계약을 단일화하여 앱·서비스 간 통합 오류를 컴파일 시점에 잡아낸다.
- **실질 목적**: 계약 변경 비용을 *변경 주체*(API 추가자)에서 *소비 주체들*(각 팀)에게 분산·이연하며, 그 분산 자체를 컴파일러·CI가 대행해 조직적 협상을 줄인다.
- **간격의 구조적 원인**: 계약 정의층은 수정 권한이 집중돼 있으나 수정 비용은 소비 표면 수에 비례하여 외부화된다. 권한과 비용이 같은 자리에 있지 않음.
- **간격의 유지 기제**: L1(강화)이 매 주기 이를 재생산. 계약 개정자는 즉각 적응 압력을 보지 않고(지연), 소비 측은 이미 쌓인 오류를 해결하는 선에서 움직여 "분산·이연이 당연"이라는 균형이 반복 강화된다. L3의 팀별 지연이 이 분산을 저항 없이 수용 가능하게 만들어 간격이 자연스러워 보이게 한다.

## STEP 6. 취약점 · 레버리지 (짝)

- **취약점**: 계약 정의층의 *확장성 변경*(선택 필드 추가 등)과 *호환성 파괴 변경*(서명 변경)이 동일 채널·동일 파급 범위를 쓰는 지점. 환경 변이(요구의 긴급도·범위)가 시스템 대응 변이 폭(단일 파급 경로)을 초과하면, 작은 개정과 큰 개정이 같은 신호 강도로 다수 팀을 동시에 멈춰 세운다. Requisite variety 결핍.
- **레버리지**: *변경의 종류*를 계약 정의층에서 구분 가능하게 만드는 지점(예: 영향권 산출 조정자가 변경 유형까지 읽어 파견 범위·시점을 달리 계산하게 하는 축). 여기에 작은 비틀림을 주면 L1 강화 극성이 일부 상쇄로 뒤집히고(소규모 개정이 대규모 개정과 같은 비용을 발생시키지 않음), 제약 2("상향 수정 불가")의 강도는 유지한 채 제약 3(변경기반 규약)의 해상도가 재설정된다.

## 마지막 한 줄
