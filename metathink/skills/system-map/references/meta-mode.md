# 메타 모드 가이드 — 자기 관찰의 함정과 절단

메타 모드는 분석 대상이 *지금 이 대화/작업 자체*일 때다. 관찰자가 시스템 바깥에 있지 않다는 사실이 모든 섹션의 해석을 바꾼다.

## 왜 어려운가

1. **재귀성**: 추상화 행위 자체가 시스템의 한 작동이다. 무한 재귀 가능.
2. **경계 모호**: "이 대화"의 끝은 어디인가 — 어시스턴트 가중치? 호스트 런타임? 사용자 디바이스? 깔끔한 답이 없다.
3. **관찰자 효과**: 추상화 결과가 다음 턴의 대화 자체를 바꾼다. 즉 관찰이 대상을 변형한다.

## 경계 설정 원칙

메타 모드에서 경계는 **분석 목적에 맞춘 절단**이다. 객관 사실이 아니다. 그 사실을 먼저 선언하라.

기본 후보:
- **대화 표면 절단**: 사용자 메시지·어시스턴트 응답·툴 결과만 안으로. 런타임·모델 가중치·사용자 환경은 밖.
- **도구층 포함 절단**: 위에 더해 호출된 MCP·스킬·메모리·훅을 안으로. 런타임과 가중치는 밖.
- **세션 환경 전체 절단**: 위에 더해 호스트 런타임·설정 파일까지 안으로. 사용자 디바이스·네트워크는 밖.

목적에 따라 하나를 *선택*한다. 어느 것이 맞느냐가 아니라 지금 무엇을 보려 하느냐다.

## 자기 지시 처리 규약

1차 재귀만 인정한다:
- 인정: "이 응답은 시스템의 한 요소(변환 계층이 방금 생산한 출력물)이다"
- 중단: "이 응답을 추상화하는 행위 자체"를 다시 추상화하는 단계는 *요청받지 않는 한* 진행하지 않는다.

이 규약이 있어야 분석이 유한한 분량에서 종결된다.

## 요소 명명 — 메타 모드 예시

| 피해야 할 이름 (구체어) | 권장 이름 (역할어) |
|---|---|
| "Claude", "어시스턴트" | "변환 계층" 또는 "응답 생성자" |
| "사용자" | "의도 발신원" |
| "스킬 레지스트리" | "행위 양식 라이브러리" |
| "메모리 파일" | "누적 기억층" |
| "MCP 서버" | "외부 기능 접점" |
| "훅" | "사전 개입 규칙" |

이름이 역할을 가리키면 다른 유사 시스템(다른 에이전트, 다른 조직)과 구조 비교가 가능해진다. 고유명사는 그것을 막는다.

## 관찰자 효과 — 출력에 반영하는 방법

메타 추상화의 결과는 다음 턴부터 대화의 작동을 *변화시킨다*. 이 사실을 숨기지 말고 STEP 7에서 한 줄로 인정한다. 예:

> "이 추상화는 의도 발신원이 앞으로 변환 계층에 무엇을 기대하는지 재조정한다 — 이는 시스템의 다음 상태에 입력된다."

이 한 줄이 있으면 관찰과 개입의 경계를 독자가 판단할 수 있다.

## 시기상조 판정

다음 중 하나라도 해당하면 메타 추상화를 중단하고 선행 작업을 제안한다:

- 대화가 너무 짧아 요소가 3개 이하로밖에 뽑히지 않는다
- 피드백 루프로 부를 만한 상호작용이 아직 한 번도 없었다
- 사용자의 실제 의도가 메타 분석이 아니라 도메인 답을 원하는 것으로 보인다

이 경우 "아직 구조가 드러나지 않았음" 한 줄로 명시하고 출력한다.
