# 프리랜서 디자이너를 위한 SaaS 플랫폼 — 기능 구조 및 MVP 정리

좋은 아이디어입니다. 포트폴리오 관리, 클라이언트 소통, 견적서 발행 — 이 세 가지는 프리랜서 디자이너가 매일 겪는 핵심 페인포인트이고, 이걸 하나로 묶는 플랫폼은 충분히 가치가 있습니다. 아래에 전체적인 기능 구조를 정리하고, MVP로 어디까지 잡을지 제안드리겠습니다.

---

## 1. 전체 기능 구조 (Feature Map)

크게 **4개 도메인**으로 나눌 수 있습니다.

### A. 포트폴리오 관리
| 기능 | 설명 |
|------|------|
| 프로젝트 등록 | 이미지/영상 업로드, 프로젝트 설명, 태그, 카테고리 분류 |
| 포트폴리오 페이지 | 커스터마이징 가능한 공개 포트폴리오 페이지 (개인 도메인 연결 가능) |
| 템플릿 시스템 | 디자인 분야별 포트폴리오 템플릿 제공 |
| 비공개 포트폴리오 | 특정 클라이언트에게만 공유하는 비밀 링크 |
| 분석 대시보드 | 포트폴리오 조회수, 어떤 프로젝트가 관심을 많이 받는지 추적 |

### B. 클라이언트 소통
| 기능 | 설명 |
|------|------|
| 클라이언트 CRM | 클라이언트 정보 관리 (회사명, 담당자, 연락처, 프로젝트 히스토리) |
| 메시지/채팅 | 플랫폼 내 클라이언트와 메시지 주고받기 |
| 파일 공유 | 시안/디자인 파일 공유 및 버전 관리 |
| 피드백 수집 | 디자인 시안 위에 코멘트/마크업 기능 |
| 프로젝트 타임라인 | 진행 상황 공유, 마일스톤 트래킹 |
| 알림 시스템 | 이메일/푸시 알림으로 주요 업데이트 전달 |

### C. 견적서 및 비즈니스
| 기능 | 설명 |
|------|------|
| 견적서 생성 | 항목별 견적 작성, 템플릿 저장, PDF 내보내기 |
| 계약서 관리 | 간단한 계약서 템플릿, 전자서명 연동 |
| 인보이스 발행 | 견적서 기반 인보이스 자동 생성 |
| 결제 연동 | 토스페이먼츠/스트라이프 등 결제 수단 연결 |
| 수익 대시보드 | 월별/분기별 수익 추적, 미수금 관리 |

### D. 공통/인프라
| 기능 | 설명 |
|------|------|
| 사용자 인증 | 회원가입, 로그인, 소셜 로그인 |
| 알림 센터 | 통합 알림 관리 |
| 설정 | 프로필, 결제 정보, 알림 설정 |
| 모바일 대응 | 반응형 웹 (모바일 앱은 이후 고려) |

## 2. MVP 범위 제안

MVP의 핵심 원칙: **"프리랜서 디자이너가 새 클라이언트를 만났을 때, 포트폴리오 보여주고 → 소통하고 → 견적서 보내는 흐름을 하나의 플랫폼에서 끝낼 수 있는가?"**

### MVP에 포함 (Phase 1 — 출시 목표: 8~12주)

**포트폴리오**
- 프로젝트 등록 (이미지 업로드 + 설명)
- 기본 포트폴리오 공개 페이지 (1~2개 템플릿)
- 비공개 링크 공유

**클라이언트 소통**
- 클라이언트 기본 정보 관리 (이름, 연락처, 메모)
- 프로젝트별 메시지 스레드 (이메일 알림 포함)
- 파일 첨부 기능

**견적서**
- 견적서 작성 (항목, 수량, 단가)
- PDF 내보내기
- 견적서 링크 공유 (클라이언트가 온라인에서 확인)

**공통**
- 이메일 기반 회원가입/로그인
- 기본 대시보드 (내 프로젝트, 최근 메시지, 견적서 현황)

### MVP 이후 (Phase 2, 3...)

| Phase | 기능 |
|-------|------|
| Phase 2 | 디자인 피드백(마크업), 인보이스 발행, 결제 연동, 포트폴리오 커스텀 도메인 |
| Phase 3 | 계약서 템플릿 + 전자서명, 수익 대시보드, 고급 분석 |
| Phase 4 | 모바일 앱, API 연동(노션/슬랙/구글캘린더), 팀 기능 |

## 3. 사용자 흐름 (핵심 시나리오)

MVP에서 검증해야 할 핵심 사용자 흐름은 다음과 같습니다:

```
1. 디자이너가 회원가입 → 포트폴리오에 프로젝트 3~5개 등록
2. 잠재 클라이언트에게 포트폴리오 링크 공유
3. 클라이언트가 관심을 보이면 → 플랫폼에서 대화 시작
4. 요구사항 논의 후 → 견적서 작성 및 전달
5. 클라이언트가 견적서 확인 → 프로젝트 시작
```

이 흐름이 매끄럽게 동작하면 MVP는 성공입니다.

## 4. 기술 스택 방향 제안

아직 기술 스택을 정하지 않았다고 하셨으므로, MVP 범위에 맞는 현실적인 선택지를 제안합니다:

| 레이어 | 추천 옵션 | 이유 |
|--------|-----------|------|
| 프론트엔드 | Next.js (React) | 포트폴리오 페이지 SEO 필요 (SSR), 풍부한 생태계 |
| 백엔드 | Next.js API Routes 또는 별도 Node.js 서버 | MVP에서는 풀스택 Next.js로 시작 가능 |
| DB | PostgreSQL (Supabase 또는 Neon) | 관계형 데이터 구조에 적합, 호스팅 서비스로 운영 부담 최소화 |
| 파일 스토리지 | AWS S3 또는 Cloudflare R2 | 이미지/디자인 파일 저장 |
| 인증 | NextAuth.js 또는 Supabase Auth | 빠른 구현 |
| 배포 | Vercel | Next.js와 자연스러운 조합, 무료 티어로 시작 가능 |
| 이메일 | Resend 또는 SendGrid | 알림 메일 발송 |

핵심은 **MVP 단계에서는 인프라에 시간을 적게 쓰고, 제품 자체에 집중하는 것**입니다.

## 5. 경쟁 환경 및 차별화 포인트

기존에 유사한 서비스들이 있습니다:

- **Behance/Dribbble** — 포트폴리오만 특화, 클라이언트 소통/견적 기능 없음
- **HoneyBook/Dubsado** — 클라이언트 관리 + 견적에 특화, 포트폴리오 없음 (주로 미국 시장)
- **노션/구글 시트 조합** — 많은 프리랜서가 이렇게 쓰지만 번거로움

차별화 가능한 방향:
1. **"디자이너 전용"이라는 특화** — 디자인 시안 리뷰/마크업 같은 기능은 일반 비즈니스 도구에 없음
2. **한국 시장 최적화** — 한국 세금 체계에 맞는 견적서/세금계산서, 원화 기반 정산
3. **포트폴리오에서 비즈니스까지 이어지는 흐름** — "보여주고 → 대화하고 → 계약하는" 일관된 경험

## 6. MVP 검증 지표 (KPI)

출시 후 무엇을 측정할지 미리 정해두면 좋습니다:

- **활성 사용자 수**: 주 1회 이상 로그인하는 디자이너 수
- **포트폴리오 생성 완료율**: 가입 후 포트폴리오를 실제로 만든 비율
- **견적서 발송 건수**: 실제로 견적서를 만들어서 클라이언트에게 보낸 횟수
- **클라이언트 초대 수**: 디자이너가 클라이언트를 플랫폼으로 초대한 횟수
- **리텐션**: 4주 후에도 계속 사용하는 비율

## 7. 다음 단계 제안

1. **와이어프레임/화면 설계**: 위 핵심 시나리오를 기반으로 주요 화면 5~7개 스케치
2. **기술 스택 확정**: 위 제안을 기반으로 팀 역량에 맞게 결정
3. **데이터 모델 설계**: User, Project, Portfolio, Client, Message, Estimate 등 핵심 엔티티 정의
4. **1주차 스프린트**: 인증 + 프로젝트 등록 + 기본 포트폴리오 페이지부터 시작

---

정리하면, **포트폴리오 + 간단한 클라이언트 소통 + 견적서** 이 세 축의 최소 기능을 먼저 만들고, 실제 프리랜서 디자이너 5~10명에게 써보게 하면서 피드백을 받는 것이 가장 효과적인 접근입니다. 어떤 부분을 더 깊이 파고 싶으신지 알려주시면 이어서 도와드리겠습니다.
