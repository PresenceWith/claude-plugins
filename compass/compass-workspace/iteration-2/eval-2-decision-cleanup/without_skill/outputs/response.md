# 내부 도구 프로젝트 — 의사결정 현황 정리

말씀하신 내용을 바탕으로 현재 상태를 정리했습니다.

---

## 확정된 사항

| 항목 | 결정 내용 |
|------|-----------|
| 사용자 인증 | OAuth 방식으로 확정 |
| 프론트엔드 | React 확정 |

---

## 논의 중 (선택지가 좁혀진 상태)

| 항목 | 후보 | 비고 |
|------|------|------|
| 데이터베이스 | PostgreSQL vs MongoDB | 두 가지 중 하나로 좁혀진 상태. 데이터 구조(정형/비정형), 트랜잭션 요구사항, 팀 경험 등을 기준으로 판단 필요 |
| API 구조 | REST vs GraphQL | 아직 결정 못 함. 클라이언트 요구사항 복잡도, 프론트-백 간 데이터 교환 패턴 등을 고려해서 결정 필요 |

---

## 미착수 (아직 논의 시작 안 됨)

| 항목 | 현재 상태 |
|------|-----------|
| 배포 전략 | 논의 자체가 아직 시작되지 않음 |
| 모니터링 | 봐야 한다는 인식은 있으나, 구체적 논의 없음 |
| CI/CD 파이프라인 | 정해야 한다는 인식은 있으나, 구체적 논의 없음 |

---

## 다음 단계 제안

우선순위를 정해서 하나씩 결정해 나가는 것을 추천합니다.

1. **DB 선택** — 백엔드 구현에 직접적인 영향을 주므로 가장 먼저 결정하는 것이 좋습니다. PostgreSQL은 관계형 데이터와 트랜잭션이 중요할 때, MongoDB는 스키마가 자주 바뀌거나 비정형 데이터가 많을 때 유리합니다.
2. **API 구조** — DB와 함께 백엔드 설계의 핵심입니다. 화면이 다양한 데이터를 조합해서 보여줘야 하면 GraphQL이 유리하고, 단순한 CRUD 위주라면 REST가 구현과 유지보수에 더 편합니다.
3. **CI/CD 파이프라인** — 개발 초기부터 세팅해두면 이후 작업 효율이 크게 올라갑니다.
4. **배포 전략** — CI/CD와 연계해서 같이 논의하면 효율적입니다.
5. **모니터링** — 배포 전략이 정해진 후에 그에 맞춰 설계하면 됩니다.

---

요구사항이 계속 바뀌는 상황이라면, 각 의사결정에 대해 "확정 / 논의 중 / 미착수" 상태를 문서로 관리하면서 변경 이력을 남겨두면 혼란을 줄일 수 있습니다.
