## 확정된 것

- **사용자 인증: OAuth** — 이건 결정된 거니까 더 이상 논의 불필요합니다.
- **프론트엔드: React** — 마찬가지로 확정. 이 두 가지는 앞으로 다른 결정의 기준점이 됩니다.

## 논의 중인 것 (선택지가 있는 상태)

- **데이터베이스: PostgreSQL vs MongoDB** — 이 결정이 API 구조에도 영향을 줍니다. 관계형 데이터가 많으면 PostgreSQL이 유리하고, 스키마가 유동적이면 MongoDB가 나을 수 있습니다.
- **API 구조: REST vs GraphQL** — 프론트엔드(React)와 DB 선택 모두에 연결됩니다. React와는 둘 다 잘 맞지만, DB 구조가 정해지면 API 설계 방향도 좁혀집니다.

## 아직 논의 시작 전

- **배포 전략** — 아직 안 건드렸다고 하셨죠.
- **CI/CD 파이프라인** — 배포 전략과 직접 연결되므로, 배포 전략이 어느 정도 윤곽이 잡힌 뒤에 같이 논의하는 게 효율적입니다.
- **모니터링** — 배포가 정해진 뒤에 구체화하면 됩니다. 지금 당장 결정할 필요는 없습니다.

**미결정 항목이 5개**인데, 전부 한꺼번에 정하려고 하면 오히려 혼란만 커집니다. 의존 관계를 보면 이런 순서가 자연스럽습니다:

> **DB 선택 → API 구조 결정 → (이후) 배포/CI/CD/모니터링**

DB를 먼저 정하는 이유는, API 설계가 데이터 모델에 크게 좌우되기 때문입니다. 반대로 배포 쪽 세 가지(배포 전략, CI/CD, 모니터링)는 앱의 기본 구조가 잡힌 뒤에 논의해도 늦지 않습니다.

**DB 선택부터 시작할까요?** PostgreSQL과 MongoDB 사이에서 고민 중이라고 하셨는데, 팀 내부 도구의 데이터 특성(관계가 복잡한지, 스키마가 자주 바뀌는지)을 기준으로 잡으면 빠르게 좁힐 수 있습니다.
