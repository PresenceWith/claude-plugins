# 피트니스 트래킹 앱 기획 정리

솔직히 말씀드리면, 지금 나열하신 기능들을 한꺼번에 다 만들려고 하면 높은 확률로 프로젝트가 산으로 갑니다. 하나씩 정리해 드릴게요.

---

## 현재 상황 진단

나열하신 기능을 정리하면 이렇습니다:

| 기능 | 복잡도 | 비고 |
|------|--------|------|
| 운동 기록 | 중 | 핵심 기능 |
| 식단 관리 | 중~상 | 음식 DB 필요 |
| 소셜 기능 | 상 | 피드, 팔로우, 알림 등 |
| 챌린지 | 중 | 소셜과 연동 필요 |
| 코칭 AI | 상 | AI 모델 연동/튜닝 |
| 웨어러블 연동 | 상 | 기기별 SDK 대응 |
| 결제 | 중~상 | 인앱결제/구독 모델 |
| 오프라인 모드 | 상 | 동기화 로직 복잡 |
| 다국어 | 중 | 초기 설계 중요 |

이건 사실상 **9개의 서로 다른 프로젝트**에 가깝습니다. 스타트업 팀이라면 이걸 동시에 하면 어느 것 하나 제대로 못 만듭니다.

---

## 추천 접근법: 3단계 페이즈 전략

### Phase 1 - 핵심 루프 (MVP, 2~3개월)

**목표: "운동을 기록하고 돌아보는 경험"이 제대로 동작하는 앱**

만들어야 할 것:
- **운동 기록** (세트/횟수/시간/거리 등 기본 로깅)
- **운동 히스토리 & 통계** (주간/월간 요약, 간단한 차트)
- **사용자 인증** (회원가입/로그인)
- **다국어 기반 설계** (i18n 프레임워크는 처음부터 깔아두되, 실제 번역은 한국어+영어만)

왜 이것부터인가:
- 피트니스 앱의 본질은 "기록"입니다. 기록이 불편하면 아무리 소셜이 화려해도 쓰지 않습니다.
- 다국어는 나중에 넣으면 코드 전체를 뜯어고쳐야 해서, 구조만 Phase 1에서 잡아두는 겁니다.

이 단계에서 **하지 말아야 할 것**:
- AI 코칭 (아직 사용자 데이터가 없음)
- 소셜 기능 (사용자가 없는데 소셜은 의미 없음)
- 오프라인 모드 (동기화 복잡도가 초기 개발 속도를 죽임)

### Phase 2 - 습관 고착화 (출시 후 2~3개월)

**목표: 유저가 매일 돌아오게 만드는 장치**

추가할 것:
- **식단 관리** (사진 기반 간편 입력, 기본 음식 DB)
- **챌린지 시스템** (개인 목표 기반, 예: "이번 주 3회 운동")
- **웨어러블 연동** (Apple Health / Google Fit 우선 - 기기 직접 연동보다 훨씬 쉬움)
- **오프라인 모드** (기록은 로컬 저장, 온라인 복귀 시 동기화)

왜 이 순서인가:
- Phase 1에서 모은 사용자 데이터와 피드백으로, 실제로 뭘 원하는지 알 수 있습니다.
- Apple Health/Google Fit 연동은 개별 웨어러블 SDK보다 투자 대비 커버리지가 훨씬 큽니다.
- 오프라인 모드는 이 시점에서 "운동 중 지하철/지하 헬스장에서 기록이 안 된다"는 실제 불만이 나올 때 넣는 게 맞습니다.

### Phase 3 - 성장 엔진 (Phase 2 이후 3~4개월)

**목표: 유저 획득과 수익화**

추가할 것:
- **소셜 기능** (피드, 친구 운동 현황, 응원/댓글)
- **코칭 AI** (축적된 데이터 기반 개인화 추천)
- **결제/구독** (프리미엄 기능 구분)
- **다국어 확장** (일본어, 중국어 등 타겟 시장 추가)

왜 마지막인가:
- 소셜은 유저 풀이 있어야 의미가 있습니다. Phase 1~2에서 유저를 모은 뒤에 넣어야 합니다.
- AI 코칭은 데이터가 쌓여야 제대로 된 추천이 가능합니다. 빈약한 데이터로 AI를 붙이면 "멍청한 AI"가 되어 오히려 역효과입니다.
- 결제는 무료 유저가 "이 앱 좋은데 더 쓰고 싶다"는 느낌이 들 때 전환율이 높습니다.

## 기술 스택 관련 조언

아직 기술 선택을 언급하지 않으셨지만, 위 요구사항을 고려하면:

- **크로스 플랫폼**: Flutter 또는 React Native (양 플랫폼 동시 대응 필수라면)
- **백엔드**: 초기에는 Firebase/Supabase 같은 BaaS로 빠르게 시작하고, 유저 늘면 자체 서버로 전환
- **오프라인 동기화**: Phase 2에서 넣을 때 SQLite + 동기화 큐 패턴 고려
- **다국어**: 처음부터 하드코딩된 문자열 없이, i18n 라이브러리(flutter_localizations, react-intl 등) 적용
