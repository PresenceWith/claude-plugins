# 내부 도구 프로젝트 - 의사결정 현황 정리

지금까지 나온 내용을 **결정 완료 / 논의 중 / 미착수** 세 가지로 분류해서 정리했습니다.

---

## 결정 완료

| 항목 | 결정 사항 | 비고 |
|------|-----------|------|
| **사용자 인증** | OAuth 방식 채택 | 확정 |
| **프론트엔드 프레임워크** | React | 확정 |

---

## 논의 중 (결정 필요)

| 항목 | 선택지 | 고려 사항 |
|------|--------|-----------|
| **데이터베이스** | PostgreSQL vs MongoDB | 데이터 구조(관계형 vs 문서형), 확장성, 팀 숙련도 등을 기준으로 판단 필요 |
| **API 구조** | REST vs GraphQL | 클라이언트 데이터 요구 패턴, 학습 비용, 캐싱 전략 등 고려 필요 |
| **모니터링** | 미정 | 도구 선정 및 범위(로그, 메트릭, 알림 등) 논의 필요 |
| **CI/CD 파이프라인** | 미정 | 도구 선정 및 워크플로우 설계 필요 |

---

## 미착수

| 항목 | 상태 |
|------|------|
| **배포 전략** | 아직 논의 시작 전 |

---

## 권장 우선순위

다음 순서로 결정을 진행하면 후속 작업의 의존성 문제를 줄일 수 있습니다.

1. **DB 선정** - API 설계와 데이터 모델링의 전제가 되므로 가장 먼저 확정
2. **API 구조 (REST vs GraphQL)** - DB가 정해지면 API 설계 방향도 잡기 수월함
3. **CI/CD 파이프라인** - 개발 초기부터 세팅해두면 이후 생산성에 직접 영향
4. **배포 전략** - CI/CD와 연결되므로 함께 또는 바로 뒤이어 논의
5. **모니터링** - 배포 환경이 정해진 후 구체적인 설계 가능

---

## 한눈에 보기

```
확정:  인증(OAuth) ---- 프론트(React)
         |                  |
논의중:  DB(Postgres/Mongo) -- API(REST/GraphQL)
         |                  |
미정:  CI/CD ----------- 모니터링
         |
미착수:  배포 전략
```

이 정리를 기준으로 팀 회의에서 하나씩 확정해 나가면 좋겠습니다. 특히 DB와 API 구조는 다른 항목에 영향을 많이 주는 핵심 결정이므로 우선적으로 논의하는 것을 추천합니다.
