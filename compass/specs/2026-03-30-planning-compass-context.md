# Planning Compass — 설계 컨텍스트 & 결정 로그

> 설계 문서(design.md)의 배경, 동기, 설계 과정에서 내린 결정들과 그 근거를 기록한다.
> 나중에 "왜 이렇게 됐지?"를 이해하기 위한 문서.

## 1. 배경 — 이 프로젝트가 시작된 이유

### 사용자의 원래 경험

Claude Code로 기획을 할 때, 대화가 진행될수록 다음과 같은 현상이 발생한다:

> "더 고도화된 내용을 받고, 대화를 하면서 옵션이나 고려 요소가 늘어나고, 알지 못하는 정보나 생각하지 못했던 사항들이 늘어나면서 점점 대화의 주도권이나 의사결정력, 집중력 등이 낮아집니다."

이것은 단일 원인의 문제가 아니라, **텍스트 기반 선형 대화가 비선형적인 기획 작업에 구조적으로 부적합**하다는 시스템적 문제다. 4가지 근본 원인(정보 표현의 한계, 상태 추적의 부재, 정보 비대칭, 결정 컨텍스트의 분산) 모두가 해당된다고 사용자가 확인했다.

### 사용자의 원래 아이디어

> "핵심을 추상화해서 도식화하여 어떤 요소가 있고 관계나 순서가 어떻게 되고 어떤 기능을 하는지 등을 보여주는 skill을 만들면 어떨까 싶습니다. 이외에도 메타적인 방식으로 해결 방안을 다양하게 모색해주세요."

두 가지 요청이 담겨있다:
1. **시각화 skill** — 핵심 요소, 관계, 순서, 기능을 도식화
2. **메타적 해결 방안** — 시각화 외의 다른 접근도 탐색

### 핵심 전제 조건

- **superpowers와 독립적인 완결성을 갖는 skill** — 기존 superpowers 플러그인에 의존하지 않는 독립 플러그인으로 배포
- 사용자는 Claude Code 환경에서 기획/브레인스토밍 작업을 빈번하게 수행

## 2. 설계 진화 과정

### Phase 1: 문제 분석

순차적 사고(sequential-thinking)로 문제를 분석한 결과, 5가지 근본 원인을 도출:
- A. 정보 표현의 한계
- B. 상태 추적의 부재
- C. 정보 비대칭
- D. 결정 컨텍스트의 분산

사용자는 4가지 모두 해당된다고 응답 → **단일 원인 해결이 아닌 시스템적 접근 필요**라는 결론.

### Phase 2: 해결 방향 탐색

6가지 메타적 해결 방향을 도출:
1. 시각적 추상화 (Visualization)
2. 구조적 대화 관리 (Structured Dialogue)
3. 결정 대시보드 (Decision Dashboard)
4. 주도권 반전 (Agency Inversion)
5. 점진적 복잡성 노출 (Progressive Disclosure)
6. 요약/체크포인트 시스템 (Checkpoint System)

추가로 4가지 더:
7. 인지 부하 모니터링
8. 역할 분리
9. 결정 저널
10. 탐색 예산

### Phase 3: 시각화 형태 확정

사용자 선택:
- 개입 방식: **하이브리드** (자동 체크포인트 + 수동 요청)
- 시각화 형태: **적응형** (기획 단계/목적에 따라 형태 전환)
- 추가 요청: 플로우차트, 타임라인 등 모든 형태를 리스트업하고 최적 사용 시점을 미리 매핑

→ 10가지 시각화 형태를 5개 기획 단계별로 카탈로그화.

### Phase 4: 접근법 결정

3가지 접근법을 제안:
- **A. 시각화 중심** — 원래 아이디어에 충실하지만 대화 구조를 안 바꿈
- **B. 프로세스 중심** — 근본을 공략하지만 시각적 힘 부족
- **C. 통합형 (Dual-Layer)** — 두 레이어를 결합, MVP 전략으로 단계적 구현

사용자 선택: **C. 통합형**

## 3. 핵심 설계 결정 로그

### 결정 1: 프로세스 레이어를 "고정 단계"에서 "진단 기반"으로 전환

**배경:** 사용자가 "목적이나 상황에 따라서 프로세스를 따라가기 어려울 때가 있을 것 같다"고 우려. 시각화는 맥락 독립적이지만, 프로세스가 고정 단계면 실제 대화와 충돌.

**결정:** 프로세스도 시각화와 동일한 패턴(상황 진단 → 적절한 개입 선택)으로 작동시킴.

**결과:** "별개의 두 레이어" → "통합 진단 엔진"으로 아키텍처 변경. 진단 엔진이 상황을 분석하고, 두 가지 출력(시각화 + 프로세스 개입)을 내는 구조.

**핵심 인사이트:** 프로세스를 "규칙"이 아니라 "배경 원칙(항상 작동) + 진단 기반 개입(상황 적응)"의 이중 구조로 분리하면 범용적으로 작동.

### 결정 2: 상태 파일 업데이트를 이중 트랙으로

**배경:** 자체 리뷰 #3에서 "매 응답마다 파일 업데이트는 비현실적"이라는 지적. 토큰 비용, 지연, 흐름 방해.

**결정:** 경량 트랙(컨텍스트 내 추적, 매 응답, 파일 I/O 없음) + 동기화 트랙(체크포인트 시에만 파일 쓰기).

**근거:** 트리거 판단에는 파일이 필요 없다(AI가 내부적으로 "미결정 몇 개쯤 됐지" 추적 가능). 파일은 진단과 시각화에 필요할 때만 동기화하면 충분.

### 결정 3: 상태 파일을 Single Source of Truth로

**배경:** "정보 비대칭", "결정 컨텍스트 분산", "상태 추적 부재"라는 3가지 근본 원인이 모두 "구조화된 상태 저장소의 부재"에서 비롯.

**결정:** `.compass/state.md`에 목표, 요소, 관계, 결정/미결정, 현재 논의를 기록. 이 파일이 진단의 입력이자 시각화의 입력.

**결과:** 하나의 데이터 소스가 진단 → 프로세스 개입, 진단 → 시각화 두 경로 모두에 쓰임. 일관성 보장.

### 결정 4: "프로세스 먼저, 시각화는 선택적"

**배경:** 자체 리뷰 #2에서 "시각화 + 개입이 동시에 쏟아지면 더 복잡해진다"는 지적.

**결정:** 진단 시 먼저 텍스트로 가볍게 정리(프로세스 개입), 사용자가 원할 때만 시각화로 확대.

**근거:** 도구 투명성 원칙. 프로세스 개입은 가볍고 자연스러운 반면, 시각화는 브라우저 전환이 필요한 무거운 작업. 항상 가벼운 것부터.

### 결정 5: 독립 플러그인으로 배포

**배경:** 사용자가 "superpowers와 독립적인 완결성을 갖는 skill"을 명시적으로 요청.

**결정:** 별도 패키지(`planning-compass/`)로 만들어 독립 설치 가능하게.

**영향:** 시각화 렌더링을 superpowers의 visual companion에 의존할 수 없으므로, 자체 렌더링 전략 필요 → 3단계 fallback 전략(chrome MCP → 자체 서버 → HTML open).

## 4. 자체 리뷰 히스토리

### 리뷰 #1 (초기 카탈로그 이후)

**발견한 핵심 이슈:**
1. 메타적 역설 — 이 기획 자체가 인지 과부하를 재현하고 있다
2. 형태(Form)에 치중, 기능(Function)이 빠져있다
3. 스코프가 MVP치고 과하다
4. 시각화 외 메타적 해결책 미탐색

**결과:** 이슈 2, 4가 접근법 제안 단계에서 해소됨 (통합형 접근으로 프로세스 개입이 메타적 해결책의 실체가 됨).

### 리뷰 #2 (통합 진단 엔진 이후)

**발견한 핵심 이슈:**
1. 진단의 기술적 실현 — LLM 직관만으로 부족 → 파일 기반 상태 추적으로 해소
2. 상호작용 모델 미정 → "프로세스 먼저, 시각화 선택적" 패턴으로 해소
3. 개입 카탈로그 누락 → 4개 추가 (8개로 확장)
4. 배경 원칙 적용 범위 → skill 세션 내로 확정

**결과:** 4개 모두 통합 설계에서 해소.

### 리뷰 #3 (통합 설계 이후)

**발견한 핵심 이슈:**
1. "매 응답마다 상태 파일 업데이트" 비현실적 → 이중 트랙으로 해소
2. 추상화 로직 부재 → 추출 가이드라인을 스펙에 추가
3. 시각화 렌더링 독립성 → 3단계 fallback 전략
4. 도구 투명성 원칙 → 설계 원칙으로 명시

**결과:** 4개 모두 스펙 문서에 반영.

## 5. 채택하지 않은 대안들

| 대안 | 왜 채택하지 않았는가 |
|------|---------------------|
| 시각화만 (접근법 A) | 대화 구조 자체가 안 바뀜. 보여주는 것만으로는 주도권 문제 해결 어려움. |
| 프로세스만 (접근법 B) | 시각적 힘 부족. "한 눈에 전체 그림"이라는 핵심 니즈를 충족 못함. |
| 고정 단계 프로세스 | 실제 기획 대화는 비선형적. 단계를 강제하면 오히려 흐름 방해. |
| 매 응답 파일 업데이트 | 토큰 비용, 지연, 대화 흐름 방해. 이중 트랙이 더 현실적. |
| superpowers 내장 skill | 사용자가 독립 플러그인을 명시적으로 요청. |
| 모든 시각화를 MVP에 | 10가지 한꺼번에 구현은 스코프 과다. 핵심 3가지부터 시작. |

## 6. 열린 질문 (구현 시 결정 필요)

1. **Skill 이름**: `planning-compass`가 최종 이름인가? "기획"에 국한되지 않는 더 범용적 이름이 나을 수 있음.
2. **시각화 HTML 디자인**: 다크 테마? 라이트 테마? 사용자 시스템 설정 따라감?
3. **다국어**: 한국어 중심? 영어? 시스템 언어 감지?
4. **시각화 라이브러리**: 순수 HTML/CSS/JS? D3.js? Mermaid? 의존성 최소화 vs 표현력.
5. **상태 파일 크기 관리**: 기획이 매우 커지면 state.md도 커짐. 아카이브/분할 전략 필요할 수 있음.
