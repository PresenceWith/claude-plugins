# Planning Compass — 설계 문서

> 기획 대화 중 인지 과부하를 해소하는 독립 Claude Code 플러그인

**관련 문서:** [설계 컨텍스트 & 결정 로그](./2026-03-30-planning-compass-context.md) — 이 설계에 이르기까지의 배경, 동기, 핵심 결정들과 근거, 채택하지 않은 대안, 3회의 자체 리뷰 히스토리.

## 1. 문제 정의

### 배경

Claude Code로 기획을 할 때, 대화를 통해 점점 고도화된 내용을 받게 되면서 옵션과 고려 요소가 기하급수적으로 늘어난다. 알지 못했던 정보와 생각하지 못했던 사항들이 계속 등장하면서, 사용자는 대화의 주도권, 의사결정력, 집중력을 점진적으로 잃는다.

이 문제의 본질은 **텍스트 기반 선형 대화가 비선형적인 기획 작업에 구조적으로 부적합**하다는 것이다. 사용자의 원래 아이디어는 "핵심을 추상화해서 도식화"하는 것이었으며, 동시에 "메타적인 방식으로 해결 방안을 다양하게 모색"하고자 했다.

### 핵심 증상

Claude Code로 기획할 때, 대화가 진행될수록 옵션/고려 요소가 늘어나고, 알지 못하는 정보와 생각하지 못했던 사항이 등장하면서 **대화의 주도권, 의사결정력, 집중력이 저하**된다.

### 근본 원인

| 원인 | 설명 |
|------|------|
| 정보 표현의 한계 | 텍스트 스트림으로는 복잡한 관계/구조를 파악하기 어려움 |
| 상태 추적의 부재 | 결정됨/미결정/논의중이 한 눈에 안 보임 |
| 정보 비대칭 | AI는 전체 컨텍스트를 갖지만 사용자는 기억에 의존 |
| 결정 컨텍스트의 분산 | 결정에 필요한 정보가 대화 여기저기에 흩어져 있음 |

### 본질

텍스트 기반 선형 대화는 비선형적인 기획 작업에 구조적으로 부적합하다. 이것은 개별 원인의 문제가 아니라 시스템적 문제이므로, 해결책도 시스템적이어야 한다.

## 2. 해결 방향

**통합형 접근 (Dual-Layer):** 프로세스 제어 + 시각화를 하나의 통합 진단 엔진에서 운용한다.

### 핵심 설계 원칙

1. **도구는 투명해야 한다** — skill의 존재 자체가 인지 부담이 되면 실패. 배경 원칙은 완전 투명, 진단은 거의 투명, 시각화만 명시적.
2. **고정 단계가 아닌 진단 기반** — 프로세스도 시각화와 동일한 패턴(상황 진단 → 적절한 개입 선택)으로 작동. 범용적, 맥락 독립적.
3. **프로세스 먼저, 시각화는 선택적** — 진단 시 텍스트로 가볍게 정리(프로세스 개입)한 뒤, 사용자가 원할 때만 시각화로 확대. 두 출력이 동시에 쏟아지지 않음.
4. **하이브리드 트리거** — 자동 체크포인트(진단 엔진이 임계값 감지) + 수동 요청(사용자가 "정리해줘" 등). 자동 개입은 거부/연기 가능.

## 3. 아키텍처

### 전체 데이터 흐름

```
[대화 흐름] & [수동 요청]
         ↓
[경량 인메모리 추적]  ← 매 응답마다 내부적으로 결정/미결정/주제 추적
         ↓
[트리거 판단]  ← 임계값 초과? 수동 요청?
         ↓ (트리거 시에만)
[상태 파일 동기화]  ← .compass/state.md에 현재 상태 기록
         ↓
[진단 엔진]  ← 상태 파일 정량 분석 + 대화 맥락 정성 판단
    ↙           ↘
[프로세스 개입]   [시각화 출력]
 (대화 내 텍스트)   (브라우저, 선택적)
```

### 상태 파일 업데이트 전략 — 이중 트랙

| 트랙 | 시점 | 방식 | 비용 |
|------|------|------|------|
| 경량 트랙 | 매 응답 | 컨텍스트 내 추적(in-context tracking). AI가 대화 컨텍스트 안에서 결정/미결정/주제 수를 계속 추적. 파일 I/O 없음. | 거의 없음 |
| 동기화 트랙 | 체크포인트 시 | 상태 파일에 전체 상태 기록 | 중간 (진단 시점에만) |

동기화 트리거: 경량 트랙에서 임계값 초과를 감지했을 때, 수동 요청 시, 주요 결정 후.

**흐름 명확화:** 경량 트랙(컨텍스트 내)이 트리거 판단을 담당 → 트리거 시 상태 파일 동기화 → 파일 기반 정량 분석 + 대화 맥락 정성 판단으로 진단 → 개입/시각화 출력.

### 상태 파일 스키마

경로: `.compass/state.md` (프로젝트 루트)

```markdown
# 목표
{원래 목표 기술}

# 요소
- 요소A [type: 아키텍처]
- 요소B [type: 데이터]
- 요소C [type: 기능]

# 관계
- 요소B →의존→ 요소A
- 요소C →포함→ 요소D

# 결정됨
- [항목]: [결정 내용] (이유)

# 미결정
- [항목]: [선택지들] (관련 요소)

# 현재 논의
- [활성 주제]

# 메타
- 생성: YYYY-MM-DD
- 최종 업데이트: YYYY-MM-DD HH:MM
```

### 추상화 로직 — 요소/관계 추출 가이드라인

대화에서 구조를 추출할 때의 규칙:

**요소 식별:**
- 사용자가 명시적으로 이름 붙인 명사/개념 = 요소 후보
- 반복적으로 언급되는 것 = 핵심 요소
- 원래 목표에 직접 연결되면 핵심, 아니면 세부

**관계 식별:**
- "A는 B를 사용한다/필요로 한다/포함한다" 구문 = 관계 후보
- 순서 관계: "A 이후에 B" = A →선행→ B
- 의존 관계: "A 없이 B는 불가" = B →의존→ A

**추상화 수준:**
- 시각화 시 상위 3-5개 요소만 1차 표시
- 하위 세부사항은 접힌 상태로 필요 시 펼침
- 한 요소의 하위 항목이 5개 이상이면 그룹화

## 4. 배경 원칙 (항상 적용)

skill이 활성화된 대화 세션 동안 AI의 모든 응답에 내재되는 규범. `/compass`로 호출된 시점부터 세션 종료까지 적용. 사용자가 의식하지 못할 정도로 투명하게 작동.

| 원칙 | 규칙 | 근거 |
|------|------|------|
| 정보 제한 | 새 정보/옵션은 한 번에 3-5개 이내. 초과 시 "더 보기"로 분리. | 선택지 과다 → 결정 피로 |
| 위치 명시 | 새 요소 추가 시, 기존 구조 내 위치와 관계를 명시. | 정보가 떠다니면 구조 파악 불가 |
| 결정 전 요약 | 결정 요구 전 반드시 관련 컨텍스트를 모아 요약. | 분산된 컨텍스트 → 잘못된 결정 |
| 필수/선택 분리 | "반드시 결정해야 할 것" vs "나중에 해도 됨" 명시적 구분. | 모든 것이 동등해 보이면 우선순위 소실 |

## 5. 진단 기반 개입 카탈로그

시각화 카탈로그와 동일한 패턴: 상태를 진단하고, 맞는 개입을 적용.

### MVP (Phase 1) — 4가지

| 상태 | 진단 조건 | 개입 | 동작 |
|------|-----------|------|------|
| 🔴 미결정 과다 | 미결정 항목 ≥ 5개 | 결정 정리 | 미결정 목록 제시, 우선순위 기반 하나씩 해결 제안 |
| 🟠 주제 분기 과다 | 병렬 논의 주제 ≥ 3개 | 초점 좁히기 | 열린 주제 나열, 어떤 것에 집중할지 사용자에게 선택권 |
| 🟡 정보 과부하 | 단일 응답에 새 개념 ≥ 5개 | 핵심 요약 | 가장 중요한 2-3개만 먼저 제시, 나머지는 접기 |
| 🔵 주도권 이동 | 사용자 응답이 수동적/짧아짐 | 선택권 되돌리기 | 열린 질문 대신 구체적 선택지, "어떤 것을 더 파볼까요?" |

### Phase 2 — 4가지 추가

| 상태 | 진단 조건 | 개입 | 동작 |
|------|-----------|------|------|
| 🕳️ 맥락 소실 | 초기 결정과 관련된 논의인데 참조 없음 | 맥락 상기 | 관련 결정/맥락을 상태 파일에서 인용 |
| 🎯 목표 이탈 | 논의가 원래 목표에서 벗어남 | 목표 재확인 | 원래 목표 상기 + 현재 논의와의 관계 제시 |
| ⚖️ 결정 갈등 | 선택지 간 트레이드오프 불명확 | 트레이드오프 명시화 | 기준별 비교 매트릭스 제시 |
| 🔁 진행 정체 | 동일 주제 3회 이상 반복 | 진행 정체 탈출 | 관점 전환 제안 또는 "일단 결정하고 진행" |

### 개입 UX

- **가벼운 개입:** 대화 흐름 내에서 자연스럽게 삽입 ("여기서 한 번 정리하면 — ")
- **본격 개입:** 명시적 체크포인트 ("지금까지의 상태를 정리합니다")
- **거부/연기:** 사용자가 "넘어가자" 또는 "나중에"로 개입을 건너뛸 수 있음

## 6. 시각화 카탈로그

진단 결과 + 상태 파일 내용 → 최적 형태 자동 선택. 사용자가 "다른 형태로 보여줘"로 전환 가능. 형태 선택은 AI가 자동 판단하되, 사용자 오버라이드 가능.

### MVP (Phase 1) — 3가지

| 형태 | 사용 맥락 | 트리거 |
|------|-----------|--------|
| 🕸️ 마인드맵 | 초기 탐색, 브레인스토밍, 발산 | 새 주제 시작, 탐색 단계 |
| 📍 상태 대시보드 | 결정 추적, 진행 상황, 체크포인트 | 자동 체크포인트, "지금 어디까지?" |
| 🗺️ 컨셉트 맵 | 관계 파악, 의존성, 시스템 설계 | 요소 간 관계 논의, 구조화 단계 |

### Phase 2 — 7가지 추가

| 형태 | 사용 맥락 | 트리거 |
|------|-----------|--------|
| 🔀 플로우차트 | 프로세스, 순서, 분기/조건 | "순서가 어떻게?", 프로세스 설계 |
| 📅 타임라인 | 시간 순서, 마일스톤, 로드맵 | 일정 논의, "언제까지?" |
| 📊 비교 매트릭스 | 옵션 비교, 트레이드오프, 장단점 | "A vs B", 기술 선택 |
| 🎯 우선순위 매트릭스 | 중요도/긴급도, 스코프, MoSCoW | "뭐부터 하지?", 스코프 조정 |
| 🌳 계층 구조 | 분류, 카테고리, 상위/하위 관계 | 컴포넌트 구조, 분류 체계 |
| 🔄 상태 머신 | 상태 전이, 이벤트 기반, 라이프사이클 | 상태 변화 논의, 이벤트 흐름 |
| ⭕ 벤 다이어그램 | 중복, 교집합, 범위 구분 | "이것과 저것의 차이/공통점" |

### 렌더링 방식

시각화의 독립적 렌더링을 위해 3가지 전략 (우선순위 순):

1. **claude-in-chrome MCP 활용** — 사용자 환경에 이미 있으면 활용. 별도 서버 불필요.
2. **자체 경량 서버** — 플러그인에 Node.js 기반 경량 서버 포함. 단일 포트에서 최신 HTML 서빙.
3. **단일 HTML + open** — 서버 없이 HTML 파일을 생성하여 브라우저로 열기. 최소 의존성 fallback.

**렌더링 선택 로직:** skill 최초 활성화 시 환경을 탐지한다. claude-in-chrome MCP 도구가 사용 가능하면 전략 1, 아니면 Node.js가 있으면 전략 2, 그것도 없으면 전략 3. 사용자가 명시적으로 전략을 지정할 수도 있다.

## 7. 상호작용 모델

### 자동 개입 시

```
진단 엔진이 임계값 감지
  → 프로세스 개입: 대화 내에서 텍스트로 가볍게 정리
    → 사용자: "시각화로 보여줘" → 브라우저 시각화 생성
    → 사용자: "넘어가자" → 대화 계속
    → 사용자: (그냥 이어서 대화) → 대화 계속
```

### 수동 요청 시

| 사용자 요청 | 동작 |
|------------|------|
| "정리해줘" | 상태 요약 (텍스트) + 자동 시각화 (브라우저) |
| "마인드맵으로 보여줘" | 지정 형태로 시각화 |
| "지금 어디야?" | 상태 대시보드 |
| "뭐가 결정됐지?" | 결정 사항 요약 |
| "다른 형태로" | 현재 시각화를 다른 형태로 전환 |

## 8. 플러그인 구조

```
planning-compass/
├── skills/
│   └── compass/
│       ├── SKILL.md            # 메인 프롬프트 (배경 원칙 + 진단 로직 + 상호작용 규칙)
│       ├── diagnosis.md        # 개입 카탈로그 (8가지) + 진단 조건 상세
│       ├── visualization.md    # 시각화 카탈로그 (10가지) + 매핑 규칙 + 렌더링 가이드
│       └── state-schema.md     # 상태 파일 스키마 + 추상화 로직 (추출 가이드라인)
└── package.json                # 플러그인 메타데이터
```

**SKILL.md의 구조:**
- 배경 원칙 (항상 적용)
- 상태 추적 지시 (경량 트랙 + 동기화 트랙)
- 진단 로직 (임계값, 트리거 조건)
- 상호작용 규칙 (개입 UX, 시각화 출력 패턴)
- 보조 파일 참조 규칙 (진단 시 diagnosis.md Read, 시각화 시 visualization.md Read)

**보조 파일은 필요 시에만 Read:**
- 진단이 개입을 결정했을 때 → `diagnosis.md` 참조
- 시각화를 생성할 때 → `visualization.md` 참조
