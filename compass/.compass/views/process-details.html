Planning Compass — Process Details

파이프라인 각 프로세스의 내부 단계 상세

항상 활성 프로세스 매 응답마다 자동 실행
1 백그라운드 원칙 적용 항상 활성
1
정보 제한
응답 내 새 정보/옵션/개념 수 카운트 → ≥5개이면 상위 3~5개만 제시
"N개 더 있습니다 — [간단 라벨]. 자세히 볼까요?" 형태로 나머지 보류
2
위치 고정
새 요소 도입 시 → 기존 확정 사항과의 연결점 확인 → "이건 [기존 요소]와 연결되는데..." 형태로 서술
연결점 없는 정보는 부유(float)하지 않도록 앵커링
3
사전 요약
결정 요청 직전 → 해당 결정에 관련된 맥락을 여러 턴에서 수집 → 한곳에 정리 후 제시
산재된 정보를 결정 시점에 모아서 의사결정 품질 향상
4
필수/선택 분리
모든 항목에 명시적 라벨 부착:
지금 결정 필요 vs 나중에 고려
5
참여도 추적
사용자 응답의 변화 패턴을 턴 단위로 모니터링: 길이 변화, 구체성 수준, 능동적 질문 빈도
수동적 패턴 감지 시 → "주도권 반환" 개입의 입력 데이터가 됨
2 경량 상태 추적 갱신 항상 활성
1
decided_count 갱신
대화에서 확정된 항목 수 추적 — 명시적 선택, 추천 승인, 암묵적 수용 포함
2
undecided_count 갱신
미결정 항목 수 추적 — 다수 옵션 제시 후 미선택, "나중에", 질문 미답변
3
active_topics 갱신
현재 병렬 진행 중인 논의 주제 수 — 주제 전환 시 +1, 주제 종결 시 -1
4
new_concepts 산정
이번 응답에서 새로 도입할 개념/정보 수 — 정보 제한 원칙과 진단 엔진의 입력
파일 I/O 없음 — 내부 메모리에서만 추적. 이 카운터들이 진단 엔진의 4가지 임계값과 직접 매핑됨.

3 시각화 요청?
A
명시적 요청 감지
"보여줘", "시각화", "마인드맵", "다이어그램", "flowchart" 등 키워드
B
정리 요청 + 고복잡도
"정리해줘", "organize" → 텍스트 정리 후 시각화 제안
"이걸 대시보드로 보여드릴까요?"
C
유형 직접 지정
"플로우차트로 보여줘" → 진단 엔진 무시, 지정 유형 사용
4 진단 임계값 초과?
카운터임계값매핑 개입
undecided≥ 5결정 정리
active_topics≥ 3초점 좁히기

수동적 응답 감지 기준 (2개 이상 해당 시 트리거):
• 연속 3회 ~20자 미만 • 부연 없는 순수 동의
• 질문/요구사항 추가 중단 • 제안 무수정 반복

가이드라인이지 엄격한 규칙이 아님 — 사용자가 활발하면 임계값 초과해도 개입하지 않음

5 상태 파일 작성/갱신 양쪽 경로 공통
파일 스키마 (6개 섹션)
# Goal 원래 목표 — 안정적 유지, 대화 진행 중 임의 수정 금지
# Elements 이름 붙은 개념들 + type 태그 (architecture, feature, constraint...)
# Relations 요소 간 연결: depends-on, contains, precedes, communicates-with
# Decided 확정 항목 + 결정 내용 + 이유
# Undecided 미결정 항목 + 옵션 목록 + 관련 요소
# Active 현재 논의 중인 주제 + 마지막 동기화 시간
추출 로직
요소 식별 (신호 강도별)
강 — 사용자가 직접 이름 붙임 → 즉시 등록
중 — 여러 턴에 걸쳐 반복 언급 → 요소 후보
약 — 1회 언급 → 다른 요소와 연결될 때만
결정 판별
• 명시적 선택: "X로 하자"
• 추천 승인: "네, 그게 맞아요"
• 암묵적 수용: 가정 위에 추가 논의 진행
가정 투명성
추론된 관계는 사실이 아닌 가정으로 제시
"이건 [X]가 [Y]에 의존한다고 보고 있는데, 맞나요?"

6 시각화 유형 선택
상태 파일의 데이터 특성을 분석하여 10가지 유형 중 선택. 사용자 지정은 항상 우선.
요소가 계층적? → 마인드맵 / 계층도
요소 간 관계가 복잡? → 컨셉맵
결정/미결정 추적이 핵심? → 상태 대시보드
순서/단계가 핵심? → 플로우차트 / 타임라인
옵션 비교? → 비교 매트릭스
우선순위 정리? → 우선순위 매트릭스
상태 전이/생명주기? → 상태 머신
영역 겹침/경계? → 벤 다이어그램

9 개입 유형 선택
핵심 개입 (4종)

undecided ≥ 5 결정 정리

1. 미결정 항목 전체 나열
2. 의존성 분석 — 어떤 결정이 다른 결정을 차단하는지
3. 가장 많은 것을 해제하는 결정부터 제안
4. 한 번에 하나만 — "N개 미결정 중 [X]부터 어떨까요?"

topics ≥ 3 초점 좁히기

1. 활성 스레드 명명: "(1) A, (2) B, (3) C"
2. 각 스레드 상태 표시 (초기? 거의 결정? 차단?)
3. 사용자에게 선택권: "어느 것부터 집중할까요?"
나머지는 삭제하지 않고 명시적으로 보류

concepts ≥ 5 핵심 요약

1. 가장 중요한/기초적인 2~3개 식별
2. 해당 항목만 전체 맥락과 함께 제시
3. 나머지 존재 언급: "[N]개 더 있습니다 — [라벨]"
단순 불릿 리스트 금지 — 계층 구조가 핵심

passive 2+ 주도권 반환

1. 새 정보/옵션/결정 제시 즉시 중단
2. 현재 상태 2~3문장 요약
3. 구체적 선택지: "(A) 더 파기, (B) 정리 검토, (C) 전환"
4. "제가 좀 많이 진행한 것 같은데, 뭐가 중요할까요?"

확장 개입 (4종)

맥락 소환

이전 결정과 현재 논의의 연결점을 표면화
"이전에 [X]로 결정했는데, 여기에 연결됩니다"

목표 재정렬

논의가 원래 Goal에서 이탈 시 확인
"원래 [A]였는데 지금 [B]를 다루고 있어요"

트레이드오프 명확화

옵션 간 차별화 기준을 추출하여 명시적 비교 제시

정체 돌파

같은 주제 3회+ 재방문 시
관점 전환/잠정 결정/보류 제안

10 텍스트 기반 개입 실행

Light 개입

응답에 자연스럽게 녹임
"지금까지 정리하면 — "

사용자가 개입을 의식하지 못하는 것이 이상적

Heavy 개입

명시적 체크포인트로 제시
"여러 스레드가 쌓였습니다. 현재 상태:"

사용자는 "skip", "later"로 회피 가능

개입 후 시각화 제안도 함께 고려: "이걸 대시보드로 시각화해서 보여드릴까요?" — 정리 요청은 시각화의 최고 가치 순간

11 일반 응답 생성
별도의 개입이나 시각화 없이 응답 생성.
단, 백그라운드 원칙 5가지는 여전히 적용됨 —
정보 제한, 위치 고정, 사전 요약, 필수/선택 분리, 참여도 추적이 모든 응답에 내재.
