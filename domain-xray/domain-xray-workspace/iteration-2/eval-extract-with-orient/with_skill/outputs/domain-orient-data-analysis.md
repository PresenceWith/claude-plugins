# 데이터 분석 오리엔테이션 맵

## 이 맵의 용도
이 맵은 데이터 분석이라는 넓은 분야를 처음 접하는 학습자가 "어디에 무엇이 있는지" 전체 지형을 파악하기 위한 뼈대입니다. 앞으로 배우는 세부 지식을 이 구조 위에 얹어 가면, 단편적 암기가 아니라 체계적인 이해로 이어집니다.

## 당신의 출발점 (앵커 포인트)
배경지식이 전혀 없다고 느끼더라도, 일상에서 이미 데이터 분석의 핵심 사고를 경험하고 있습니다.

- **가계부 작성** — 수입과 지출을 기록하고 패턴을 파악하는 행위는 "데이터 수집 → 정리 → 인사이트 도출"이라는 분석의 기본 흐름 그 자체입니다.
- **비교 쇼핑** — 여러 제품의 가격, 후기, 스펙을 비교해 결정하는 과정은 "다차원 데이터를 기준에 따라 평가"하는 것과 같습니다.
- **날씨 예보 확인** — "비 올 확률 60%"를 보고 우산을 챙기는 판단은 확률 기반 의사결정의 일상적 형태입니다.
- **요리 레시피 조절** — 4인분 레시피를 2인분으로 줄이는 것은 비례/비율 사고이며, 데이터 스케일링의 직관적 버전입니다.

이런 일상 경험들이 데이터 분석 개념을 배울 때 "아, 이런 느낌이구나"라고 연결되는 다리 역할을 합니다.

## 도메인 전체 지형

### 영역 1: 데이터 수집과 저장
**무엇을 하는가**: 분석에 필요한 데이터를 다양한 소스(데이터베이스, API, 로그 파일, 스프레드시트)에서 가져오고, 나중에 쓸 수 있도록 구조화된 형태로 저장합니다. 실무자는 SQL로 데이터베이스를 조회하고, CSV/엑셀 파일을 다루며, 때로는 웹 스크래핑으로 외부 데이터를 확보합니다.
**왜 중요한가**: 분석의 품질은 원재료인 데이터의 품질에 달려 있습니다. 데이터를 제대로 수집하지 못하면 이후 모든 단계가 무의미해집니다.
**핵심 용어**:
- **SQL (Structured Query Language)**: 데이터베이스에서 데이터를 꺼내오는 표준 언어
- **스키마(Schema)**: 데이터가 어떤 구조(테이블, 열, 타입)로 저장되어 있는지 정의한 설계도
- **ETL (Extract, Transform, Load)**: 데이터를 추출하고, 변환하고, 적재하는 파이프라인 과정
- **데이터 웨어하우스**: 분석용으로 정리된 대규모 데이터 저장소
**다른 영역과의 연결**: 이 영역에서 꺼낸 데이터가 "데이터 정제"와 "탐색적 분석"의 입력값이 됩니다.

### 영역 2: 데이터 정제와 전처리
**무엇을 하는가**: 현실의 데이터는 항상 지저분합니다. 빠진 값, 중복, 형식 불일치, 이상치를 찾아내고 수정합니다. 실무에서는 전체 분석 시간의 60-80%가 이 단계에 소요됩니다.
**왜 중요한가**: "쓰레기가 들어가면 쓰레기가 나온다(Garbage In, Garbage Out)"는 데이터 분석의 제1원칙입니다. 정제되지 않은 데이터로 만든 분석 결과는 잘못된 의사결정으로 이어집니다.
**핵심 용어**:
- **결측값(Missing Value)**: 비어 있는 데이터 — 삭제할지, 대체할지 판단이 필요
- **이상치(Outlier)**: 나머지 데이터와 극단적으로 다른 값 — 오류인지 실제 현상인지 구분해야 함
- **정규화(Normalization)**: 서로 다른 단위나 범위의 데이터를 비교 가능하게 맞추는 작업
- **데이터 타입 변환**: 문자열을 날짜로, 텍스트를 숫자로 바꾸는 등의 형식 통일
**다른 영역과의 연결**: 수집된 데이터를 받아 정제한 뒤, 탐색적 분석과 시각화 단계로 넘깁니다.

### 영역 3: 탐색적 데이터 분석 (EDA)
**무엇을 하는가**: 데이터를 이리저리 살펴보면서 패턴, 추세, 이상한 점을 발견하는 과정입니다. 가설을 세우기 전에 "데이터가 어떤 이야기를 하고 있는지" 감을 잡는 단계이며, 통계 요약과 간단한 그래프를 반복적으로 그리면서 진행합니다.
**왜 중요한가**: EDA 없이 바로 분석 모델을 만들면 데이터의 특성을 놓쳐 잘못된 결론에 도달할 위험이 큽니다. 의사가 진찰 전에 문진하는 것과 같습니다.
**핵심 용어**:
- **분포(Distribution)**: 데이터 값들이 어떻게 퍼져 있는지 — 히스토그램으로 시각화
- **상관관계(Correlation)**: 두 변수가 함께 움직이는 정도 — 인과관계와 다름에 주의
- **기술통계(Descriptive Statistics)**: 평균, 중앙값, 표준편차 등 데이터를 요약하는 수치
- **피벗(Pivot)**: 데이터를 다른 관점에서 재구성하여 보는 기법
**다른 영역과의 연결**: EDA의 발견이 "통계 분석"의 가설을 형성하고, "시각화"의 방향을 결정합니다.

### 영역 4: 통계 분석과 추론
**무엇을 하는가**: 표본 데이터에서 얻은 결과가 우연인지 의미 있는 패턴인지 판단합니다. A/B 테스트 결과 해석, 예측 모델 구축, 추세 예측 등에 통계적 방법론을 적용합니다.
**왜 중요한가**: "매출이 올랐다"는 관찰에서 "이 변화가 통계적으로 유의미하다"는 결론으로 넘어가려면 통계적 추론이 필수입니다. 이것이 직감과 데이터 기반 의사결정의 차이입니다.
**핵심 용어**:
- **p-값(p-value)**: 관찰된 결과가 우연히 발생할 확률 — 작을수록 "의미 있음"
- **A/B 테스트**: 두 버전을 비교하여 어떤 것이 더 나은지 실험으로 검증하는 방법
- **회귀분석(Regression)**: 변수 간 관계를 수학적으로 모델링하는 기법
- **신뢰구간(Confidence Interval)**: 참값이 존재할 것으로 추정되는 범위
**다른 영역과의 연결**: EDA에서 발견한 패턴을 검증하며, 결과는 "시각화"와 "커뮤니케이션" 단계를 통해 전달됩니다.

### 영역 5: 데이터 시각화
**무엇을 하는가**: 숫자와 패턴을 그래프, 차트, 대시보드로 변환하여 사람이 직관적으로 이해할 수 있게 만듭니다. 단순히 "예쁜 그래프"를 만드는 것이 아니라, 데이터의 핵심 메시지를 정확하고 정직하게 전달하는 기술입니다.
**왜 중요한가**: 아무리 뛰어난 분석도 의사결정권자에게 전달되지 못하면 무용지물입니다. 시각화는 분석과 행동 사이의 다리입니다.
**핵심 용어**:
- **대시보드(Dashboard)**: 핵심 지표들을 한눈에 볼 수 있도록 구성한 시각화 화면
- **KPI (Key Performance Indicator)**: 비즈니스 성과를 측정하는 핵심 지표
- **차트 유형 선택**: 비교에는 막대, 추세에는 선, 비율에는 파이 — 메시지에 맞는 차트가 다름
- **인터랙티브 시각화**: 사용자가 필터링, 드릴다운할 수 있는 동적 시각화
**다른 영역과의 연결**: 모든 분석 영역의 결과물을 시각적으로 표현하며, BI 리포팅과 직접 연결됩니다.

### 영역 6: BI 리포팅과 대시보드
**무엇을 하는가**: Tableau, Power BI, Looker 같은 도구를 사용하여 조직 전체가 정기적으로 참고하는 보고서와 대시보드를 구축하고 유지합니다. 일회성 분석이 아니라 반복 가능한 모니터링 시스템을 만드는 작업입니다.
**왜 중요한가**: 개인이 한 번 분석하는 것과 조직이 지속적으로 데이터를 보는 것은 다릅니다. BI 시스템은 데이터 기반 문화의 인프라입니다.
**핵심 용어**:
- **셀프서비스 BI**: 비기술 직군도 직접 데이터를 조회하고 보고서를 만들 수 있는 환경
- **드릴다운(Drill-down)**: 요약 수치에서 세부 데이터로 파고드는 탐색
- **데이터 모델링**: BI 도구에서 테이블 간 관계를 설정하여 분석 가능하게 만드는 설계
- **스케줄링**: 보고서를 자동으로 갱신하고 배포하는 설정
**다른 영역과의 연결**: 데이터 수집/저장 인프라 위에 구축되며, 시각화 역량이 직접 활용됩니다.

### 영역 7: 분석 결과 커뮤니케이션
**무엇을 하는가**: 분석 결과를 비기술적 이해관계자가 이해하고 행동할 수 있는 형태로 전달합니다. 프레젠테이션, 보고서 작성, 의사결정 회의에서의 설명이 포함됩니다. "그래서 뭘 해야 하는데?"라는 질문에 답하는 영역입니다.
**왜 중요한가**: 데이터 분석의 최종 목적은 더 나은 의사결정입니다. 분석 자체가 아니라 분석에 기반한 행동이 가치를 만듭니다. 커뮤니케이션 능력이 없으면 분석가의 영향력은 제한됩니다.
**핵심 용어**:
- **데이터 스토리텔링**: 숫자를 맥락과 내러티브로 엮어 전달하는 기술
- **액셔너블 인사이트(Actionable Insight)**: 구체적인 행동으로 이어질 수 있는 분석 결과
- **이해관계자(Stakeholder)**: 분석 결과를 받아 의사결정을 내리는 사람들
- **가정과 한계 명시**: 분석의 전제조건과 제약을 투명하게 밝히는 것
**다른 영역과의 연결**: 모든 영역의 최종 산출물이 이 단계를 거쳐 비즈니스 가치로 전환됩니다.

## 학습 우선순위 맵
배경지식이 전무한 상태에서 실무 역량을 목표로 하는 당신의 상황에 맞춘 순서입니다.

**여기서 시작하세요** (가장 큰 레버리지):
- **데이터 수집과 저장 (SQL 중심)** — 데이터를 꺼내올 수 없으면 아무것도 시작할 수 없습니다. SQL은 데이터 분석가의 가장 기본적인 도구이며, 배우기 시작하면 즉시 실습이 가능합니다.
- **데이터 정제와 전처리** — 현실의 데이터를 다루는 순간 반드시 만나는 단계입니다. 엑셀이나 Python pandas로 직접 더러운 데이터를 정리하면서 데이터 감각이 형성됩니다.

**다음으로 배우세요** (중요하지만 급하지 않음):
- **탐색적 데이터 분석 (EDA)** — SQL과 정제 기술을 익힌 후, 데이터에서 이야기를 찾아내는 연습으로 자연스럽게 넘어갑니다.
- **데이터 시각화** — EDA와 함께 배우면 시너지가 큽니다. 차트를 그리면서 데이터를 더 깊이 이해하게 됩니다.

**나중에 배우세요** (가치 있지만 기반이 먼저):
- **통계 분석과 추론** — 기초가 잡힌 후 체계적으로 배워야 오히려 효과적입니다. 데이터 감각 없이 통계부터 배우면 공식 암기에 그칩니다.
- **BI 리포팅과 대시보드** — 도구(Tableau, Power BI) 학습은 기본 분석 역량이 갖춰진 후가 적절합니다.
- **분석 결과 커뮤니케이션** — 전달할 분석 결과가 생긴 후에 자연스럽게 발전시킬 수 있는 영역입니다.

## 전문가는 알지만 초보자는 모르는 것
- **"정답"은 거의 없다**: 같은 데이터를 보고도 분석가마다 다른 인사이트를 도출합니다. 데이터 분석은 과학이면서 동시에 기술(craft)입니다.
- **질문 설계가 분석보다 어렵다**: 초보자는 "이 데이터로 뭘 알 수 있을까"라고 묻지만, 전문가는 "이 비즈니스 문제를 풀려면 어떤 데이터가 필요한가"부터 시작합니다. 질문의 방향이 정반대입니다.
- **상관관계와 인과관계의 혼동은 가장 흔한 실수다**: 두 지표가 함께 움직인다고 해서 하나가 다른 하나를 일으킨 것이 아닙니다. 이 구분을 본능적으로 하게 되기까지 오래 걸립니다.
- **데이터가 없는 곳을 아는 것도 전문성이다**: 전문가는 "이 분석에는 어떤 데이터가 빠져 있고, 그래서 이 결론에는 이런 한계가 있다"를 항상 의식합니다. 보이는 데이터만큼 보이지 않는 데이터가 중요합니다.
- **도구보다 사고방식이 먼저다**: Python이냐 R이냐, Tableau냐 Power BI냐는 부차적입니다. "이 현상을 어떻게 측정할 수 있을까", "이 숫자가 정말로 의미하는 것은 무엇인가"라는 분석적 사고방식이 도구 선택보다 100배 중요합니다.
