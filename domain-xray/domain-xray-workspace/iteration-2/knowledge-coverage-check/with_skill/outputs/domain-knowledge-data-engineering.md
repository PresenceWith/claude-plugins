# Data Engineering: 현장 전문가의 암묵지

이 문서는 Data Engineering(데이터 엔지니어링) 분야에서 20년 이상의 경력을 가진 실무자가 "당연히 알고 있겠지"라고 전제하고 넘어가는 지식 — 교과서에는 없지만 현장에서는 공기처럼 사용하는 암묵지를 체계적으로 발굴한 것이다. 백엔드 개발자로서 Python과 SQL 경험이 있는 학습자가 데이터 엔지니어링 팀과 효과적으로 협업하기 위해 필요한 실무 지식에 초점을 맞춘다.

데이터 엔지니어링의 전체 라이프사이클은 다음 단계로 구성된다: (1) Data Pipeline Design, (2) Data Quality Management, (3) Data Storage Architecture, (4) Data Modeling, (5) Workflow Orchestration, (6) Stream Processing, (7) Data Governance and Lineage. 앞의 네 단계는 우선 학습 영역으로 깊이 있게 다루고, 나머지는 맥락 파악 수준으로 다룬다.

# 1. Data Pipeline Design (데이터 파이프라인 설계)

데이터 파이프라인은 원천 데이터를 비즈니스에 유용한 형태로 변환하여 목적지까지 전달하는 일련의 처리 과정이다. 백엔드 개발에서 API 요청-응답이 밀리초 단위의 동기적 흐름이라면, 데이터 파이프라인은 수 분에서 수 시간에 걸친 비동기적 배치 처리가 기본이다. 이 근본적인 시간 감각의 차이가 설계 철학 전체를 지배한다.

## 실무 용어와 뉘앙스

**ETL vs ELT**: Extract-Transform-Load와 Extract-Load-Transform은 단순한 순서 차이가 아니다. ETL은 데이터가 목적지에 도착하기 전에 변환하는 패턴으로, 전통적인 on-premise 데이터 웨어하우스 시대의 산물이다. 스토리지와 컴퓨트가 비쌌기 때문에 "깨끗한 데이터만 넣자"는 철학이었다. ELT는 클라우드 시대의 패턴으로, 일단 raw 데이터를 그대로 넣고 웨어하우스의 컴퓨팅 파워로 변환한다. 현장에서 "우리는 ELT 한다"고 말하면, 대부분 Snowflake나 BigQuery 같은 클라우드 웨어하우스를 쓰고 있고, dbt 같은 transformation 도구를 사용한다는 뜻이다. 반면 "ETL 파이프라인"이라고 하면 Spark나 Python 스크립트로 변환 로직을 직접 작성하는 경우가 많다.

**Idempotency(멱등성)**: 백엔드 개발자에게 멱등성은 REST API의 PUT 요청처럼 "같은 요청을 여러 번 보내도 결과가 같다"는 의미일 것이다. 데이터 파이프라인에서의 멱등성은 훨씬 구체적이고 절실하다. "어제 날짜 파이프라인을 세 번 돌려도 어제의 데이터가 세 벌이 아니라 한 벌만 있어야 한다"는 뜻이다. 실무에서는 이를 위해 파이프라인 실행 시 먼저 해당 파티션의 기존 데이터를 삭제(DELETE)하고 다시 삽입(INSERT)하는 "delete-then-insert" 패턴이나, MERGE/UPSERT 문을 사용한다. 팀원이 "이 파이프라인 idempotent 해?"라고 물으면, "재실행해도 데이터 중복 안 생기지?"라는 질문이다.

**Backfill(백필)**: 과거 데이터를 소급하여 재처리하는 작업이다. "3월 데이터 backfill 해야 해"라는 말은 3월 1일부터 31일까지의 파이프라인을 순차적으로 다시 돌려야 한다는 뜻이다. 실무에서 backfill은 놀라울 정도로 자주 발생한다 — 로직 수정, 소스 데이터 정정, 새로운 컬럼 추가 등의 이유로. 따라서 파이프라인 설계 시 backfill 가능성을 처음부터 고려해야 한다. Backfill이 불가능한 파이프라인은 현장에서 "brittle(깨지기 쉬운)"하다고 평가받는다.

**DAG (Directed Acyclic Graph, 방향성 비순환 그래프)**: 파이프라인들 사이의 의존 관계를 표현하는 방식이다. "DAG이 꼬였다"는 말은 파이프라인 간 순환 의존이 생겼거나, 의존 관계가 지나치게 복잡해져서 한 파이프라인의 실패가 수십 개의 후속 파이프라인을 연쇄적으로 멈추게 만드는 상황을 가리킨다.

## 숨겨진 전제 조건

파이프라인 설계에서 전문가들이 당연시하지만 초보자가 모르는 가장 결정적인 전제는 **"파이프라인은 반드시 실패한다"는 가정 하에 설계한다**는 것이다. 백엔드 개발에서도 에러 핸들링을 하지만, 웹 서비스는 실패하면 사용자에게 에러를 보여주고 끝이다. 파이프라인은 실패하면 그 위에 의존하는 모든 후속 파이프라인이 멈추고, 최종 대시보드의 데이터가 갱신되지 않으며, 비즈니스 의사결정이 지연된다. 따라서 모든 파이프라인에는 반드시 다음이 설계되어야 한다: (a) 실패 시 알림(Slack, PagerDuty 등), (b) 어디서 실패했는지 식별 가능한 로깅, (c) 중간 지점부터 재시작할 수 있는 checkpoint 또는 전체 재실행 가능한 멱등성.

또 하나의 중요한 전제는 **시간 개념의 구분**이다. 데이터 엔지니어링에서 "시간"은 최소 세 가지를 의미한다: event time(이벤트가 실제로 발생한 시간), processing time(파이프라인이 이벤트를 처리한 시간), ingestion time(데이터가 시스템에 도착한 시간). 백엔드 개발자는 보통 이 세 시간이 거의 같다고 생각하지만, 데이터 엔지니어링에서는 이들이 수 시간에서 수일까지 차이 날 수 있다. 예를 들어, 모바일 앱에서 오프라인 상태로 생성된 이벤트는 event time이 3일 전이지만 ingestion time은 오늘이다. 파이프라인이 "어제 데이터를 처리한다"고 할 때, 어떤 시간 기준인지 명확히 하지 않으면 데이터 누락이나 중복이 발생한다.

세 번째 전제는 **소스 시스템은 당신의 통제 밖에 있다**는 것이다. 백엔드에서는 자신의 데이터베이스 스키마를 직접 관리하지만, 데이터 파이프라인의 소스는 외부 API, 다른 팀의 데이터베이스, SaaS 도구의 export 등이다. 이들은 사전 공지 없이 스키마가 변경되거나, 데이터 형식이 바뀌거나, 갑자기 응답이 느려질 수 있다. 전문가들은 이를 당연히 가정하고, 소스 데이터에 대한 validation을 파이프라인 초입에 반드시 넣는다.

구체적 시나리오: 한 초보 데이터 엔지니어가 외부 결제 API에서 일일 정산 데이터를 가져오는 파이프라인을 만들었다. API가 안정적으로 작동했기에 에러 핸들링 없이 배포했다. 2주 뒤, API 제공자가 응답 JSON에 새 필드를 추가하면서 기존 필드의 중첩 구조가 변경되었다. 파이프라인은 조용히 실패했고(Python의 KeyError가 어딘가에서 잡혀서 빈 데이터로 처리됨), 3일간 정산 데이터가 0원으로 기록되었다. 재무팀에서 보고서 이상을 발견한 뒤에야 문제가 드러났다. 이것이 전형적인 "소스를 신뢰한 대가"이다.

## 초보자의 함정

**함정 1: "한 번에 전부" 처리하려는 설계**. 백엔드 개발자 출신은 종종 "전체 데이터를 한 번에 읽어서 처리하자"는 접근을 한다. 1만 건의 주문 데이터라면 문제 없지만, 데이터 엔지니어링에서 다루는 것은 수억 건이다. 전체 데이터를 매일 다시 처리하면 비용과 시간이 기하급수적으로 증가한다. 전문가들은 이를 위해 incremental processing(증분 처리) 패턴을 사용한다. "어제 이후에 변경된 데이터만 처리한다"는 접근이다. 그러나 이 패턴에는 전제 조건이 있다 — 소스에 reliable한 timestamp 컬럼이나 change data capture(CDC) 메커니즘이 있어야 한다. 이 전제가 없으면 "새로운 데이터"와 "이미 처리한 데이터"를 구분할 수 없다.

구체적 시나리오: 한 팀이 PostgreSQL의 `updated_at` 컬럼을 기준으로 증분 처리를 구현했다. 문제는 원천 시스템의 일부 UPDATE 쿼리가 `updated_at`을 갱신하지 않는 코드 경로가 있었다는 것이다. 결과적으로 특정 유형의 변경사항이 파이프라인에 영원히 반영되지 않았고, 이를 발견하기까지 6주가 걸렸다. 올바른 접근은 CDC(예: Debezium을 통한 WAL 기반 변경 캡처)를 사용하거나, `updated_at`의 신뢰성을 소스 팀과 명시적으로 합의하는 것이다.

**함정 2: 재시도 로직 없는 파이프라인**. 네트워크 타임아웃, 일시적 소스 장애, 메모리 부족 등은 일상적으로 발생한다. 그런데 파이프라인 재시도가 멱등하지 않으면 재시도 자체가 문제를 일으킨다. 예를 들어, append-only 테이블에 INSERT하는 파이프라인을 재시도하면 데이터가 중복된다. 전문가들은 "재시도해도 안전한" 파이프라인만이 재시도 로직을 가질 자격이 있다고 본다.

**함정 3: Failure recovery를 "로그 보고 수동으로 고치면 되지"로 생각하는 것**. 하나의 파이프라인이 하루에 한 번 돌 때는 수동 복구가 가능하다. 그러나 수백 개의 파이프라인이 매시간 돌고 있을 때, 하나의 실패가 cascade로 수십 개를 멈추게 한다. 이 규모에서는 "자동으로 복구 가능한" 설계만이 생존한다.

## 경험 법칙

**"파이프라인 하나가 하나의 논리적 단위를 처리하게 하라."** 주문 데이터와 사용자 데이터를 하나의 파이프라인에서 처리하면, 주문 소스가 장애일 때 사용자 데이터까지 처리가 멈춘다. 각각 분리하면 독립적으로 실패하고 독립적으로 복구할 수 있다. 이 법칙이 깨지는 때: 두 데이터가 트랜잭션적으로 일관성을 유지해야 하는 경우(예: 주문과 결제가 반드시 같은 시점에 반영되어야 할 때)에는 하나의 파이프라인으로 묶는 것이 오히려 맞다.

**"Full refresh가 가능하면 Full refresh부터 시작하라."** 증분 처리는 복잡하고 버그가 숨기 쉽다. 데이터 규모가 작거나(수천만 건 이하) 처리 시간이 SLA 내에 들어오면, 매번 전체를 다시 처리하는 것이 훨씬 단순하고 안전하다. 증분으로 전환하는 것은 full refresh가 SLA를 위반하기 시작할 때 하면 된다. 이 법칙이 깨지는 때: 소스 데이터가 DELETE를 포함하는 경우. 증분 처리로는 삭제된 레코드를 감지하기 어렵기 때문에, 주기적 full refresh를 병행해야 한다(이를 "full refresh reconciliation"이라 부른다).

**"파이프라인의 SLA는 가장 느린 소스에 의해 결정된다."** 세 개의 소스에서 데이터를 가져오는 파이프라인에서, 두 소스는 1분 내에 응답하지만 하나가 30분 걸리면 전체 SLA는 30분+alpha이다. 전문가들은 이를 알기에, 느린 소스는 별도 파이프라인으로 분리하고 JOIN은 후처리 단계에서 수행한다.

# 2. Data Quality Management (데이터 품질 관리)

데이터 품질은 데이터 엔지니어링에서 가장 과소평가되면서 동시에 가장 많은 사고의 원인이 되는 영역이다. 코드에 버그가 있으면 테스트가 실패하고 배포가 막히지만, 데이터에 문제가 있으면 조용히 잘못된 대시보드와 잘못된 의사결정으로 이어진다. "garbage in, garbage out(쓰레기가 들어오면 쓰레기가 나간다)"은 진부한 격언이지만, 데이터 엔지니어링 현장에서는 매주 몸으로 느끼는 현실이다.

## 실무 용어와 뉘앙스

**Schema Drift(스키마 드리프트)**: 소스 데이터의 스키마가 예고 없이 변경되는 현상이다. 컬럼이 추가되거나 제거되고, 데이터 타입이 변경되며, nullable이었던 필드가 갑자기 null을 보내기 시작한다. 현장에서 "스키마가 drift 했다"고 말하면, 소스의 변경이 파이프라인을 깨뜨렸거나 깨뜨릴 위험이 있다는 경고이다. 실무에서는 이를 감지하기 위해 파이프라인 초입에 schema validation을 넣는다. Great Expectations, Soda, dbt tests 같은 도구가 여기에 쓰인다.

**Data Contract(데이터 계약)**: 소스 팀(producer)과 소비 팀(consumer) 사이에 데이터의 스키마, 품질 기준, SLA를 명시적으로 합의하는 것이다. "우리 API 응답의 `user_id` 필드는 항상 non-null이고 양의 정수이며, 스키마 변경 시 최소 2주 전에 공지한다"와 같은 합의이다. Data Contract라는 개념이 2023~2024년부터 크게 부상했는데, 그 전에는 이런 합의가 암묵적이거나(그래서 자주 깨지거나) 아예 없었다.

**Data Freshness(데이터 신선도)**: 대시보드나 테이블의 데이터가 얼마나 최신인지를 나타낸다. "이 테이블의 freshness가 2시간이다"는 것은 가장 최근 데이터가 2시간 전 것이라는 뜻이다. Freshness SLA가 1시간인데 실제가 2시간이면 stale(부패한) 데이터라고 한다. 비즈니스 유저가 "대시보드 숫자가 이상해요"라고 할 때, 첫 번째로 확인하는 것이 freshness이다.

**Data Observability(데이터 관측성)**: 소프트웨어의 observability(로그, 메트릭, 트레이스)에 대응하는 개념으로, 데이터의 상태를 실시간으로 모니터링하는 것이다. Volume(데이터 양이 갑자기 줄었는가), Freshness(데이터가 제시간에 도착하는가), Schema(스키마가 변했는가), Distribution(값의 분포가 이상한가), Lineage(어떤 테이블이 어떤 테이블에 영향을 주는가)의 다섯 축으로 관측한다. Monte Carlo, Elementary, Datafold 같은 도구가 이 영역에 있다.

## 숨겨진 전제 조건

전문가들이 당연시하는 첫 번째 전제는 **"데이터 품질 검증은 파이프라인의 일부이지, 별도의 작업이 아니다"**라는 것이다. 초보자는 파이프라인을 만들고 나서 "나중에 품질 체크를 추가하지"라고 생각하지만, 전문가에게 검증이 없는 파이프라인은 미완성품이다. 파이프라인의 각 단계 — 추출 후, 변환 후, 적재 후 — 에 assertion이 있어야 한다.

구체적 시나리오: 마케팅팀이 "어제 캠페인 전환율이 200%인데요?"라고 문의했다. 원인을 추적해보니, 소스 시스템에서 특정 이벤트가 중복 발생하여 전환 건수가 2배로 기록된 것이었다. 파이프라인에 "전환율은 0~100% 사이여야 한다"는 간단한 assertion만 있었어도 적재 시점에 알림이 갔을 것이다. 이 사고는 "검증 없는 파이프라인은 거짓을 생산하는 기계"라는 교훈의 전형적 예시이다.

두 번째 전제는 **"'정확한 데이터'의 정의는 비즈니스가 결정한다"**는 것이다. 매출이 10,000원인지 10,500원인지는 기술적 문제가 아니라, "매출에 취소 건을 포함하는가?", "세금을 포함하는가?", "미확정 결제를 포함하는가?"라는 비즈니스 정의의 문제이다. 데이터 엔지니어가 자의적으로 판단하면 반드시 나중에 "숫자가 안 맞아요"라는 사고가 터진다. 전문가들은 이런 비즈니스 로직을 코드 리뷰에서 가장 꼼꼼히 검토한다.

세 번째 전제는 **"테스트 데이터로는 데이터 품질 문제를 발견할 수 없다"**는 것이다. 소프트웨어 테스트에서는 fixture 데이터로 충분히 검증할 수 있지만, 데이터 품질 문제의 80%는 "예상하지 못한 실제 데이터"에서 발생한다 — null이 올 리 없는 필드에 null이 오고, 날짜 형식이 ISO 8601이 아니라 "January 5th, 2024"로 오고, 금액에 음수가 들어오는 식이다. 따라서 데이터 품질 검증은 프로덕션 데이터 위에서 돌아야 의미가 있다.

## 초보자의 함정

**함정 1: Schema Drift를 단순한 에러로 취급하는 것**. 소스 컬럼 하나가 추가되었을 때, 초보자는 "에러 나면 그때 고치지"라고 생각한다. 그러나 schema drift는 두 가지 유형이 있다: additive(추가)와 breaking(파괴적). 컬럼 추가는 보통 안전하지만, 컬럼 타입 변경(string에서 integer로)이나 컬럼 제거는 파이프라인을 즉시 깨뜨린다. 더 위험한 것은 "의미적 drift" — 컬럼 이름과 타입은 같지만 내용의 의미가 바뀌는 것이다.

구체적 시나리오: 외부 SaaS 도구에서 `status` 필드가 기존에 "active", "inactive" 두 값만 가졌는데, 어느 날 "suspended"가 추가되었다. 스키마 자체는 여전히 string이므로 파이프라인은 에러 없이 돌아갔다. 그러나 분석 쿼리에서 `WHERE status = 'active'`로 필터링하는 로직이, "suspended"(본질적으로 inactive에 가까운) 사용자를 active 수에 잘못 포함시켰다. 이런 "조용한 오류"가 데이터 품질에서 가장 치명적이다.

**함정 2: 데이터 품질 체크를 배포 시점에만 하는 것**. CI/CD에서 테스트를 돌리듯, 데이터 품질 체크도 배포 때 한 번 돌리면 된다고 생각하기 쉽다. 하지만 데이터 품질 문제는 코드 변경 없이도 발생한다 — 소스 데이터가 바뀌면 어제 정상이었던 파이프라인이 오늘 쓰레기를 생산한다. 따라서 데이터 품질 체크는 파이프라인이 실행될 때마다 돌아야 한다.

**함정 3: 모든 품질 문제를 blocking으로 처리하는 것**. 초보자는 "데이터 품질 체크 실패 시 파이프라인 중단"을 모든 검증에 적용하려 한다. 그러나 실무에서는 severity를 나눈다: critical(중단해야 함 — 예: primary key 중복), warning(알림만 보내고 계속 진행 — 예: 특정 컬럼의 null 비율이 평소보다 높음), info(로깅만 — 예: 데이터 볼륨이 평소의 80%). Critical만 blocking으로, 나머지는 non-blocking으로 처리하는 것이 현실적이다. 모든 것을 blocking으로 하면 파이프라인이 너무 자주 멈춰서 결국 체크를 꺼버리게 된다.

## 경험 법칙

**"소스에 가까울수록 검증을 엄격히 하라."** 파이프라인 초입(raw 데이터 적재 직후)에서 문제를 잡으면 영향 범위가 좁지만, downstream 테이블까지 퍼진 후에 발견하면 수십 개의 테이블을 backfill해야 한다. 이 법칙이 깨지는 때: 비즈니스 로직에 의존하는 검증(예: "매출이 전일 대비 50% 이상 하락했는가")은 raw 단계에서 할 수 없고 transformation 이후에 해야 한다.

**"새로운 데이터 소스를 연결한 첫 2주는 매일 눈으로 확인하라."** 자동화된 검증은 "예상한 문제"만 잡는다. 새로운 소스는 어떤 문제가 있을지 예상할 수 없으므로, 초기에는 사람이 직접 데이터를 살펴보며 패턴을 파악해야 한다. 이 관찰 기간에 발견한 이상 패턴이 이후 자동 검증 규칙의 기초가 된다. 이 법칙이 깨지는 때: 이미 잘 알려진 표준적 소스(예: Stripe, Salesforce 등 주요 SaaS의 공식 API)는 커뮤니티에서 검증된 best practice가 있으므로 관찰 기간을 단축할 수 있다.

**"데이터 품질 문제의 가장 흔한 원인 3가지: null, 중복, 타이밍."** 이 세 가지가 전체 데이터 사고의 80%를 차지한다. null은 예상치 못한 곳에서 나타나고, 중복은 재시도나 소스 문제로 발생하며, 타이밍은 파이프라인 실행 순서나 late-arriving 데이터로 인한 것이다. 새 파이프라인의 최소 검증 세트는 이 세 가지에 대한 체크이다.

# 3. Data Storage Architecture (데이터 저장소 아키텍처)

데이터를 어디에, 어떤 형태로 저장할 것인가는 비용, 성능, 유연성의 삼각 트레이드오프이다. 백엔드 개발자에게 데이터 저장소란 PostgreSQL이나 MySQL 같은 OLTP(Online Transaction Processing) 데이터베이스일 것이다. 데이터 엔지니어링에서는 이와 근본적으로 다른 OLAP(Online Analytical Processing) 워크로드를 위한 저장소를 다룬다.

## 실무 용어와 뉘앙스

**Data Warehouse(데이터 웨어하우스)**: "웨어하우스"라고 하면 현장에서는 거의 항상 Snowflake, BigQuery, Redshift 중 하나를 가리킨다. 교과서적 정의(주제 지향적, 통합적, 비휘발적, 시변적 데이터 저장소)보다 실무적으로 중요한 특징은: (a) SQL로 쿼리한다, (b) 컴퓨트와 스토리지가 분리되어 있다(특히 Snowflake와 BigQuery), (c) 컬럼 기반 저장이라 분석 쿼리에 최적화되어 있다. 백엔드의 PostgreSQL과 가장 큰 차이는, 웨어하우스는 수십억 행에 대한 집계 쿼리를 수십 초 만에 처리하지만, 단일 행 조회(point query)는 오히려 느리다는 것이다.

**Data Lake(데이터 레이크)**: 원래는 "모든 데이터를 원본 그대로 저장하는 곳"이었다. 실무에서 "레이크"라고 하면 S3, GCS, ADLS 같은 오브젝트 스토리지에 Parquet이나 JSON 파일로 데이터를 저장하는 것을 의미한다. 레이크의 핵심 장점은 스키마를 강제하지 않으므로 어떤 데이터든 일단 저장할 수 있다는 유연성이다. 단점은 "무엇이 저장되어 있고 어디에 있는지" 관리가 안 되면 data swamp(데이터 늪)로 전락한다는 것이다 — 데이터는 있는데 아무도 찾을 수 없고 신뢰할 수 없는 상태.

**Lakehouse(레이크하우스)**: 레이크의 유연성과 웨어하우스의 성능 및 관리 기능을 결합한 아키텍처이다. Databricks의 Delta Lake, Apache Iceberg, Apache Hudi가 대표적인 구현체이다. 실무적으로 "lakehouse를 쓴다"는 것은 S3 같은 오브젝트 스토리지 위에 Delta Lake나 Iceberg 같은 테이블 포맷을 얹어서, 레이크인데 ACID 트랜잭션과 스키마 관리가 되는 환경을 만들었다는 뜻이다. 2024~2025년 기준으로 이 아키텍처가 빠르게 표준이 되어가고 있다.

**Parquet(파케이)**: 컬럼 기반 파일 포맷으로, 데이터 엔지니어링의 사실상 표준 파일 포맷이다. CSV와 비교하면: (a) 컬럼별 압축으로 파일 크기가 1/5 이하, (b) 특정 컬럼만 읽을 수 있어 분석 쿼리가 빠름, (c) 스키마가 파일에 내장되어 있음. "그 데이터 Parquet으로 떨궈줘"라는 요청이 있으면, "분석에 적합한 컬럼 기반 포맷으로 변환하여 저장하라"는 뜻이다.

**Partitioning(파티셔닝)**: 데이터를 물리적으로 분할하여 저장하는 전략이다. 가장 흔한 것은 날짜 기반 파티셔닝으로, `s3://data-lake/orders/dt=2024-01-15/` 처럼 날짜별로 디렉터리를 나눈다. 파티셔닝이 중요한 이유는 쿼리 성능과 직결되기 때문이다: "1월 데이터만 분석"할 때 전체 데이터를 스캔하지 않고 1월 파티션만 읽으면 된다. 이를 partition pruning(파티션 프루닝)이라 한다.

## 숨겨진 전제 조건

전문가들이 당연시하는 핵심 전제는 **"저장소 선택은 기술적 문제가 아니라 워크로드 패턴의 문제"**라는 것이다. "Snowflake가 좋은가 BigQuery가 좋은가"는 의미 없는 질문이다. "동시에 쿼리를 실행하는 분석가가 50명인가 5명인가", "쿼리 패턴이 ad-hoc인가 정해진 리포트인가", "데이터 볼륨이 테라바이트인가 페타바이트인가"에 따라 정답이 달라진다.

구체적 시나리오: 한 스타트업이 "요즘 다 Snowflake 쓰니까"라는 이유로 Snowflake를 도입했다. 월 데이터 볼륨은 10GB에 불과했고, 쿼리하는 사람은 3명이었다. Snowflake의 최소 비용(warehouse가 켜져 있는 시간 × 크기)이 실제 워크로드 대비 과도했다. PostgreSQL에 분석용 read replica를 두는 것으로 충분했을 것이다. 반대로, 일 수백 GB의 데이터를 수십 명이 동시에 분석해야 하는 환경에서 PostgreSQL을 고집하면 쿼리 하나에 30분씩 걸리고, 분석 DB가 서비스 DB에 부하를 줘서 서비스 장애까지 이어질 수 있다.

두 번째 전제는 **"스토리지 계층은 여러 겹이다"**라는 것이다. 백엔드에서는 보통 하나의 데이터베이스에 모든 것을 저장하지만, 데이터 엔지니어링에서는 데이터가 여러 계층을 거친다: Raw(원본 그대로) → Staging(기본 정제) → Curated/Mart(비즈니스 로직 적용 완료). 각 계층마다 저장소와 접근 권한이 다르다. Raw는 오브젝트 스토리지에 원본을 보존하고, Mart는 웨어하우스에 분석가가 바로 사용할 수 있는 형태로 있다.

## 초보자의 함정

**함정 1: OLTP 데이터베이스로 분석을 시도하는 것**. "이미 PostgreSQL에 데이터가 다 있는데 왜 웨어하우스를 따로 만들어요?"는 초보자의 전형적 질문이다. 소규모에서는 실제로 문제 없지만, 분석 쿼리의 특성(전체 테이블 스캔, 대량 집계, 여러 테이블 JOIN)은 OLTP 데이터베이스의 설계 목적과 정반대이다. 서비스 DB에서 무거운 분석 쿼리를 돌리면 서비스의 응답 시간이 느려지고, 최악의 경우 커넥션 풀이 고갈되어 서비스 장애로 이어진다.

구체적 시나리오: 데이터 분석가가 프로덕션 PostgreSQL에서 3개 테이블을 JOIN하는 집계 쿼리를 실행했다. 10분째 돌아가는 중에 해당 DB의 커넥션이 모두 점유되었고, 웹 서비스가 500 에러를 반환하기 시작했다. 이후 "분석은 read replica에서만"이라는 규칙이 생겼지만, 그마저도 replica lag 때문에 실시간 데이터와 차이가 생겨 분석가가 혼란스러워했다. 결국 웨어하우스로 데이터를 복제하는 파이프라인을 구축하게 되었다.

**함정 2: 파티셔닝 키를 잘못 선택하는 것**. 가장 흔한 실수는 카디널리티(고유 값의 수)가 너무 높은 컬럼으로 파티셔닝하는 것이다. 예를 들어, `user_id`로 파티셔닝하면 수백만 개의 파티션이 생기고, 각 파티션에 데이터가 몇 건씩만 들어간다. 이를 "small files problem(작은 파일 문제)"이라 하며, 메타데이터 관리 비용이 실제 데이터보다 커지는 역설이 발생한다. 반대로, 카디널리티가 너무 낮은 컬럼(예: boolean)으로 파티셔닝하면 파티션이 2개뿐이라 의미가 없다.

## 경험 법칙

**"일 데이터 볼륨 100GB 이하, 동시 분석 사용자 10명 이하면 PostgreSQL read replica로 시작해도 된다."** 초기 스타트업에서 웨어하우스 도입은 오버엔지니어링일 수 있다. 이 법칙이 깨지는 때: 데이터 소스가 3개 이상이고 서로 JOIN해야 하는 경우, 또는 데이터 히스토리를 장기 보관해야 하는 경우에는 초기부터 웨어하우스를 고려할 가치가 있다.

**"Raw 데이터는 절대 삭제하지 마라."** 변환 로직에 버그가 있었을 때, raw 데이터가 있으면 수정 후 재처리하면 된다. 없으면 소스에서 다시 가져와야 하는데, 소스가 이미 데이터를 삭제했거나 스키마가 바뀌었을 수 있다. 스토리지 비용(특히 S3 같은 오브젝트 스토리지)은 충분히 싸다. 이 법칙이 깨지는 때: GDPR 같은 규정에 의해 개인정보 데이터를 반드시 삭제해야 하는 경우. 이때는 raw 데이터에서도 해당 사용자의 데이터를 삭제하는 프로세스가 필요하다.

**"파티셔닝 키는 쿼리 패턴을 따른다."** 대부분의 분석 쿼리가 "지난 7일", "이번 달" 처럼 날짜 범위로 필터링한다면 날짜 파티셔닝이 맞다. "특정 국가의 데이터"로 자주 필터링한다면 국가 기반 파티셔닝이 맞다. 이 법칙이 깨지는 때: 쿼리 패턴이 매우 다양하거나 예측 불가능한 ad-hoc 분석 환경에서는, 파티셔닝보다 클러스터링(데이터를 정렬하여 저장)이 더 효과적일 수 있다.

# 4. Data Modeling (데이터 모델링)

데이터 모델링은 "데이터를 어떤 구조로 정리할 것인가"에 대한 답이다. 백엔드의 데이터베이스 모델링이 "어떻게 하면 데이터 무결성을 유지하면서 효율적으로 저장하는가"에 집중한다면, 분석을 위한 데이터 모델링은 "어떻게 하면 비즈니스 질문에 빠르고 정확하게 답할 수 있는가"에 집중한다. 이 목적의 차이가 모든 설계 원칙을 바꾼다.

## 실무 용어와 뉘앙스

**Dimensional Modeling(차원 모델링)**: Ralph Kimball이 체계화한 분석용 데이터 모델링 방법론이다. 핵심은 데이터를 fact(팩트, 사실)와 dimension(디멘션, 차원)으로 분리하는 것이다. Fact table(팩트 테이블)은 측정 가능한 비즈니스 이벤트(주문, 클릭, 결제 등)를 담고, dimension table(디멘션 테이블)은 그 이벤트의 맥락(누가, 어디서, 무엇을 등)을 담는다. 현장에서 "그 테이블 fact야 dimension이야?"라는 질문은 "측정값(금액, 수량)이 있는 이벤트 테이블이야, 아니면 설명적 속성(이름, 카테고리)을 가진 참조 테이블이야?"라는 뜻이다.

**Star Schema(스타 스키마)**: 중앙에 fact table이 있고 주변에 dimension table이 방사형으로 연결된 구조로, 별 모양이라 star schema라 부른다. `fact_orders` 테이블이 `dim_customer`, `dim_product`, `dim_date`, `dim_store` 등과 foreign key로 연결된다. 현장에서 "스타 스키마로 모델링했다"는 말은 정규화를 의도적으로 풀어서(denormalize) 분석 쿼리의 JOIN 수를 최소화했다는 뜻이다.

**Slowly Changing Dimension (SCD, 느리게 변하는 차원)**: 차원 데이터가 시간에 따라 변할 때 이력을 어떻게 관리하는가의 문제이다. 가장 흔히 쓰이는 유형은 SCD Type 2로, 변경이 발생하면 기존 레코드를 종료(end_date 설정)하고 새 레코드를 추가하는 방식이다. "고객이 서울에서 부산으로 이사했을 때, 이사 전 주문은 서울로, 이사 후 주문은 부산으로 연결"되어야 하기 때문이다. 현장에서 "이 dimension SCD2로 관리해?"라고 물으면, "변경 이력 추적하고 있어?"라는 뜻이다.

**Grain(그레인)**: 팩트 테이블에서 한 행이 나타내는 것의 세밀함 수준이다. "이 테이블의 grain은 주문-상품 단위"라는 것은 한 행이 "한 주문에 포함된 하나의 상품"을 나타낸다는 뜻이다. Grain 정의가 데이터 모델링에서 가장 중요한 첫 번째 결정인데, 이것이 모호하면 이후의 모든 집계가 잘못된다.

**Surrogate Key(대리 키)**: Dimension table에 부여하는 시스템 생성 키(보통 자동 증가 정수)이다. 소스 시스템의 natural key(자연 키, 예: 고객번호)와 별도로 유지한다. 그 이유는 (a) 소스 키가 바뀔 수 있고(시스템 마이그레이션 등), (b) SCD Type 2에서 같은 고객이 여러 레코드로 존재할 때 각각을 식별해야 하며, (c) 여러 소스에서 오는 데이터를 통합할 때 키 충돌을 방지하기 위해서이다.

## 숨겨진 전제 조건

가장 중요한 전제는 **"분석용 모델링에서 정규화는 미덕이 아니라 장애물이다"**라는 것이다. 백엔드 개발자는 3NF(제3정규형)까지 정규화하는 것을 좋은 설계로 배웠다. 그러나 분석 환경에서 정규화된 테이블은 많은 JOIN을 필요로 하고, JOIN은 대용량 데이터에서 가장 비싼 연산이다. Dimensional modeling은 의도적으로 데이터를 중복 저장(denormalize)하여 JOIN을 줄인다. 이것이 "나쁜 설계"가 아니라 "다른 목적을 위한 다른 설계"임을 이해하는 것이 첫 번째 관문이다.

구체적 시나리오: 백엔드 경험이 있는 데이터 엔지니어가 웨어하우스에 정규화된 스키마를 그대로 복제했다. 분석가가 "월별 카테고리별 매출"이라는 단순한 질문에 답하려면 7개 테이블을 JOIN해야 했다. 쿼리가 복잡해지니 실수가 잦았고, 두 분석가가 같은 질문에 다른 숫자를 내놓는 일이 반복되었다. Star schema로 재설계한 후 같은 질문이 2개 테이블 JOIN으로 해결되었고, "하나의 진실(single source of truth)"이 확립되었다.

두 번째 전제는 **"모델링은 비즈니스 프로세스를 따른다"**는 것이다. 기술적으로 데이터를 구성하는 것이 아니라, 비즈니스가 실제로 측정하고 분석하는 프로세스 단위로 팩트 테이블을 설계한다. "주문 프로세스", "반품 프로세스", "마케팅 캠페인 프로세스" 각각이 별도의 팩트 테이블이 된다.

세 번째 전제는 **"grain을 먼저 정하고 나머지를 결정한다"**는 것이다. Fact table을 설계할 때 "이 테이블에 어떤 컬럼을 넣을까?"가 아니라 "이 테이블에서 한 행은 무엇을 의미하는가?"부터 시작한다. Grain이 정해지면 어떤 dimension이 연결되어야 하는지, 어떤 measure가 가능한지가 자동으로 결정된다.

구체적 시나리오: 한 팀이 "주문 매출 팩트 테이블"을 만들면서 grain을 명확히 정의하지 않았다. 어떤 행은 주문 단위(주문 1건 = 1행)이고, 어떤 행은 주문-상품 단위(주문의 각 상품 = 1행)였다. `amount` 컬럼이 때로는 주문 총액, 때로는 상품별 금액을 나타냈다. SUM(amount)를 구하면 같은 주문의 총액이 중복 합산되어 매출이 부풀려졌다. 이 문제는 grain 정의의 부재에서 시작된 전형적인 사례이다.

## 초보자의 함정

**함정 1: "하나의 거대한 테이블에 모든 것을 넣자" (Wide Table 집착)**. 모든 컬럼을 한 테이블에 모아놓으면 JOIN이 필요 없어서 편하다고 생각한다. 그러나 200개 컬럼짜리 wide table은: (a) 하나의 dimension이 변경될 때 전체 테이블을 재구축해야 하고, (b) 쿼리 시 불필요한 컬럼까지 스캔하여 비용이 증가하며, (c) 컬럼 간 의존 관계를 파악하기 어려워 데이터 lineage가 불투명해진다. Star schema의 fact/dimension 분리는 이런 문제를 해결하는 실증된 패턴이다.

구체적 시나리오: 한 팀이 모든 주문 관련 데이터를 하나의 `orders_all` 테이블에 넣었다. 고객 이름이 200만 행에 반복 저장되었고, 고객이 이름을 변경하면 200만 행을 모두 UPDATE해야 했다. dim_customer 테이블이 있었다면 1행만 수정하면 되었을 것이다.

**함정 2: Surrogate key 없이 source key만 사용하는 것**. 소스 시스템의 `customer_id`를 그대로 웨어하우스 dimension의 primary key로 쓰면, 소스 시스템이 마이그레이션하면서 ID 체계가 바뀌었을 때 웨어하우스 전체가 깨진다. 또한 SCD Type 2에서는 같은 고객이 여러 레코드로 존재하므로 source key만으로는 유일성을 보장할 수 없다.

**함정 3: Fact table에 텍스트 속성을 넣는 것**. Fact table에 `customer_name`이나 `product_category` 같은 설명적 속성을 넣으면 데이터 중복이 발생하고, 해당 값이 변경되었을 때 일관성이 깨진다. 이런 속성은 dimension table에 있어야 하고, fact table에서는 dimension의 surrogate key만 참조해야 한다.

## 경험 법칙

**"Grain은 가장 세밀한 수준으로 잡아라."** 주문-상품 단위로 저장하면 나중에 주문 단위로 집계할 수 있지만, 주문 단위로 저장하면 상품 단위로 쪼갤 수 없다. 세밀한 grain은 미래의 분석 유연성을 보장한다. 이 법칙이 깨지는 때: 데이터 볼륨이 너무 커서(예: 초당 수만 건의 클릭스트림) 가장 세밀한 grain으로는 비용과 성능이 감당이 안 될 때. 이때는 의도적으로 grain을 올려서(예: 분 단위 집계) 비용과 유연성 사이에서 타협한다.

**"Fact와 Dimension이 헷갈리면 '측정할 수 있는가?'를 물어라."** SUM, AVG, COUNT 등으로 집계할 수 있는 숫자 값이면 fact의 measure이고, 그 숫자를 필터링하거나 그룹핑하는 데 쓰이는 설명적 속성이면 dimension이다. "주문 금액"은 fact(SUM 가능), "고객 등급"은 dimension(GROUP BY에 사용). 이 법칙이 깨지는 때: "재고 수량"같은 snapshot measure는 SUM이 되지만 사실은 특정 시점의 상태이므로, accumulating snapshot이라는 별도 패턴이 필요하다.

**"모델링은 한 번에 완성하려 하지 마라. 비즈니스 질문 3개를 먼저 받고, 그것에 답할 수 있는 최소 모델을 만들어라."** 전체 비즈니스를 포괄하는 데이터 모델을 한 번에 설계하려 하면 6개월이 걸리고 결국 쓰이지 않는다. Agile하게, 가장 급한 분석 요구부터 해결하는 작은 모델을 만들고 점진적으로 확장하는 것이 실무적이다. 이 법칙이 깨지는 때: 규제 보고 요건처럼 변경이 어렵고 처음부터 완전해야 하는 경우에는 upfront 설계에 시간을 투자하는 것이 맞다.

# 5. Workflow Orchestration (워크플로 오케스트레이션)

오케스트레이션은 "어떤 파이프라인을 어떤 순서로, 언제 실행할 것인가"를 관리하는 것이다. 파이프라인이 1~2개일 때는 cron job으로 충분하지만, 수십~수백 개의 파이프라인이 서로 의존하며 실행될 때는 전용 도구가 필요하다.

## 실무 용어와 뉘앙스

**Airflow(에어플로)**: Apache Airflow는 오케스트레이션의 사실상 표준이다. 현장에서 "Airflow DAG 만들어줘"라고 하면, 파이프라인의 실행 순서와 스케줄을 Python 코드로 정의하라는 뜻이다. Airflow의 핵심 개념은 DAG(파이프라인 그래프), Task(개별 작업), Operator(작업 유형 — BashOperator, PythonOperator 등), Schedule(실행 주기)이다.

**Dagster와 Prefect**: Airflow의 대안으로 부상한 차세대 도구이다. Dagster는 "asset(자산)" 중심 접근으로, "이 테이블을 만드는 데 필요한 것"을 선언적으로 정의한다. Prefect는 Python 함수에 데코레이터를 붙이는 방식으로, Airflow보다 Python-native한 경험을 제공한다.

## 숨겨진 전제 조건

**오케스트레이터는 실행자가 아니라 지휘자이다.** Airflow 자체가 데이터를 처리하는 것이 아니라, "이 Spark job을 실행해라", "이 dbt model을 돌려라"라는 명령을 내리는 역할이다. 무거운 처리를 Airflow 워커에서 직접 실행하면(예: Airflow 워커에서 pandas로 10GB 데이터를 처리) 워커가 죽고 전체 스케줄링이 멈춘다.

## 초보자의 함정

Airflow DAG을 처음 만들 때, `execution_date`의 의미를 오해하는 것이 가장 흔하다. Airflow에서 `execution_date`는 "DAG이 실행되는 시간"이 아니라 "처리 대상 기간의 시작"이다. 매일 자정에 실행되는 DAG의 1월 15일 `execution_date`는, 1월 15일 자정에 실행되어 1월 15일 데이터를 처리한다는 뜻이 아니라, 1월 16일 자정에 실행되어 1월 15일 데이터를 처리한다는 뜻이다. 이 역직관적 동작 때문에 backfill 시 하루 차이 데이터를 처리하는 실수가 빈번하다.

# 6. Stream Processing (스트림 처리)

배치 처리가 "모아서 한 번에" 처리하는 것이라면, 스트림 처리는 "발생하는 즉시" 처리하는 것이다. Kafka, Flink, Spark Streaming이 대표적 도구이다.

## 실무 용어와 뉘앙스

**Kafka(카프카)**: 엄밀히는 메시지 큐가 아니라 distributed commit log(분산 커밋 로그)이다. 현장에서 "Kafka에 넣어"라는 말은 "이벤트를 Kafka topic에 publish해"라는 뜻이다. Kafka의 핵심 차별점은 메시지를 소비해도 삭제되지 않는다는 것이다(retention 기간 동안 유지). 따라서 여러 consumer가 같은 데이터를 독립적으로 읽을 수 있다.

**Consumer Group과 Offset**: Consumer group은 같은 topic을 읽는 consumer의 논리적 그룹이며, offset은 "어디까지 읽었는지"를 나타내는 위치이다. "offset을 리셋한다"는 것은 과거 데이터를 다시 처리한다는 뜻이다(스트림의 backfill에 해당).

## 경험 법칙

**"진짜 실시간이 필요한지 먼저 물어라."** 비즈니스가 "실시간 대시보드가 필요해"라고 할 때, 정말 밀리초 단위 반영이 필요한 것인지 5분 배치면 충분한 것인지 확인하라. 스트림 처리의 운영 복잡도와 비용은 배치의 3~5배이다. 이 법칙이 깨지는 때: 부정거래 탐지, 실시간 추천 등 비즈니스 가치가 지연 시간에 직결되는 경우.

# 7. Data Governance and Lineage (데이터 거버넌스와 리니지)

거버넌스는 "누가 어떤 데이터에 접근할 수 있는가, 데이터가 어디서 왔고 어디로 가는가"를 관리하는 것이다. 파이프라인이 수백 개 이상이 되면 "이 숫자는 어디서 온 거지?"라는 질문에 답하기 어려워지며, 이때 lineage(데이터 계보)가 필수가 된다.

## 실무 용어와 뉘앙스

**Lineage(리니지)**: 데이터의 출처와 흐름을 추적하는 것이다. "이 대시보드의 매출 숫자가 이상한데, 원천이 뭐지?"라는 질문에 답하기 위해 역추적하는 과정이다. Lineage 도구(dbt docs, DataHub, Atlan 등)는 테이블 간 의존 관계를 시각화하여 "A 테이블에 문제가 생기면 어떤 테이블이 영향받는가"를 보여준다.

**PII (Personally Identifiable Information, 개인식별정보)**: 이름, 이메일, 전화번호 등 개인을 식별할 수 있는 데이터이다. GDPR, 개인정보보호법 등의 규정에 의해 특별 관리가 필요하며, 데이터 파이프라인에서 PII 마스킹, 암호화, 접근 제어를 적용해야 한다.

## 경험 법칙

**"거버넌스는 나중에 추가하는 것이 아니라 처음부터 설계하는 것이다."** PII가 raw 데이터에 평문으로 들어가고 나면, 이후에 마스킹을 적용하는 것은 기존 모든 데이터를 재처리하는 대규모 작업이 된다. 이 법칙이 깨지는 때: 초기 프로토타입 단계에서는 speed가 우선이므로, 최소한의 접근 제어만 설정하고 정식 거버넌스는 production 전에 적용하는 것이 현실적이다.
