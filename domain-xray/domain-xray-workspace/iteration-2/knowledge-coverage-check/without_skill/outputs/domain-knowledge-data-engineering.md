# 데이터 엔지니어링: 전문가의 암묵지 해부

## 개요: 데이터 엔지니어링의 전체 생명주기

데이터 엔지니어링은 단순히 "데이터를 A에서 B로 옮기는 일"이 아니다. 그것은 조직의 모든 의사결정이 의존하는 데이터의 신뢰성, 적시성, 접근성을 보장하는 인프라를 설계하고 운영하는 일이다. 백엔드 개발자가 API의 reliability(신뢰성)와 latency(지연시간)를 고민하듯, 데이터 엔지니어는 데이터의 freshness(신선도), correctness(정확성), completeness(완전성)를 고민한다.

전체 생명주기는 다음 단계로 구성된다:

1. Data Ingestion(데이터 수집) -- 원천 시스템에서 데이터를 가져오는 단계
2. Data Pipeline Design(데이터 파이프라인 설계) -- ETL/ELT 패턴 선택과 파이프라인 아키텍처 수립
3. Data Storage Architecture(데이터 저장소 아키텍처) -- 데이터를 어디에 어떤 형태로 저장할 것인가
4. Data Transformation(데이터 변환) -- 원시 데이터를 분석 가능한 형태로 가공
5. Data Quality Management(데이터 품질 관리) -- 데이터의 신뢰성을 보장하는 모든 활동
6. Data Modeling(데이터 모델링) -- 분석에 최적화된 데이터 구조 설계
7. Orchestration and Observability(오케스트레이션과 관측성) -- 파이프라인의 실행 관리와 상태 모니터링
8. Data Governance and Lineage(데이터 거버넌스와 계보 추적) -- 데이터의 출처, 변환 과정, 접근 권한 관리

# 1단계: Data Pipeline Design -- ETL/ELT 패턴, 실패 복구, 멱등성

이 영역은 데이터 엔지니어링의 심장부다. 백엔드 개발자가 "요청-응답" 사이클을 중심으로 사고하듯, 데이터 엔지니어는 "추출-변환-적재" 사이클을 중심으로 사고한다. 그러나 이 단순해 보이는 사이클 안에 수십 년의 실패 경험에서 축적된 암묵지가 숨어 있다.

## 1.1 ETL vs ELT: 현장에서의 실제 의미

### 실무 용어와 뉘앙스

ETL(Extract-Transform-Load)과 ELT(Extract-Load-Transform)의 교과서적 차이는 "변환을 먼저 하느냐, 적재 후에 하느냐"이다. 그러나 현장에서 이 용어가 실제로 의미하는 바는 훨씬 깊다.

ETL은 전통적으로 Informatica, Talend, DataStage 같은 전용 도구가 데이터를 소스에서 꺼내 중간 서버(staging server)에서 변환한 뒤 목적지 데이터 웨어하우스에 넣는 패턴을 의미했다. 여기서 핵심은 "변환에 사용되는 컴퓨팅 자원이 목적지 시스템과 분리되어 있다"는 점이다. 변환 로직이 별도의 서버에서 실행되므로, 목적지 시스템(예: Oracle 데이터 웨어하우스)의 부하를 줄일 수 있었다. 이것이 온프레미스 시대에 ETL이 표준이 된 이유다 -- 데이터 웨어하우스 라이선스가 CPU 코어당 수천만 원이었기 때문에, 비싼 웨어하우스에서 변환 작업까지 돌리는 것은 경제적으로 비합리적이었다.

ELT는 클라우드 데이터 웨어하우스(Snowflake, BigQuery, Redshift)가 등장하면서 주류가 되었다. 이들 시스템은 컴퓨팅과 스토리지가 분리(decoupled)되어 있어, 변환 작업을 위해 컴퓨팅을 일시적으로 scale up(확장)한 뒤 다시 줄일 수 있다. 따라서 "일단 원시 데이터를 그대로 적재하고, 나중에 웨어하우스 안에서 SQL로 변환하자"는 접근이 경제적으로 합리적이 되었다. dbt(data build tool)가 ELT 시대의 대표적인 변환 도구로 부상한 이유가 여기에 있다.

### 숨겨진 전제 조건

현장에서 ETL과 ELT 논의를 할 때, 팀원들이 말하지 않지만 당연히 전제하는 것들이 있다.

첫째, "raw layer(원시 계층)의 존재"이다. ELT 패턴을 쓴다고 말할 때, 소스 데이터를 있는 그대로 저장하는 raw layer가 반드시 존재한다는 것을 전제한다. 이 계층은 "우리가 받은 데이터의 원본 증거"로서, 변환 로직에 버그가 있었을 때 원본으로 되돌아가 재처리할 수 있게 해준다. 초보자가 흔히 놓치는 것이 이 raw layer의 중요성이다 -- "어차피 변환할 건데 왜 원본을 따로 저장하지?"라고 생각하기 쉽지만, 실무에서 raw layer 없이 운영하다가 변환 로직 버그로 데이터를 영구적으로 손상시킨 사고 사례는 셀 수 없이 많다.

둘째, "멱등성(idempotency) 보장이 설계의 출발점"이라는 전제다. 이것은 아래에서 깊이 다룬다.

### 초보자의 함정: "ETL이냐 ELT냐"를 기술 선택 문제로 보는 것

초보자는 ETL vs ELT를 "어떤 도구를 쓸 것인가"의 문제로 접근하는 경향이 있다. 그러나 실무에서 이 선택은 "조직의 데이터 성숙도와 팀 구조"에 의해 결정된다.

구체적인 시나리오를 보자. 한 스타트업에서 데이터 분석가 2명이 데이터 엔지니어 없이 직접 파이프라인을 관리하고 있다. 이들은 SQL에 능숙하지만 Python 기반의 복잡한 변환 파이프라인을 유지보수할 역량이 없다. 이 상황에서 ETL 패턴을 도입하면 -- 즉 Python 스크립트로 변환 로직을 짜면 -- 분석가들이 변환 로직을 이해하지 못해 데이터 파이프라인이 "블랙박스"가 된다. 반면 ELT 패턴으로 dbt를 도입하면, 분석가들이 익숙한 SQL로 변환 로직을 직접 작성하고 수정할 수 있다. 이 경우 "올바른" 선택은 ELT이지만, 그 이유는 기술적 우월성이 아니라 팀의 역량 분포(skill distribution) 때문이다.

### 현장 경험 법칙

**"변환 로직의 소유자가 누구인지가 ETL/ELT 선택을 결정한다."** 데이터 엔지니어가 변환을 소유하면 ETL(Python, Spark 등으로 복잡한 변환 가능), 분석가/분석 엔지니어가 변환을 소유하면 ELT(SQL 기반 변환)가 자연스럽다. 이 법칙이 깨지는 경우는 비정형 데이터(이미지, 로그 파싱 등)를 다룰 때다 -- 이런 데이터는 SQL로 변환하기 어려우므로, ELT 환경에서도 ETL 방식의 전처리 단계가 필요하다.

## 1.2 Idempotency(멱등성): 파이프라인 설계의 제1원칙

### 실무 용어와 뉘앙스

Idempotency(멱등성)는 "같은 입력으로 파이프라인을 여러 번 실행해도 결과가 동일하다"는 속성이다. REST API에서 PUT이 멱등적이라고 배운 백엔드 개발자에게 이 개념은 익숙할 것이다. 그러나 데이터 파이프라인에서의 멱등성은 API에서의 멱등성보다 훨씬 미묘하고 달성하기 어렵다.

API에서 PUT의 멱등성은 "같은 요청을 여러 번 보내도 리소스 상태가 동일"이라는 단순한 의미다. 데이터 파이프라인에서의 멱등성은 "2024년 3월 15일 데이터를 처리하는 파이프라인을 3번 실행하면, 결과 테이블에 3월 15일 데이터가 1번만 반영되어야 한다"는 의미다. 이것이 어려운 이유는 파이프라인이 "append(추가)" 방식으로 동작하는 경우가 많기 때문이다 -- 같은 날짜 데이터를 3번 실행하면 3배로 중복 적재된다.

### 숨겨진 전제 조건: Partition Key(파티션 키)와 Backfill(재처리)

멱등성을 논의할 때 전문가들이 당연히 전제하는 것은 "모든 파이프라인에는 logical partition key(논리적 파티션 키)가 있다"는 것이다. 대부분의 경우 이것은 날짜(execution_date 또는 data_date)다. 파이프라인을 멱등적으로 만드는 가장 일반적인 패턴은 다음과 같다:

**DELETE-INSERT 패턴**: 특정 파티션의 기존 데이터를 삭제한 후 새로 삽입한다. 예를 들어, 3월 15일 데이터를 처리하는 파이프라인은 먼저 결과 테이블에서 `WHERE data_date = '2024-03-15'` 조건으로 삭제한 뒤, 새로 계산한 데이터를 삽입한다. 이렇게 하면 같은 파이프라인을 몇 번을 실행해도 3월 15일 데이터는 항상 최신 결과 한 벌만 존재한다.

**MERGE/UPSERT 패턴**: 데이터가 이미 존재하면 업데이트하고, 없으면 삽입한다. Snowflake의 MERGE, BigQuery의 MERGE, PostgreSQL의 INSERT ... ON CONFLICT 구문이 이에 해당한다. 이 패턴은 DELETE-INSERT보다 안전하다 -- DELETE 도중 파이프라인이 실패하면 데이터가 사라질 수 있지만, MERGE는 atomic(원자적)하게 실행되기 때문이다.

**Backfill(재처리/백필)**은 멱등성이 있어야만 가능한 핵심 운영 작업이다. "지난주 변환 로직에 버그가 있었으니 지난 30일 데이터를 재처리하자"라는 상황은 데이터 엔지니어링에서 일상이다. 멱등적이지 않은 파이프라인은 backfill 시 데이터가 중복되거나 꼬이기 때문에, 멱등성 없는 파이프라인은 사실상 "운영 불가능한" 파이프라인이다.

### 초보자의 함정: INSERT만 쓰는 파이프라인

백엔드 개발자가 데이터 파이프라인을 처음 만들 때 가장 많이 저지르는 실수가 단순 INSERT를 사용하는 것이다. 구체적인 시나리오를 보자.

당신이 주문 데이터를 집계하는 일일 파이프라인을 만들었다. 매일 자정에 실행되어 전날의 주문을 집계해 `daily_order_summary` 테이블에 INSERT한다. 첫 주에는 완벽하게 동작한다. 그런데 목요일에 소스 시스템 장애로 파이프라인이 실패했다. 금요일 아침에 복구 후 목요일 파이프라인을 수동으로 재실행한다. 그런데 Airflow의 retry(재시도) 기능이 이미 한 번 부분적으로 성공한 목요일 작업을 다시 실행하면서, 목요일 데이터의 일부가 두 번 들어간다. 결과적으로 목요일 매출이 실제의 1.3배로 잡힌다. 분석팀이 "목요일 매출이 비정상적으로 높은데요?"라고 물었을 때야 비로소 중복을 인지한다.

이 시나리오는 과장이 아니다. 데이터 엔지니어링 커리어에서 가장 많이 마주치는 사고 유형 중 하나다. DELETE-INSERT나 MERGE 패턴을 처음부터 적용했다면 retry가 몇 번이건 결과는 동일했을 것이다.

### 현장 경험 법칙

**"파이프라인의 첫 번째 설계 질문은 '이것을 다시 실행하면 어떻게 되는가?'이다."** 모든 파이프라인은 반드시 재실행될 것이라고 가정하라. 네트워크 타임아웃, 소스 시스템 장애, 잘못된 변환 로직 수정 후 backfill 등 재실행 사유는 무한하다. 이 법칙이 깨지는 경우는 거의 없지만, 진정한 append-only(추가 전용) 시스템인 이벤트 로그 저장소(예: S3에 날짜 파티션으로 쌓는 raw 이벤트)는 예외다 -- 이 경우 멱등성 대신 "중복 제거(deduplication)"를 하류 단계에서 처리한다.

## 1.3 Failure Recovery(실패 복구): 파이프라인은 반드시 실패한다

### 실무 용어와 뉘앙스

**Retry(재시도)**는 실패한 작업을 자동으로 다시 시도하는 것이다. 그러나 현장에서 retry를 설정할 때 전문가가 반드시 고려하는 것은 "이 작업이 retry-safe(재시도 안전)한가?"이다. 위에서 설명한 멱등성이 보장되지 않으면 retry는 오히려 상황을 악화시킨다.

**Circuit breaker(서킷 브레이커)** 패턴은 백엔드 개발자에게 익숙할 것이다. 데이터 파이프라인에서도 이 패턴이 중요한데, 특히 소스 시스템에 대한 API 호출이 포함된 파이프라인에서 그렇다. 소스 시스템이 다운되었는데 retry가 계속 시도되면, 이미 과부하 상태인 소스 시스템에 추가 부하를 가하는 것이다. Exponential backoff(지수적 백오프)와 최대 retry 횟수 설정은 기본이다.

**Dead letter queue(사신 대기열, DLQ)**는 처리에 실패한 레코드를 별도 저장소에 격리하는 패턴이다. 스트리밍 파이프라인에서 특히 중요한데, 100만 개 이벤트 중 3개가 파싱 실패한다고 전체 파이프라인을 멈출 수는 없기 때문이다. 실패한 레코드를 DLQ로 보내고, 나머지는 정상 처리한 뒤, DLQ에 쌓인 레코드를 나중에 개별 분석하는 것이 표준 패턴이다.

### 숨겨진 전제 조건

실패 복구를 설계할 때 전문가들이 당연히 전제하지만 초보자가 놓치는 것은 **"파이프라인의 각 task(작업)가 독립적으로 재실행 가능해야 한다"**는 점이다. Airflow의 DAG(Directed Acyclic Graph, 방향성 비순환 그래프)에서 5개 task 중 3번째가 실패했을 때, 3번째 task만 재실행할 수 있어야 한다. 이를 위해서는 각 task가 자체적으로 멱등적이고, 이전 task의 결과를 영속적인 저장소(테이블, 파일)에서 읽어야 한다 -- 메모리 내 변수를 통해 task 간 데이터를 전달하면 중간 task만 재실행하는 것이 불가능하다.

### 초보자의 함정: 모든 실패를 같은 방식으로 처리하기

구체적인 시나리오를 보자. API에서 데이터를 추출하는 파이프라인이 HTTP 500 에러로 실패했다. 초보자는 이것을 단순히 retry로 해결하려 한다. 그러나 전문가는 먼저 "이 실패가 transient(일시적)인가, persistent(영구적)인가?"를 판단한다.

HTTP 500은 일시적일 가능성이 높으므로 retry가 적절하다. 그러나 HTTP 422(Unprocessable Entity)는 요청 자체에 문제가 있으므로 retry해봤자 계속 실패한다. Schema change(스키마 변경)로 인한 파싱 에러도 마찬가지다 -- 소스 시스템이 JSON 필드명을 바꾼 것이라면 retry는 무의미하고, 파이프라인 코드를 수정해야 한다.

더 미묘한 경우도 있다. 소스 시스템의 API가 "최근 24시간 데이터만 제공"하는 경우, 파이프라인이 48시간 동안 다운되면 중간 24시간의 데이터를 영영 가져올 수 없다. 이런 경우를 대비해 raw 데이터를 수집 즉시 영속적 저장소에 보관하는 것이 표준 관행이다.

### 현장 경험 법칙

**"알림은 '행동 가능한(actionable)' 것만 보내라."** 초보자는 모든 실패에 알림을 설정하지만, 자동 retry로 복구되는 일시적 실패까지 알림을 보내면 팀이 "알림 피로(alert fatigue)"에 빠진다. retry 3회 후에도 실패한 경우만 알림을 보내는 것이 일반적이다. 이 법칙이 깨지는 경우는 SLA(Service Level Agreement, 서비스 수준 협약)가 극히 엄격한 파이프라인이다 -- 예를 들어 "매일 오전 8시까지 경영진 대시보드가 갱신되어야 한다"는 SLA가 있다면, 첫 번째 실패부터 알림을 보내 수동 개입 시간을 확보해야 한다.

# 2단계: Data Quality Management -- 검증, 스키마 드리프트, 데이터 계약

데이터 품질은 데이터 엔지니어링에서 가장 과소평가되면서도 가장 많은 사고를 일으키는 영역이다. 백엔드 개발에서 "입력 검증"이 중요하듯, 데이터 엔지니어링에서 "데이터 검증"은 생존의 문제다. 차이점이 있다면, API 입력 검증은 즉시 에러를 반환하지만 데이터 품질 문제는 몇 주 후에야 발견되는 경우가 많다는 것이다.

## 2.1 Data Validation(데이터 검증): 무엇을, 언제, 어떻게

### 실무 용어와 뉘앙스

**Data validation(데이터 검증)**은 현장에서 여러 층위로 나뉜다.

**Schema validation(스키마 검증)**은 "데이터의 구조가 기대와 일치하는가"를 확인한다. 필드명, 데이터 타입, nullable 여부 등을 검사한다. 이것은 API 개발에서 요청 body의 JSON 스키마를 검증하는 것과 동일한 개념이다.

**Semantic validation(의미 검증)**은 "데이터의 값이 비즈니스적으로 유효한가"를 확인한다. 예를 들어 주문 금액이 음수가 아닌지, 날짜 필드가 미래가 아닌지, 국가 코드가 ISO 표준에 맞는지 등이다.

**Statistical validation(통계 검증)**은 "데이터의 분포가 기대 범위 안에 있는가"를 확인한다. 어제 일일 주문 건수가 10만 건이었는데 오늘 갑자기 100건이라면, 개별 레코드는 모두 유효하지만 전체적으로 뭔가 잘못된 것이다. Great Expectations, dbt tests, Soda 같은 도구가 이 층위의 검증을 지원한다.

현장에서 "data quality check"라고 하면 주로 통계 검증을 의미한다. 스키마 검증은 너무 기본적이어서 별도로 언급하지 않는 경우가 많고, 의미 검증은 비즈니스 로직에 종속되어 일반화하기 어렵기 때문이다.

### 숨겨진 전제 조건

데이터 검증을 논의할 때 전문가가 전제하는 핵심은 **"검증 실패 시 어떻게 할 것인가"가 검증 자체보다 중요하다**는 것이다. 검증이 실패했을 때 선택지는 세 가지다:

1. **Hard stop(경성 중단)**: 파이프라인을 즉시 중단한다. 후속 변환과 적재를 하지 않는다. 데이터 정합성이 최우선인 경우(예: 재무 데이터)에 사용한다.
2. **Soft warning(연성 경고)**: 알림을 보내지만 파이프라인은 계속 실행한다. "약간의 데이터 이상보다 데이터 부재가 더 나쁜" 경우에 사용한다.
3. **Quarantine(격리)**: 문제 있는 레코드만 별도 테이블로 분리하고, 나머지는 정상 처리한다.

초보자는 이 세 가지 전략을 혼동하거나, 모든 검증에 같은 전략을 적용하는 경향이 있다.

### 초보자의 함정: 검증 없이 파이프라인을 "완성"했다고 생각하기

가장 흔한 함정은 파이프라인의 "행복 경로(happy path)"만 구현하고 끝내는 것이다. 구체적인 시나리오를 보자.

당신이 외부 API에서 사용자 활동 데이터를 가져와 웨어하우스에 적재하는 파이프라인을 만들었다. 잘 동작한다. 2주 후, API 제공사가 버전 업데이트를 하면서 `user_id` 필드를 `userId`로 바꿨다. 파이프라인은 에러 없이 실행된다 -- 다만 `user_id` 컬럼에 모든 값이 NULL로 들어갈 뿐이다. NULL이 기존 스키마에서 허용되기 때문에 적재 단계에서도 에러가 발생하지 않는다. 이 문제가 발견되는 것은 3주 후 분석가가 "왜 최근 3주간 활성 사용자가 0명이죠?"라고 물었을 때다. 3주간의 데이터가 사실상 쓸모없어진 것이다.

이 시나리오를 방지하려면 **NOT NULL 검증**(핵심 필드가 NULL이 아닌지), **row count 검증**(적재된 행 수가 기대 범위 내인지), **freshness 검증**(데이터의 최신 타임스탬프가 기대 범위 내인지)을 파이프라인에 내장했어야 한다.

### 현장 경험 법칙

**"데이터 품질 검증의 80%는 세 가지로 커버된다: row count, NULL rate, freshness."** 이 세 가지 기본 검증만으로 대부분의 심각한 데이터 사고를 사전에 탐지할 수 있다. 복잡한 통계적 이상 탐지는 그 다음이다. 이 법칙이 깨지는 경우는 정밀한 수치 정확도가 요구되는 금융 데이터다 -- 이 경우 합계 검증(source와 target의 총합 비교), 분포 검증 등 추가적인 검증이 필수다.

## 2.2 Schema Drift(스키마 드리프트): 소리 없이 다가오는 재앙

### 실무 용어와 뉘앙스

Schema drift(스키마 드리프트)는 소스 시스템의 데이터 구조가 사전 통보 없이 변경되는 현상이다. 새 컬럼이 추가되거나, 기존 컬럼의 타입이 바뀌거나, 필드명이 변경되거나, 컬럼이 삭제되는 모든 경우를 포함한다.

현장에서 schema drift가 특히 문제가 되는 이유는, 소스 시스템 팀과 데이터 팀이 보통 다른 조직에 속해 있기 때문이다. 백엔드 팀이 프로덕션 데이터베이스의 테이블에 컬럼을 추가할 때, 하류의 데이터 파이프라인에 미치는 영향을 고려하지 않는 경우가 허다하다. "우리 API에는 영향 없으니까"라는 논리다.

**Schema-on-read(읽기 시 스키마 적용)**와 **schema-on-write(쓰기 시 스키마 적용)**는 schema drift 대응 전략과 직결된 개념이다. Data lake(데이터 레이크)는 전형적인 schema-on-read 시스템으로, 데이터를 적재할 때는 스키마를 강제하지 않고 읽을 때 해석한다. Data warehouse(데이터 웨어하우스)는 전형적인 schema-on-write 시스템으로, 미리 정의된 스키마에 맞지 않는 데이터는 적재 자체가 거부된다.

### 초보자의 함정: 스키마가 영원히 고정되어 있다고 가정하기

구체적인 시나리오를 보자. 당신의 파이프라인이 PostgreSQL의 `orders` 테이블을 CDC(Change Data Capture, 변경 데이터 캡처)로 읽어온다. 백엔드 팀이 배포하면서 `orders` 테이블에 `discount_type`이라는 ENUM 컬럼을 추가했다. 당신의 CDC 파이프라인은 새 컬럼을 감지하지만, 목적지 웨어하우스의 테이블에는 해당 컬럼이 없어 적재가 실패한다. 이것은 "좋은" 실패다 -- 최소한 문제를 즉시 인지할 수 있으니까.

"나쁜" 실패는 이렇다. 백엔드 팀이 `price` 컬럼의 타입을 `integer`(센트 단위)에서 `decimal`(달러 단위)로 바꿨다. 컬럼명은 같고 타입도 호환되므로 파이프라인은 에러 없이 실행된다. 그러나 "1500"이 "$15.00"이 아니라 "$1,500.00"으로 해석되면서, 매출 리포트가 100배 뻥튀기된다. 이런 의미론적 스키마 변경(semantic schema change)은 탐지하기 가장 어렵고 피해가 가장 크다.

### 현장 경험 법칙

**"소스 시스템을 소유하지 않는 한, 스키마는 언제든 바뀐다고 가정하라."** 이에 대한 방어는 다층적이어야 한다: 스키마 변경 감지 자동화, 핵심 필드의 타입/범위 검증, 소스 팀과의 변경 알림 프로세스 합의. 이 법칙이 "과잉"인 경우는 소스와 목적지를 같은 팀이 소유할 때다 -- 이 경우 코드 리뷰 프로세스에서 스키마 변경 영향을 함께 검토할 수 있다.

## 2.3 Data Contracts(데이터 계약): 조직적 해결책

### 실무 용어와 뉘앙스

Data contract(데이터 계약)은 최근 2~3년간 데이터 엔지니어링 커뮤니티에서 급부상한 개념이다. 이것은 소스 시스템(producer, 생산자)과 데이터 파이프라인(consumer, 소비자) 사이에 "이 데이터의 스키마, 의미, SLA를 보장합니다"라는 명시적 합의를 뜻한다.

현장에서 data contract는 기술적 문제가 아니라 **조직적 문제**다. YAML이나 Protobuf로 스키마를 정의하는 것은 쉽다. 어려운 것은 백엔드 팀이 "스키마를 바꾸기 전에 데이터 팀에 알리겠다"는 프로세스를 실제로 지키게 만드는 것이다. 이것은 기술이 아니라 문화의 영역이며, 경영진의 지원이 필요한 경우가 많다.

### 숨겨진 전제 조건

Data contract가 효과적으로 작동하려면 "계약 위반을 자동으로 감지하고 차단하는 시스템"이 있어야 한다는 것을 전문가는 전제한다. 문서로만 존재하는 contract는 코드 없는 API 스펙과 같다 -- 아무도 안 지킨다. Contract를 CI/CD 파이프라인에 통합해서, 소스 시스템의 스키마 변경이 contract를 위반하면 배포 자체를 차단하는 수준까지 가야 실효성이 있다.

# 3단계: Data Storage Architecture -- 웨어하우스, 레이크, 레이크하우스

## 3.1 핵심 아키텍처 패턴의 실무적 이해

### 실무 용어와 뉘앙스

**Data Warehouse(데이터 웨어하우스)**는 "구조화된 데이터를 분석에 최적화된 형태로 저장하는 시스템"이다. Snowflake, BigQuery, Redshift가 대표적이다. 핵심 특성은 schema-on-write, 즉 데이터를 넣을 때 반드시 미리 정의된 스키마를 따라야 한다는 것이다. 이것은 제약이자 장점이다 -- 스키마가 강제되므로 데이터의 일관성이 보장된다.

현장에서 웨어하우스를 선택할 때 가장 중요한 고려 사항은 **compute-storage separation(컴퓨팅-스토리지 분리)** 여부다. Snowflake와 BigQuery는 이 분리가 완전히 이루어져 있어, 데이터를 저장만 해놓고 쿼리할 때만 컴퓨팅 비용을 지불할 수 있다. Redshift는 역사적으로 이 분리가 불완전했으나 Redshift Serverless로 개선되고 있다. 이 분리가 중요한 이유는 비용 모델 때문이다 -- 데이터 양은 계속 증가하지만 쿼리 빈도는 비례하여 증가하지 않으므로, 스토리지와 컴퓨팅을 독립적으로 확장할 수 있어야 비용 효율적이다.

**Data Lake(데이터 레이크)**는 "모든 형태의 데이터를 원시 형태로 저장하는 저장소"이다. 보통 S3, GCS, Azure Blob Storage 같은 오브젝트 스토리지 위에 구축된다. Schema-on-read 방식으로, 데이터를 넣을 때는 어떤 형태든 받아들이고 읽을 때 해석한다. Parquet, ORC, Avro 같은 파일 포맷으로 저장된다.

현장에서 data lake가 실제로 의미하는 바는 "웨어하우스에 넣기 전 또는 넣지 않을 데이터의 장기 저장소"다. 이미지, 동영상, 로그 파일, IoT 센서 데이터 같은 비정형/반정형 데이터는 웨어하우스에 넣기 부적합하므로 data lake에 저장한다. 구조화된 데이터도 raw layer로서 data lake에 저장하는 것이 일반적이다.

**Data Lakehouse(데이터 레이크하우스)**는 data lake의 유연성과 warehouse의 성능/트랜잭션 보장을 결합하려는 아키텍처다. Delta Lake(Databricks), Apache Iceberg, Apache Hudi가 이 범주의 기술이다. 이들은 data lake의 오브젝트 스토리지 위에 ACID transaction(원자성/일관성/고립성/지속성 트랜잭션)과 스키마 관리 기능을 추가한다.

### 숨겨진 전제 조건: Medallion Architecture(메달리온 아키텍처)

전문가들이 저장소 아키텍처를 논의할 때 암묵적으로 전제하는 것은 **데이터가 여러 "계층"을 거쳐 정제된다**는 것이다. 가장 일반적인 패턴이 Medallion Architecture(메달리온 아키텍처)다:

**Bronze layer(브론즈 계층)**: 소스 시스템에서 가져온 raw data를 있는 그대로 저장한다. 어떤 변환도 하지 않는다. 이 계층의 존재 이유는 "원본 증거 보존"과 "재처리 가능성 보장"이다. 데이터 포맷은 소스에 따라 JSON, CSV, Parquet 등 다양하다.

**Silver layer(실버 계층)**: Bronze 데이터를 정제(cleansed), 표준화(standardized), 중복 제거(deduplicated)한 결과를 저장한다. 이 계층에서 schema enforcement(스키마 강제)가 이루어진다. 여러 소스의 데이터를 조인하거나 통합하는 작업도 이 계층에서 수행된다.

**Gold layer(골드 계층)**: 비즈니스 로직이 적용된 최종 분석용 데이터를 저장한다. Dimensional model(차원 모델)로 구성된 fact/dimension 테이블이 이 계층에 해당한다. 대시보드와 리포트가 직접 참조하는 데이터다.

이 계층 구조를 모르는 초보자가 "데이터를 어디에 저장하지?"라고 물으면, 전문가는 "어떤 계층을 말하는 거지?"라고 되묻는다. 이 간극이 커뮤니케이션 단절의 원인이 된다.

### 초보자의 함정: "Data Lake를 만들었으니 분석할 수 있다"는 착각

가장 흔하면서도 치명적인 함정이다. 구체적인 시나리오를 보자.

팀이 "모든 데이터를 한 곳에 모으자"며 S3 기반 data lake를 구축한다. 각 팀에서 데이터를 JSON, CSV, Parquet 등 제각각의 형식으로 S3에 올린다. 폴더 구조도 `team_name/data_source/date/` 같은 비표준적 형태다. 3개월 후, 분석가가 이 데이터를 사용하려 하지만 문제가 산적하다: 같은 고객을 A팀은 `customer_id`, B팀은 `cust_id`, C팀은 `user_id`로 부른다. 날짜 형식도 팀마다 다르다. 카탈로그(데이터가 어디에 뭐가 있는지)도 없다. 결국 이 data lake는 **data swamp(데이터 늪)**이 된다 -- 데이터는 있지만 아무도 쓸 수 없는 상태.

이 시나리오는 2015~2018년 빅데이터 붐 시기에 수많은 기업에서 실제로 벌어진 일이다. Data lake는 "저장"만 해결할 뿐, "발견 가능성(discoverability)", "일관성(consistency)", "접근성(accessibility)"은 별도로 해결해야 한다.

### 현장 경험 법칙

**"90%의 분석 워크로드에는 클라우드 데이터 웨어하우스가 정답이다."** Data lake가 필요한 경우는 비정형 데이터가 많거나, 데이터 양이 극히 방대하여 웨어하우스 비용이 감당 불가한 경우, 또는 ML 워크로드가 핵심인 경우다. "혹시 나중에 필요할까 봐" data lake부터 구축하는 것은 과잉 설계(over-engineering)다. 이 법칙이 깨지는 경우는 IoT, 게임 로그, 광고 기술(ad tech) 같이 초당 수십만 이벤트가 발생하는 환경이다 -- 이 경우 웨어하우스에 직접 적재하면 비용이 폭발하므로 data lake를 경유해야 한다.

**"Lakehouse를 선택할 때는 팀의 Spark 역량을 먼저 확인하라."** Delta Lake, Iceberg 등 lakehouse 기술은 대부분 Spark 생태계에 뿌리를 두고 있다. 팀에 Spark를 다룰 수 있는 엔지니어가 없다면, lakehouse를 도입해도 운영할 수 없다. BigQuery나 Snowflake가 SQL만으로 대부분의 분석 요구를 충족할 수 있으므로, 불필요하게 기술 스택을 복잡하게 만들지 않는 것이 현명하다. 이 법칙이 깨지는 경우는 "time travel(시간 여행)"이나 "schema evolution(스키마 진화)" 같은 lakehouse 고유 기능이 핵심 요구사항일 때다 -- 다만 Snowflake와 BigQuery도 이런 기능을 자체적으로 지원하기 시작했으므로, 기능 격차는 점점 줄어들고 있다.

# 4단계: Data Modeling -- 차원 모델링, 팩트/디멘션 테이블

## 4.1 Dimensional Modeling(차원 모델링)의 실무적 이해

### 실무 용어와 뉘앙스

**Dimensional modeling(차원 모델링)**은 Ralph Kimball이 체계화한 데이터 웨어하우스 설계 방법론으로, 분석 쿼리에 최적화된 데이터 구조를 만드는 것이 목적이다. 백엔드 개발자가 익숙한 3NF(Third Normal Form, 제3정규형) 설계와 근본적으로 다른 철학을 가진다.

3NF는 데이터 중복을 최소화하는 것이 목표다 -- 같은 정보가 여러 곳에 저장되면 일관성 문제가 생기기 때문이다. 이것은 OLTP(Online Transaction Processing, 온라인 트랜잭션 처리) 시스템에서 올바른 접근이다. 그러나 분석 환경(OLAP, Online Analytical Processing)에서는 반대다. 분석 쿼리는 여러 테이블을 조인해야 하는데, 테이블이 지나치게 정규화되어 있으면 조인 횟수가 늘어나 쿼리 성능이 떨어지고 쿼리 작성도 복잡해진다.

차원 모델링은 의도적으로 데이터를 비정규화(denormalize)한다. 데이터를 **fact table(팩트 테이블)**과 **dimension table(디멘션 테이블)**로 나누는 것이 핵심이다.

**Fact table(팩트 테이블)**은 비즈니스에서 "발생한 사건"을 기록한다. 주문, 클릭, 거래, 배송 등이 대표적이다. 팩트 테이블의 행은 하나의 "이벤트" 또는 "측정값"을 나타낸다. 핵심 특성은 세 가지다: (1) 숫자형 측정값(measure)을 포함한다(주문 금액, 클릭 수, 거래액), (2) 일반적으로 가장 큰 테이블이다(이벤트는 계속 쌓이므로), (3) 여러 dimension table을 참조하는 foreign key(외래 키)를 가진다.

**Dimension table(디멘션 테이블)**은 fact를 "설명하는 맥락"을 제공한다. 고객 정보, 상품 정보, 날짜 정보, 지역 정보 등이 대표적이다. 디멘션 테이블의 행은 하나의 "엔티티"를 나타낸다. 핵심 특성은 세 가지다: (1) 텍스트형 설명 속성(attribute)을 포함한다(고객명, 상품 카테고리, 지역명), (2) fact table에 비해 상대적으로 작다, (3) 분석의 "필터"와 "그룹핑" 기준이 된다.

### 숨겨진 전제 조건: Star Schema와 Snowflake Schema

전문가가 차원 모델을 논의할 때 전제하는 것은 **star schema(스타 스키마)**라는 배치 패턴이다. 중앙에 fact table이 있고, 주변에 dimension table들이 방사형으로 연결된 형태가 별(star) 모양이라 하여 이렇게 부른다.

**Snowflake schema(스노우플레이크 스키마)**는 star schema에서 dimension table이 추가로 정규화된 형태다. 예를 들어, `dim_product` 테이블이 `dim_category`와 `dim_brand`로 더 세분화되는 경우다. 이론적으로는 저장 공간을 절약하지만, 실무에서는 star schema가 압도적으로 선호된다. 이유는 단순하다 -- 분석가가 쿼리를 쉽게 작성할 수 있어야 하고, 조인이 많아지면 실수 확률과 쿼리 복잡도가 동시에 증가하기 때문이다.

### 초보자의 함정: OLTP 사고방식으로 분석용 테이블을 설계하기

이것은 백엔드 개발자가 데이터 모델링에 참여할 때 가장 많이 빠지는 함정이다. 구체적인 시나리오를 보자.

백엔드 개발자인 당신이 주문 분석을 위한 웨어하우스 테이블을 설계해달라는 요청을 받았다. OLTP 경험을 바탕으로, `orders`, `order_items`, `products`, `categories`, `customers`, `addresses`, `cities`, `countries` 등 완벽하게 정규화된 10개 테이블을 설계한다. 각 테이블은 3NF를 만족하며, 중복 데이터가 전혀 없다.

그런데 분석가가 "지난 분기 국가별/카테고리별 매출 추이"를 보려면 7개 테이블을 조인해야 한다. 쿼리가 30줄이 넘고, 성능도 느리다. 분석가가 조인 조건 하나를 잘못 써서 매출이 2배로 뻥튀기되는 사고가 발생한다(one-to-many 조인에서의 fan-out 문제). 결국 분석가들은 이 테이블을 직접 쓰지 않고, 자기들끼리 비공식적으로 "분석용 요약 테이블"을 만들기 시작한다.

차원 모델링 접근이었다면 `fact_orders`(주문 팩트)와 `dim_customer`, `dim_product`, `dim_date` 정도로 설계하고, `dim_product`에 카테고리명을 비정규화하여 포함시켰을 것이다. 분석가는 2~3개 테이블만 조인하면 대부분의 분석이 가능하다.

### 현장 경험 법칙

**"분석용 테이블 설계의 목표는 '분석가가 쿼리를 잘못 쓰기 어렵게 만드는 것'이다."** 테이블 수가 적고, 조인 관계가 단순하고, 팩트와 디멘션의 관계가 명확할수록 분석가의 실수 확률이 줄어든다. 저장 공간의 중복은 클라우드 환경에서 무시 가능한 비용이지만, 잘못된 분석 결과는 비즈니스 의사결정에 직접적 피해를 준다. 이 법칙이 깨지는 경우는 dimension의 변경 빈도가 극히 높을 때다 -- 예를 들어 상품 가격이 하루에 수십 번 바뀌는 동적 가격 책정(dynamic pricing) 환경에서는 비정규화된 디멘션이 데이터 불일치를 야기할 수 있으므로, 정규화가 오히려 나을 수 있다.

## 4.2 Slowly Changing Dimensions(SCD, 천천히 변하는 차원)

### 실무 용어와 뉘앙스

SCD(Slowly Changing Dimension, 천천히 변하는 차원)는 "디멘션의 속성이 시간에 따라 변할 때 이를 어떻게 처리할 것인가"의 문제다. 가장 흔한 예는 고객의 주소 변경이다. 고객 A가 서울에서 부산으로 이사했을 때, `dim_customer` 테이블에서 어떻게 처리할 것인가?

**SCD Type 1**: 그냥 덮어쓴다. 고객 A의 주소를 "부산"으로 업데이트한다. 과거 이력은 사라진다. "현재 상태만 중요하고 과거 이력은 필요 없다"는 경우에 사용한다. 구현이 가장 단순하다.

**SCD Type 2**: 새 행을 추가한다. 고객 A의 기존 행을 "비활성(inactive)"으로 표시하고(`is_current = false`, `end_date = '2024-03-14'`), 새 행을 추가한다(`is_current = true`, `start_date = '2024-03-15'`). 이렇게 하면 "이 고객이 서울에 살 때 한 주문"과 "부산으로 이사한 후의 주문"을 정확히 구분할 수 있다. 구현이 복잡하지만, 정확한 분석을 위해 가장 많이 사용된다.

**SCD Type 3**: 이전 값과 현재 값을 별도 컬럼으로 보관한다(`current_city = '부산'`, `previous_city = '서울'`). 변경 이력이 1단계만 필요할 때 사용하지만, 실무에서는 거의 쓰이지 않는다.

현장에서 "SCD"라고 하면 거의 항상 Type 2를 의미한다. Type 1은 너무 단순하여 별도로 언급하지 않고, Type 3은 실용성이 낮아 거의 사용하지 않기 때문이다.

### 숨겨진 전제 조건

SCD Type 2를 구현할 때 전문가가 당연히 전제하는 것은 **surrogate key(대리 키)**의 사용이다. 고객 A의 자연 키(natural key)가 `customer_id = 123`이라 하자. SCD Type 2에서는 같은 고객이 2개 이상의 행을 가지므로, `customer_id`만으로는 행을 유일하게 식별할 수 없다. 따라서 `customer_sk`(surrogate key)라는 별도의 인조 키를 생성하여 각 행을 유일하게 식별한다. Fact table은 `customer_id`가 아니라 `customer_sk`로 dimension을 참조한다.

이것을 모르는 초보자가 fact table에 `customer_id`로 dimension을 조인하면, 같은 고객의 여러 SCD 행과 조인되어 데이터가 중복되는(fan-out) 심각한 버그가 발생한다.

### 초보자의 함정: 모든 디멘션에 SCD Type 2를 적용하기

SCD Type 2의 중요성을 배운 초보자는 "모든 디멘션을 Type 2로 하면 안전하겠지"라고 생각하기 쉽다. 그러나 Type 2는 비용이 크다. 첫째, 디멘션 테이블 크기가 변경 이력만큼 증가한다. 둘째, 조인 로직이 복잡해진다(유효 기간 기반 조인). 셋째, 파이프라인의 변환 로직이 복잡해진다.

실무에서는 "분석에 과거 이력이 실제로 필요한 속성"만 Type 2로 관리하고, 나머지는 Type 1로 처리하는 hybrid(하이브리드) 접근이 일반적이다. 예를 들어, 고객의 주소는 "지역별 매출 분석"에 과거 이력이 필요하므로 Type 2, 고객의 전화번호는 분석에 쓰이지 않으므로 Type 1로 처리한다.

# 5단계: Data Ingestion -- 데이터 수집의 실제

이 단계는 위에서 파이프라인 설계만큼 깊이 다루지 않지만, 핵심적인 암묵지를 짚는다.

## 5.1 수집 패턴의 실무적 이해

### 실무 용어와 뉘앙스

**Batch ingestion(배치 수집)**은 정해진 주기(시간별, 일별)로 데이터를 일괄 가져오는 방식이다. **Full load(전체 적재)**는 매번 소스 데이터 전체를 가져오는 것이고, **incremental load(증분 적재)**는 마지막 수집 이후 변경된 데이터만 가져오는 것이다.

현장에서 초보자가 놓치는 것은 incremental load의 전제 조건이다. 증분 적재가 가능하려면 소스 데이터에 "변경 시점을 식별할 수 있는 필드"(예: `updated_at` 타임스탬프)가 있어야 한다. 이 필드가 없으면 "이전 수집 이후 뭐가 바뀌었는지" 알 수 없으므로 full load를 할 수밖에 없다. 소스 시스템을 설계할 때 `updated_at` 컬럼을 넣는 것은 데이터 엔지니어링 관점에서 "공짜로 얻을 수 있는 가장 큰 이득"이다.

**CDC(Change Data Capture, 변경 데이터 캡처)**는 데이터베이스의 트랜잭션 로그(WAL, binlog)를 읽어 변경사항을 실시간으로 캡처하는 기술이다. Debezium이 대표적인 오픈소스 CDC 도구다. CDC는 incremental load의 가장 정확하고 효율적인 형태이지만, 운영 복잡도가 높다 -- 트랜잭션 로그를 읽으려면 소스 데이터베이스에 대한 관리자 수준 접근 권한이 필요하고, 로그 보존 기간 관리도 해야 한다.

### 현장 경험 법칙

**"가능하면 incremental, 불가피하면 full load, 요구사항이 실시간이면 CDC."** 이 순서가 복잡도와 비용의 상승 순서이기도 하다. "실시간이 필요한가?"라는 질문에 대부분의 비즈니스 사용자는 "예"라고 답하지만, 실제로 "15분 이내"면 충분한 경우가 대부분이다. 진정한 실시간(초 단위)이 필요한 유즈케이스는 사기 탐지, 실시간 추천, 실시간 모니터링 정도로 제한적이다.

# 6단계: Orchestration과 Observability -- 워크플로우 관리

## 6.1 Orchestration(오케스트레이션) 도구의 실무적 이해

### 실무 용어와 뉘앙스

**Orchestration(오케스트레이션)**은 "여러 파이프라인 작업을 올바른 순서와 조건으로 실행하는 것"이다. Apache Airflow가 사실상의 표준(de facto standard)이며, Dagster, Prefect가 차세대 대안으로 부상하고 있다.

현장에서 중요한 개념은 **DAG(Directed Acyclic Graph, 방향성 비순환 그래프)**이다. 파이프라인의 작업 간 의존 관계를 그래프로 표현한 것으로, "A가 끝나야 B를 시작하고, B와 C가 모두 끝나야 D를 시작한다" 같은 로직을 정의한다.

Airflow에서 초보자가 반드시 이해해야 할 개념은 **execution_date(실행 날짜)**의 의미다. 이것은 "파이프라인이 실제로 실행된 시간"이 아니라 "파이프라인이 처리하는 데이터의 논리적 날짜"이다. 예를 들어, 일일 파이프라인이 2024년 3월 16일 새벽 2시에 실행될 때, execution_date는 2024년 3월 15일이다 -- "3월 15일의 데이터를 처리한다"는 의미이기 때문이다. 이 개념을 잘못 이해하면 하루 어긋난 데이터를 처리하는 버그를 만들게 된다.

### 현장 경험 법칙

**"오케스트레이터에서 비즈니스 로직을 실행하지 마라."** Airflow DAG 안에 SQL 변환이나 데이터 처리 로직을 직접 넣는 것은 안티패턴이다. Airflow는 "언제, 어떤 순서로 실행할 것인가"만 관리해야 하고, 실제 처리 로직은 dbt, Spark, Python 스크립트 등 별도 도구에서 실행해야 한다. 이렇게 분리해야 오케스트레이터 교체(예: Airflow에서 Dagster로)가 가능하다.

# 7단계: Stream Processing -- 실시간 처리의 기초

이 영역은 가볍게 다루되, 백엔드 개발자에게 특히 중요한 개념만 짚는다.

## 7.1 핵심 개념

**Event streaming(이벤트 스트리밍)**은 데이터를 개별 이벤트 단위로 실시간 처리하는 패턴이다. Apache Kafka가 이 영역의 사실상 표준이다. Kafka를 메시지 큐(message queue)로만 이해하는 백엔드 개발자가 많지만, 데이터 엔지니어링 맥락에서 Kafka는 "영속적 이벤트 로그(durable event log)"에 가깝다. RabbitMQ 같은 전통적 메시지 큐는 메시지가 소비되면 삭제되지만, Kafka는 설정된 보존 기간(retention period) 동안 메시지를 유지한다. 따라서 여러 소비자가 같은 이벤트를 독립적으로, 서로 다른 속도로 읽을 수 있다.

**Event time vs Processing time(이벤트 시간 vs 처리 시간)**은 스트림 처리에서 가장 미묘한 구분이다. Event time은 이벤트가 실제로 발생한 시간이고, processing time은 시스템이 이벤트를 처리한 시간이다. 모바일 앱에서 발생한 이벤트가 네트워크 지연으로 5분 늦게 도착할 수 있다. "최근 1시간 매출"을 계산할 때 event time 기준인지 processing time 기준인지에 따라 결과가 달라진다. 대부분의 경우 event time이 올바른 기준이지만, 늦게 도착하는(late-arriving) 이벤트를 처리하기 위한 **watermark(워터마크)** 메커니즘이 필요하다.

# 8단계: Data Governance와 Lineage -- 데이터 거버넌스와 계보

## 8.1 핵심 개념

**Data lineage(데이터 계보)**는 "이 데이터가 어디서 왔고, 어떤 변환을 거쳤는가"를 추적하는 것이다. 분석가가 대시보드의 숫자에 의문을 품었을 때 "이 숫자가 어떻게 계산된 건지?"를 추적할 수 있어야 한다. dbt는 자동으로 lineage graph(계보 그래프)를 생성해주는데, 이것이 dbt가 인기를 얻은 주요 이유 중 하나다.

**Data catalog(데이터 카탈로그)**는 "조직에 어떤 데이터가 있고, 각 데이터의 의미와 소유자가 누구인지"를 정리한 메타데이터 저장소다. DataHub, Atlan, Alation이 대표적인 도구다. 카탈로그 없는 조직에서는 "이 데이터 어디 있어요?"라는 질문이 Slack에서 끝없이 반복된다.

**PII(Personally Identifiable Information, 개인식별정보)** 관리는 거버넌스의 핵심 영역이다. 이름, 이메일, 전화번호, 주민등록번호 등이 파이프라인을 통해 분석 환경으로 무분별하게 복제되면 규정 위반(GDPR, 개인정보보호법) 위험이 발생한다. 실무에서는 masking(마스킹), hashing(해싱), 또는 접근 제어(row-level security)로 PII를 보호한다.

# 백엔드 개발자를 위한 도메인 간 브릿지

당신의 기존 경험을 데이터 엔지니어링 맥락으로 번역하면 다음과 같다.

**API의 request validation은 data quality check에 대응한다.** API에서 잘못된 입력을 거부하듯, 파이프라인에서 잘못된 데이터를 걸러내야 한다. 차이점은 API는 동기적으로 즉시 거부하지만, 파이프라인의 검증 실패는 비동기적으로 나중에 처리해야 하는 경우가 많다는 것이다.

**CI/CD 파이프라인은 데이터 파이프라인의 오케스트레이션에 대응한다.** GitHub Actions에서 "빌드 성공 후 테스트, 테스트 성공 후 배포"라는 의존 관계를 정의하듯, Airflow에서 "추출 성공 후 변환, 변환 성공 후 적재"라는 의존 관계를 정의한다.

**Database migration은 schema evolution에 대응한다.** Alembic이나 Flyway로 스키마 변경을 관리하듯, 데이터 웨어하우스에서도 스키마 변경을 버전 관리하고 점진적으로 적용해야 한다. 차이점은 웨어하우스의 스키마 변경은 수십 TB의 기존 데이터에 영향을 미칠 수 있으므로, migration의 비용과 리스크가 훨씬 크다는 것이다.

**Microservice 간 API contract은 data contract에 대응한다.** OpenAPI(Swagger) 스펙으로 서비스 간 인터페이스를 정의하듯, data contract로 소스 시스템과 데이터 파이프라인 간 인터페이스를 정의한다.

# 디자인 리뷰에서 물어야 할 질문들

데이터 엔지니어링 팀의 디자인 리뷰에 참여할 때, 다음 질문들은 당신이 도메인을 깊이 이해하고 있음을 보여줄 것이다:

**파이프라인 설계에 대해**: "이 파이프라인은 멱등적인가요? 같은 날짜를 두 번 실행하면 어떻게 되나요?" / "Backfill이 필요할 때 어떻게 처리하나요? 지난 30일을 재처리하려면 시간이 얼마나 걸리나요?"

**데이터 품질에 대해**: "이 파이프라인에 어떤 데이터 검증이 포함되어 있나요? 검증 실패 시 hard stop인가요, soft warning인가요?" / "소스 시스템의 스키마가 변경되면 어떻게 감지하나요?"

**저장소에 대해**: "이 데이터는 Medallion Architecture의 어느 계층에 해당하나요?" / "Raw data는 얼마나 오래 보존하나요?"

**모델링에 대해**: "이 디멘션은 SCD Type 1인가요, Type 2인가요? 과거 이력 분석이 필요한 속성은 어떤 건가요?"

이러한 질문들은 단순히 지식을 과시하는 것이 아니라, 파이프라인의 신뢰성과 유지보수성에 직결되는 핵심 설계 결정을 확인하는 것이다. 데이터 엔지니어링 팀은 이런 질문을 할 줄 아는 협업자를 높이 평가한다.
