# Prime Thinking Session: 데이터 엔지니어링

> 도메인 오리엔테이션 맵을 기반으로 학습 전 사고를 활성화하는 질문-응답 세션

---

## Q1. 백엔드 API와 데이터 파이프라인의 근본적 차이는 무엇일까?

**질문 의도**: 맵에서 "API 요청/응답 흐름과 파이프라인의 입력/변환/출력 흐름이 같은 패턴"이라고 했는데, 그렇다면 왜 별도의 분야가 필요할까요? 당신이 아는 백엔드 API와 데이터 파이프라인 사이에서 어떤 차이가 결정적이라고 생각하나요?

**예상 답변**: "API는 실시간 요청-응답이고, 파이프라인은 배치 처리라서 다르지 않을까요?"

**확장 피드백**: 맞는 방향입니다. 하지만 더 근본적인 차이가 있습니다. API는 하나의 요청이 실패하면 그 요청만 영향받지만, 파이프라인은 하나의 단계가 실패하면 하류(downstream) 전체가 멈춥니다. 맵에서 전문가 인사이트로 "파이프라인은 코드가 아니라 시스템이다"라고 한 이유가 이것입니다. API 서버를 만들 때는 "이 엔드포인트가 잘 동작하는가"를 생각하지만, 파이프라인을 만들 때는 "이것이 실패하면 어떻게 복구하는가"를 먼저 생각해야 합니다. 이 관점 전환이 학습의 첫 번째 열쇠입니다.

---

## Q2. "Idempotency"가 데이터 파이프라인에서 왜 핵심 용어일까?

**질문 의도**: 맵의 Area 1 핵심 용어에 idempotency가 포함되어 있습니다. 백엔드 개발에서도 이 개념을 알고 있을 텐데, 데이터 파이프라인에서 이것이 특별히 중요한 이유는 무엇일까요?

**예상 답변**: "같은 작업을 여러 번 실행해도 결과가 같다는 뜻인데... 파이프라인이 자주 재실행되니까 중요한 건가요?"

**확장 피드백**: 정확합니다. API에서 idempotency는 "중복 요청 방지" 정도의 의미이지만, 데이터 파이프라인에서는 생존 전략입니다. 파이프라인은 네트워크 오류, 메모리 부족, 소스 시스템 장애 등으로 일상적으로 실패합니다. 이때 "그냥 다시 돌리면 된다"고 말할 수 있으려면 idempotency가 보장되어야 합니다. 만약 보장되지 않으면? 데이터가 중복 적재되거나, 일부만 처리된 상태로 남거나, 하류 시스템이 잘못된 집계 결과를 보여줍니다. 학습할 때 "이 파이프라인은 idempotent한가?"를 항상 자문해보세요.

## Q3. 데이터 웨어하우스와 데이터 레이크의 차이를 식당에 비유한다면?

**질문 의도**: 맵의 Area 2에서 웨어하우스, 레이크, 레이크하우스를 선택해야 한다고 합니다. 이 세 가지를 자신만의 비유로 설명해볼 수 있나요?

**예상 답변**: "웨어하우스는 정리된 창고 같고, 레이크는 원재료를 그냥 쌓아둔 곳 같은 느낌인데..."

**확장 피드백**: 좋은 비유입니다. 좀 더 확장하면 이렇습니다. 데이터 웨어하우스는 고급 레스토랑의 주방입니다 -- 재료가 들어오기 전에 손질(스키마 정의)을 마쳐야 하고, 모든 것이 레시피(쿼리)에 최적화되어 있습니다. 데이터 레이크는 대형 냉장 창고입니다 -- 뭐든 일단 넣어두고 나중에 필요할 때 꺼내 씁니다. 레이크하우스는 냉장 창고에 정리 시스템을 붙인 것입니다 -- 유연성과 구조를 동시에 가집니다. 맵에서 "schema-on-read vs schema-on-write"라는 용어가 바로 이 차이를 기술적으로 표현한 것입니다. 이것을 학습할 때 "우리 팀 데이터는 어떤 패턴에 더 가까울까?"를 생각해보세요.

---

## Q4. CI/CD와 Airflow는 정말 같은 개념일까?

**질문 의도**: 맵에서 "오케스트레이션은 데이터 버전의 CI/CD"라고 했습니다. 둘 다 파이프라인을 자동으로 실행하는 도구인데, 본질적으로 같은 것일까요? 아니면 어디서 갈라지나요?

**예상 답변**: "CI/CD는 코드를 배포하는 거고, Airflow는 데이터 작업을 실행하는 거니까... 트리거가 다른 건가요?"

**확장 피드백**: 핵심을 짚었습니다. CI/CD는 "코드가 변경되면" 트리거되고, Airflow는 "시간이 되면" 또는 "데이터가 도착하면" 트리거됩니다. 하지만 더 큰 차이가 있습니다. CI/CD 파이프라인은 보통 한 방향으로 끝까지 달리지만, 데이터 오케스트레이터는 과거로 돌아가야 할 때가 있습니다. 이것이 맵에 나온 "backfill"입니다. 예를 들어 어제 파이프라인이 잘못된 로직으로 돌았다면, 수정 후 어제 데이터를 다시 처리해야 합니다. CI/CD에서는 과거 배포를 "다시 하는" 일이 거의 없지만, 데이터 세계에서는 일상입니다.

---

## Q5. "이 데이터 믿을 수 있어?"라는 질문에 어떻게 답할 수 있을까?

**질문 의도**: 맵에서 데이터 품질 관리를 최우선 학습 영역으로 꼽으면서 "데이터팀과의 대화에서 가장 흔한 질문"이라고 했습니다. 백엔드 개발자 입장에서 이 질문에 답하려면 무엇을 알아야 할까요?

**예상 답변**: "데이터가 null이 아닌지, 타입이 맞는지 정도 확인하면 되지 않을까요?"

**확장 피드백**: 그것은 첫 번째 레이어에 불과합니다. 맵의 핵심 용어를 보면 validation, schema drift, freshness, completeness, anomaly detection, data contract가 있습니다. 백엔드에서 API 스키마를 검증하는 것처럼, 데이터에도 "계약"이 필요합니다. 하지만 데이터의 독특한 점은 시간 차원입니다. 전문가 인사이트에서 "정확한 데이터보다 언제 기준의 데이터인지가 더 중요하다"고 한 이유입니다. 오늘 조회한 매출 데이터가 어제 집계된 것인지, 한 시간 전 집계된 것인지에 따라 의사결정이 달라집니다. "freshness"라는 용어가 이 개념을 담고 있습니다.

## Q6. 배치 처리와 스트림 처리 중 하나만 선택해야 한다면?

**질문 의도**: 맵에서 실시간 처리를 "나중에 배울 것"으로 분류했지만, 요즘 실시간이 트렌드라고 들었을 수도 있습니다. 왜 배치를 먼저 배우는 것이 맞을까요?

**예상 답변**: "실시간이 더 어렵고, 배치로도 대부분의 일을 할 수 있어서?"

**확장 피드백**: 맞습니다. 그리고 한 가지 더 -- 현실에서 대부분의 데이터 워크로드는 여전히 배치입니다. 실시간 처리는 "지연 시간이 중요한 유스케이스"에만 필요하다고 맵에서 명시했습니다. 배치 파이프라인을 제대로 이해하지 못한 채 스트리밍으로 넘어가면, 결국 "느린 스트리밍 파이프라인" -- 즉 배치보다 복잡하면서 배치만큼 느린 시스템을 만들게 됩니다. 맵에서 "상호 보완적"이라고 한 이유는, 많은 현실 아키텍처가 실시간 데이터를 수집한 뒤 배치로 정제하는 하이브리드 패턴을 쓰기 때문입니다.

## Q7. "스키마 변경이 데이터 엔지니어의 가장 큰 적"이라면, 백엔드에서 비슷한 경험은?

**질문 의도**: 맵의 전문가 인사이트 중 하나입니다. 백엔드 개발에서 비슷한 고통을 겪어본 적이 있나요? 그 경험과 연결지어 생각해보세요.

**예상 답변**: "API 버전 변경이나 DB 마이그레이션 할 때 비슷한 느낌이 들었던 것 같아요."

**확장 피드백**: 아주 좋은 연결입니다. DB 마이그레이션은 자기가 통제하는 스키마를 변경하는 것이지만, 데이터 엔지니어링에서 스키마 변경은 남이 통제하는 스키마가 바뀌는 것입니다. 외부 API의 응답 형식이 예고 없이 바뀌거나, 업스트림 팀이 테이블 컬럼을 추가/삭제하면, 파이프라인이 조용히 깨집니다. 에러가 나면 차라리 낫고, 스키마가 변해도 파이프라인은 돌아가는데 결과가 틀리는 경우가 가장 위험합니다. 이것이 맵에서 "data contract"와 "schema drift"라는 용어가 나온 배경입니다.

## Q8. Star Schema는 왜 "별"이라고 부를까?

**질문 의도**: 맵의 Area 6에서 dimensional modeling의 핵심으로 star schema를 언급했습니다. 이름에서 구조를 유추해볼 수 있나요?

**예상 답변**: "가운데 테이블이 있고 주변에 여러 테이블이 연결되어서 별 모양이 되는 건가요?"

**확장 피드백**: 정확합니다. 가운데에 fact table(매출, 주문 같은 이벤트 데이터)이 있고, 주변에 dimension table(고객, 상품, 날짜 같은 설명 데이터)이 별처럼 연결됩니다. 백엔드의 정규화된 DB 설계와 비교하면 의도적으로 중복(denormalization)을 허용합니다. 왜? JOIN을 줄여서 분석 쿼리 성능을 높이기 위해서입니다. 맵에서 "좋은 모델링은 복잡한 비즈니스 질문을 단순한 쿼리로 바꾼다"고 한 것이 이 의미입니다. OLTP(트랜잭션)와 OLAP(분석)의 설계 철학이 다르다는 것을 이해하는 것이 핵심입니다.

## Q9. 완벽한 파이프라인 vs 관찰 가능한 파이프라인 -- 왜 후자가 이기나?

**질문 의도**: 맵의 마지막 전문가 인사이트입니다. "완벽한 시스템"을 만들려는 것은 개발자의 본능인데, 왜 그것보다 관찰 가능성(observability)이 프로덕션에서 더 중요할까요?

**예상 답변**: "완벽한 건 불가능하니까 문제가 생겼을 때 빨리 발견하는 게 더 현실적이라서?"

**확장 피드백**: 핵심을 정확히 짚었습니다. 여기에 데이터 특유의 이유를 추가하면 -- 데이터 파이프라인의 입력은 통제할 수 없습니다. 외부 데이터 소스, 사용자 행동, 서드파티 API 등이 예측 불가능하게 변합니다. 아무리 완벽한 코드를 짜도 입력이 바뀌면 깨집니다. 그래서 전문가들은 "깨지지 않는 파이프라인"이 아니라 "깨졌을 때 즉시 아는 파이프라인"을 만듭니다. 구체적으로는 데이터 건수 모니터링, 처리 시간 추적, 이상치 알림 등을 파이프라인 코드 자체에 내장합니다. 백엔드에서 APM(Application Performance Monitoring)을 다뤄봤다면, 그 경험이 여기서 직접적으로 적용됩니다.
