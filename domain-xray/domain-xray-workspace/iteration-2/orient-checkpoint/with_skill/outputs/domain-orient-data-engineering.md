# 데이터 엔지니어링 오리엔테이션 맵

## 이 맵의 용도
이 맵은 데이터 엔지니어링이라는 분야의 전체 지형을 한눈에 보여주는 학습 뼈대입니다. 세부 지식을 쌓기 전에 먼저 이 구조를 머릿속에 넣어두면, 이후에 배우는 개념들이 자연스럽게 제자리를 찾아갑니다. 데이터팀과 대화할 때 "지금 이 사람이 어떤 영역 이야기를 하는 건지"를 파악하는 나침반으로 활용하세요.

## 당신의 앵커 포인트

당신은 이미 데이터 엔지니어링의 핵심 사고방식 상당 부분을 갖추고 있습니다. 다만 이름과 맥락이 다를 뿐입니다.

- **백엔드 API 설계 경험 → 데이터 파이프라인 설계**: API에서 요청을 받아 처리하고 응답을 돌려주는 흐름을 설계해본 적 있죠? 데이터 파이프라인도 본질적으로 같습니다 — 데이터를 소스에서 받아서, 변환하고, 목적지로 보내는 흐름입니다. 입력/처리/출력 패턴을 이미 체화하고 있습니다.
- **SQL 사용 경험 → 데이터 변환(Transformation)**: SELECT, JOIN, GROUP BY로 데이터를 조회하고 가공해본 경험은 ETL에서 가장 핵심인 T(Transform) 단계와 직결됩니다. 데이터 엔지니어가 매일 하는 일의 상당 부분이 SQL 기반 변환입니다.
- **Python 사용 경험 → 파이프라인 코드 작성**: 데이터 엔지니어링의 주력 언어가 Python입니다. Airflow, dbt, Spark 등 핵심 도구 대부분이 Python 생태계 안에 있습니다.
- **서버 배포/운영 경험 → 스케줄링과 모니터링**: 백엔드에서 cron job 설정하거나, 서버 로그 모니터링하거나, 장애 대응한 경험이 있다면 — 데이터 파이프라인의 스케줄링과 모니터링도 같은 감각입니다.
- **DB 스키마 설계 경험 → 데이터 모델링**: 테이블 설계하고, 정규화하고, 인덱스 걸어본 경험은 데이터 웨어하우스 모델링의 기초와 맞닿아 있습니다.

## 도메인 랜드스케이프

### 영역 1: 데이터 파이프라인 설계
**무엇을 하는가**: 다양한 소스(DB, API, 파일, 이벤트 스트림)에서 데이터를 추출해서, 분석이나 서비스에 쓸 수 있는 형태로 목적지까지 옮기는 전체 흐름을 설계합니다. 배치(정기적으로 모아서 처리)와 실시간(이벤트가 발생할 때마다 처리) 두 가지 패러다임이 있고, 대부분의 조직은 둘 다 씁니다.
**왜 중요한가**: 파이프라인이 없으면 데이터는 각 시스템에 갇혀 있어서 분석도, 의사결정도 할 수 없습니다. 데이터팀이 "파이프라인이 깨졌다"고 하면, 의사결정 자체가 멈춘다는 뜻입니다.
**핵심 용어**:
- **배치 처리(Batch Processing)**: 데이터를 모아서 정해진 주기(시간/일 단위)에 한꺼번에 처리하는 방식
- **스트리밍(Streaming)**: 데이터가 발생하는 즉시 처리하는 방식 (Kafka, Kinesis 등)
- **DAG (Directed Acyclic Graph)**: 파이프라인의 작업 순서를 표현하는 방향 있는 비순환 그래프. "이 작업이 끝나야 저 작업을 시작할 수 있다"를 정의
- **소스(Source) / 싱크(Sink)**: 데이터의 출발지와 도착지
**다른 영역과의 연결**: ETL/ELT의 상위 개념이며, 오케스트레이션으로 실행되고, 모니터링으로 관리됩니다.

### 영역 2: ETL/ELT 프로세스
**무엇을 하는가**: Extract(추출)-Transform(변환)-Load(적재)의 약자로, 데이터를 실제로 뽑고, 바꾸고, 넣는 구체적인 작업을 수행합니다. 전통적 ETL은 변환 후 적재하지만, 현대적 ELT는 먼저 적재한 뒤 목적지에서 변환합니다. 실무에서는 dbt 같은 도구로 SQL 기반 변환을 관리하는 ELT 패턴이 빠르게 확산 중입니다.
**왜 중요한가**: 원본 데이터는 거의 항상 분석에 바로 쓸 수 없는 상태입니다. 포맷이 다르고, 중복이 있고, 필드명이 제각각입니다. 변환 없이는 "쓸 수 있는 데이터"가 만들어지지 않습니다.
**핵심 용어**:
- **dbt (data build tool)**: SQL 기반으로 데이터 변환을 버전 관리하고 테스트하는 도구. 현대 데이터 엔지니어링의 핵심 도구
- **스테이징(Staging)**: 원본 데이터를 최소한의 가공만 해서 임시로 저장하는 중간 단계
- **Idempotent(멱등성)**: 같은 작업을 여러 번 실행해도 결과가 같은 성질. 파이프라인에서 매우 중요한 설계 원칙
- **스키마 진화(Schema Evolution)**: 소스 데이터의 구조가 변할 때 파이프라인이 깨지지 않도록 대응하는 것
**다른 영역과의 연결**: 파이프라인 설계의 구체적 실행이며, 데이터 품질 관리와 밀접합니다.

### 영역 3: 데이터 웨어하우스 & 스토리지
**무엇을 하는가**: 변환된 데이터를 분석에 최적화된 형태로 저장하고 관리합니다. Snowflake, BigQuery, Redshift 같은 클라우드 웨어하우스가 주류이며, 데이터 레이크(비정형 데이터 포함 대규모 저장소)와 레이크하우스(둘의 결합)라는 패턴도 있습니다. 테이블 구조를 어떻게 설계하느냐(차원 모델링)가 분석 성능과 직결됩니다.
**왜 중요한가**: 잘 설계된 웨어하우스가 없으면 분석가와 데이터 사이언티스트가 데이터를 찾고 이해하는 데 대부분의 시간을 소비합니다. "데이터는 있는데 쓸 수가 없다"는 상황의 원인이 대부분 여기에 있습니다.
**핵심 용어**:
- **차원 모델링(Dimensional Modeling)**: 팩트 테이블(수치 데이터)과 디멘전 테이블(설명 데이터)로 구조화하는 설계 방법론 (스타 스키마, 스노우플레이크 스키마)
- **데이터 레이크(Data Lake)**: 구조화/비구조화 데이터를 원본 그대로 저장하는 대규모 저장소
- **파티셔닝(Partitioning)**: 대용량 테이블을 날짜 등 기준으로 나눠 쿼리 성능을 높이는 기법
**다른 영역과의 연결**: ETL/ELT의 목적지이며, 데이터 품질 관리와 거버넌스의 대상입니다.

### 영역 4: 데이터 품질 & 거버넌스
**무엇을 하는가**: 데이터가 정확하고, 완전하고, 일관된지 검증하고, 누가 어떤 데이터에 접근할 수 있는지, 데이터의 출처와 흐름이 추적 가능한지(리니지)를 관리합니다. Great Expectations, Soda 같은 도구로 자동화된 품질 테스트를 수행합니다.
**왜 중요한가**: 품질이 나쁜 데이터로 만든 분석과 의사결정은 쓰레기입니다. "Garbage in, garbage out"은 데이터 엔지니어링에서 가장 현실적인 경구입니다. 거버넌스 없이는 개인정보 유출이나 규정 위반 리스크도 있습니다.
**핵심 용어**:
- **데이터 리니지(Data Lineage)**: 데이터가 어디서 왔고, 어떤 변환을 거쳤고, 어디로 갔는지 추적하는 것
- **데이터 카탈로그(Data Catalog)**: 조직 내 데이터 자산의 목록과 메타데이터를 관리하는 시스템
- **SLA (Service Level Agreement)**: 데이터 제공 시한과 품질 기준에 대한 약속
- **데이터 계약(Data Contract)**: 데이터를 제공하는 팀과 소비하는 팀 간의 스키마/품질 합의
**다른 영역과의 연결**: 모든 영역에 횡단적으로 적용되며, 특히 데이터팀과 백엔드팀의 협업 지점에서 핵심적입니다.

### 영역 5: 오케스트레이션 & 스케줄링
**무엇을 하는가**: 여러 파이프라인 작업을 언제, 어떤 순서로, 어떤 조건에서 실행할지 관리합니다. Apache Airflow가 사실상 표준이며, Dagster, Prefect 같은 차세대 도구도 성장 중입니다. 작업 간 의존성, 재시도 정책, 실패 처리를 정의합니다.
**왜 중요한가**: 파이프라인은 하나의 스크립트가 아니라 수십~수백 개의 작업이 엮인 복잡한 워크플로입니다. 오케스트레이션 없이는 "A가 끝나야 B를 돌릴 수 있는데 A가 실패했으면 어떻게 하지?"라는 문제를 수동으로 처리해야 합니다.
**핵심 용어**:
- **Airflow**: Python으로 DAG를 정의하는 가장 널리 쓰이는 오케스트레이션 도구
- **백필(Backfill)**: 과거 날짜의 데이터를 소급해서 다시 처리하는 것
- **의존성(Dependency)**: 작업 간 "이것이 끝나야 저것을 시작" 관계
- **SLA 미스(SLA Miss)**: 정해진 시간까지 파이프라인이 완료되지 않은 상황
**다른 영역과의 연결**: 파이프라인 설계를 실행하는 엔진이며, 모니터링과 쌍을 이룹니다.

### 영역 6: 모니터링 & 옵저버빌리티
**무엇을 하는가**: 파이프라인이 정상적으로 돌아가는지 실시간으로 감시하고, 문제가 생겼을 때 빠르게 원인을 찾을 수 있게 합니다. 작업 실패 알림, 데이터 지연 감지, 데이터 볼륨 이상 탐지 등을 포함합니다. 백엔드의 APM(Application Performance Monitoring)과 개념이 같습니다.
**왜 중요한가**: 파이프라인은 조용히 실패합니다. 에러가 나서 멈추는 것보다, 잘못된 데이터를 조용히 흘려보내는 "사일런트 실패"가 더 위험합니다. 문제를 얼마나 빨리 발견하느냐가 데이터 신뢰도를 결정합니다.
**핵심 용어**:
- **데이터 프레시니스(Freshness)**: 데이터가 얼마나 최신인지. "이 테이블은 마지막으로 3시간 전에 업데이트됨"
- **사일런트 실패(Silent Failure)**: 에러 없이 진행됐지만 결과 데이터가 틀린 상황
- **알러팅(Alerting)**: 이상 상황 발생 시 Slack, PagerDuty 등으로 알림 보내기
**다른 영역과의 연결**: 오케스트레이션과 데이터 품질 모두에 걸쳐 있으며, 운영 안정성의 마지막 방어선입니다.

## 학습 우선순위 맵
당신의 상황(백엔드 개발자로서 데이터팀과의 협업을 위해 데이터 엔지니어링의 기본을 빠르게 파악해야 함)에 맞춘 우선순위입니다:

**여기서 시작하세요** (당신의 목표에 가장 직접적으로 도움되는 영역):
- **데이터 파이프라인 설계** — 데이터팀이 하는 일의 큰 그림을 이해하는 것이 협업의 첫걸음입니다. "우리가 보내주는 데이터를 저쪽에서 어떻게 가져가서 쓰는지"를 이해하면 대화가 됩니다.
- **ETL/ELT 프로세스** — 데이터팀이 가장 많이 시간을 쓰는 영역이고, 백엔드에서 보내주는 데이터의 형태가 이 과정의 난이도를 직접 결정합니다. 여기를 이해하면 "우리 쪽에서 뭘 바꾸면 데이터팀이 덜 고생하는지"를 알 수 있습니다.

**다음으로 배우세요** (중요하지만 급하지 않은 영역):
- **데이터 품질 & 거버넌스** — 특히 "데이터 계약" 개념은 백엔드팀과 데이터팀 사이의 협업 인터페이스 그 자체입니다.
- **오케스트레이션 & 스케줄링** — Airflow DAG를 읽을 수 있으면 데이터팀의 워크플로를 이해할 수 있습니다.

**나중에 배워도 되는 영역** (가치 있지만 지금 급하지 않은 영역):
- **데이터 웨어하우스 & 스토리지** — 직접 설계할 일이 아니라면, 개념만 알아도 충분합니다.
- **모니터링 & 옵저버빌리티** — 백엔드 모니터링 경험이 있으므로 개념적으로는 이미 익숙합니다. 도구 이름만 알면 됩니다.

## 전문가는 알지만 당신은 아직 모르는 것들

- **파이프라인은 코드보다 운영이 어렵다**: 처음 만드는 것보다 6개월 뒤에도 안정적으로 돌아가게 유지하는 것이 데이터 엔지니어링의 진짜 난이도입니다. 전문가는 "만들기 쉬운 것"이 아니라 "고치기 쉬운 것"을 설계합니다.
- **스키마 변경이 가장 흔한 장애 원인이다**: 백엔드에서 컬럼 하나 추가하거나 이름을 바꾸면 하류의 파이프라인 전체가 깨질 수 있습니다. 전문가는 이 "상류 변경의 파급효과"를 항상 경계합니다 — 이것이 바로 데이터팀이 백엔드팀과 소통해야 하는 이유입니다.
- **"정확한 데이터"의 정의는 맥락에 따라 다르다**: 같은 "매출" 숫자도 집계 시점, 환불 포함 여부, 환율 기준에 따라 달라집니다. 전문가는 숫자 자체보다 "이 숫자의 정의가 뭔지"를 먼저 확인합니다.
- **배치와 실시간의 선택은 기술이 아니라 비즈니스 결정이다**: "실시간이 항상 좋다"는 초보의 오해입니다. 전문가는 비용, 복잡도, 실제 필요를 따져서 대부분의 경우 배치를 선택합니다.
- **데이터 엔지니어의 70%는 디버깅과 유지보수다**: 새로운 파이프라인을 만드는 것보다, 기존 파이프라인이 왜 이상한 결과를 내는지 추적하는 데 대부분의 시간을 씁니다.
