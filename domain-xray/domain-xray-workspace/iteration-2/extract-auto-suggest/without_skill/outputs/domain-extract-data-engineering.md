# 데이터 엔지니어링 — 암묵지 추출 프롬프트

## 목적
이 프롬프트는 데이터 엔지니어링 분야의 **암묵지(tacit knowledge)** — 숙련된 실무자는 직관적으로 알지만 명시적으로 설명하지 않는 지식 — 를 체계적으로 추출하기 위해 설계되었습니다. Orientation Map에서 식별된 6개 핵심 영역을 기반으로 합니다.

---

## 프롬프트

당신은 10년 이상 경력의 시니어 데이터 엔지니어입니다. 대규모 데이터 파이프라인을 설계, 구축, 운영해 온 실무 전문가로서, 아래 각 영역에 대해 **교과서에는 없지만 실무에서 반드시 알아야 하는 것들**을 빠짐없이 공유해 주세요.

각 영역에 대해 다음 구조로 답변하세요:

---

### 영역 1: 데이터 파이프라인 설계 (ETL/ELT)

**핵심 용어의 실무적 뉘앙스**
- ETL, ELT, DAG, backfill, idempotency, data lineage — 이 용어들이 실제 현장에서는 교과서 정의와 어떻게 다르게 쓰이는가?
- 같은 용어를 팀마다 다른 의미로 사용하는 경우가 있다면?

**숨겨진 전제 조건과 의존성**
- 파이프라인을 설계할 때, 경험 많은 엔지니어가 가장 먼저 확인하지만 초보자는 놓치는 것은 무엇인가?
- "이걸 먼저 확인하지 않으면 나중에 반드시 문제가 된다"는 것들의 체크리스트를 제시하라.

**전형적인 초보자 실수와 그 결과**
- 백엔드 개발자가 파이프라인을 처음 만들 때 가장 흔히 범하는 실수 3가지는?
- 각 실수가 프로덕션에서 어떤 장애로 이어지는지 구체적 시나리오를 제시하라.

**실무 의사결정 휴리스틱**
- ETL vs ELT 선택 — 교과서적 기준 말고, 실제로 결정을 좌우하는 현실적 요인은?
- "이 상황이면 무조건 이걸 선택한다"는 경험칙이 있다면?
- 파이프라인의 실패 복구를 설계할 때, idempotency를 실제로 보장하는 패턴은?

**"파이프라인은 코드가 아니라 시스템이다"의 실무적 의미**
- 이 말이 일상적 작업에서 구체적으로 어떤 차이를 만드는가?
- 코드 리뷰에서 "시스템적 사고"가 부족한 파이프라인을 어떤 징후로 알아보는가?

---

### 영역 2: 데이터 스토리지 아키텍처

**핵심 용어의 실무적 뉘앙스**
- Data warehouse, data lake, lakehouse, partitioning, Parquet, schema-on-read vs schema-on-write — 마케팅 용어와 실제 의미의 간극은?
- "레이크하우스"라고 부르지만 실제로는 그냥 레이크인 경우를 어떻게 구별하는가?

**비용 폭탄을 피하는 실무 지식**
- 스토리지 선택에서 "쿼리 성능 10-100배 차이"가 발생하는 구체적 상황은?
- 클라우드 환경에서 비용이 예상치 못하게 폭발하는 패턴 3가지는?
- 파티셔닝 전략을 잘못 세웠을 때 벌어지는 일의 구체적 사례는?

**실무 의사결정 휴리스틱**
- Warehouse vs Lake vs Lakehouse — "우리 팀 규모가 X이고 데이터가 Y 형태면 이걸 선택하라"는 경험칙은?
- columnar format을 써야 할 때와 쓰면 안 되는 때는?

---

### 영역 3: 워크플로우 오케스트레이션

**핵심 용어의 실무적 뉘앙스**
- DAG, task dependency, scheduler, sensor, SLA — 이 개념들의 실무에서의 미묘한 차이는?
- Airflow에서 "DAG"이라고 부르는 것과 컴퓨터 과학의 DAG 개념이 실무적으로 어떻게 다른가?

**오케스트레이터 선택과 운영의 실무 지식**
- Airflow vs Dagster vs Prefect — 문서에서 말하지 않는 실제 운영 경험 기반의 장단점은?
- 오케스트레이터 자체가 장애 포인트가 되는 상황과 그 대응법은?
- "DAG를 수십 개 관리할 때" 처음에는 몰랐지만 나중에 깨달은 것은?

**전형적인 초보자 실수**
- 스케줄링에서 타임존 관련 실수가 프로덕션 장애로 이어지는 구체적 시나리오는?
- 재시도 로직을 잘못 설계해서 문제가 악화된 사례는?

---

### 영역 4: 데이터 품질 관리

**핵심 용어의 실무적 뉘앙스**
- Data validation, schema drift, freshness, completeness, data contract — 이 용어들이 실제로 의미하는 바와 팀에서 쓰이는 방식의 차이는?

**"스키마 변경은 데이터 엔지니어의 가장 큰 적이다"의 실무적 의미**
- Upstream 스키마 변경이 발생했을 때 실제로 벌어지는 일의 타임라인은?
- 이를 사전에 감지하고 대응하는 실무적 방어 전략은?
- Data contract는 이론적으로 좋지만, 현실에서 작동하게 만드는 데 필요한 것은?

**"정확한 데이터보다 '언제 기준의 데이터인지'가 더 중요하다"의 실무적 의미**
- 이 원칙이 실제 의사결정에서 어떻게 적용되는지 구체적 사례를 3가지 제시하라.
- 데이터의 시점(timeliness)을 추적하기 위해 실무에서 사용하는 패턴은?

**전형적인 초보자 실수**
- 데이터 품질 검증을 "나중에 붙이자"고 미루면 어떤 일이 벌어지는가?
- 지나치게 엄격한 검증과 지나치게 느슨한 검증 — 각각의 실무적 비용은?

---

### 영역 5: 실시간 데이터 처리 (Stream Processing)

**핵심 용어의 실무적 뉘앙스**
- Event streaming, consumer group, offset, windowing, exactly-once processing, back-pressure — 이 용어들의 실무적 의미와 흔한 오해는?
- "Exactly-once processing"이 실제로 보장되는 범위와 한계는?

**배치에서 스트리밍으로 전환할 때의 실무 지식**
- "실시간 처리가 필요하다"고 말하지만 실제로는 배치로 충분한 경우를 어떻게 판별하는가?
- 스트리밍 도입을 후회한 경험이 있다면, 어떤 상황이었는가?
- Kafka를 처음 프로덕션에 도입할 때 아무도 말해주지 않는 운영 복잡성은?

**실무 의사결정 휴리스틱**
- "지연 시간이 X분 이내여야 한다"라는 요구사항이 나왔을 때, 진짜 필요한 건 스트리밍인지 마이크로배치인지 판단하는 기준은?

---

### 영역 6: 데이터 모델링

**핵심 용어의 실무적 뉘앙스**
- Fact table, dimension table, star schema, SCD(Slowly Changing Dimension), grain, denormalization — 이 개념들의 교과서 정의와 현실 적용의 간극은?
- "Grain을 잘못 정의했다"는 것을 나중에 어떻게 알게 되는가?

**"좋은 모델링은 복잡한 질문을 단순한 쿼리로 바꾼다"의 실무적 의미**
- 모델링을 잘한 사례와 못한 사례를 각각 하나씩, 그 차이가 어떤 결과를 낳았는지 구체적으로 설명하라.
- 분석가가 데이터 엔지니어의 모델링에 불만을 갖는 가장 흔한 이유는?

**실무 의사결정 휴리스틱**
- Kimball vs Inmon vs Data Vault — "우리 조직 상황에서는 이걸 선택하라"는 경험칙은?
- denormalization을 해야 할 때와 하면 안 되는 때를 구별하는 실무 기준은?
- dbt를 사용한 모델링에서 처음에는 몰랐지만 나중에 깨달은 것은?

---

### 공통: 디버깅과 운영의 암묵지

**"코드를 짜는 시간보다 디버깅과 모니터링에 더 많은 시간을 쓴다"의 의미**
- 데이터 파이프라인 디버깅이 일반 소프트웨어 디버깅과 근본적으로 다른 점은?
- "데이터가 이상하다"는 보고를 받았을 때, 시니어 엔지니어가 따르는 디버깅 순서는?
- 모니터링에서 "이 지표만은 반드시 보라"는 것은?

**"완벽한 파이프라인보다 관찰 가능한 파이프라인이 살아남는다"의 의미**
- Observability를 파이프라인에 내장하는 구체적 방법은?
- 로깅, 메트릭, 알럿 각각에서 "이건 반드시 하고, 이건 절대 하지 마라"는 것은?

## 답변 가이드라인

1. **구체적 사례 우선**: 추상적 원칙보다 "우리 팀에서 실제로 벌어진 일" 수준의 구체성으로 답하라.
2. **반직관적 지식 강조**: "당연하다고 생각했지만 실제로는 달랐던 것"을 우선적으로 다뤄라.
3. **의사결정 기준 명시**: "A를 선택해야 할 때"와 "B를 선택해야 할 때"를 구체적 조건과 함께 제시하라.
4. **실패 시나리오 포함**: 각 지식이 없었을 때 어떤 장애/손실이 발생하는지 구체적으로 서술하라.
5. **백엔드 개발자 관점 유지**: 이 프롬프트의 대상은 백엔드 개발 경험이 있는 학습자이므로, 가능하면 백엔드 개발과의 비교/대조를 통해 설명하라.
