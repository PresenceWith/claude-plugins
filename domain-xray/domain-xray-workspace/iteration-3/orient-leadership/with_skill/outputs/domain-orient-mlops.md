# MLOps & ML Platform Engineering 오리엔테이션 맵

## 이 맵의 용도

이 맵은 MLOps 팀 리더로서 알아야 할 영역의 전체 지형을 보여주는 뼈대입니다. 상세 지식을 쌓기 전에 이 구조를 먼저 머릿속에 세워두면, 이후 배우는 모든 것이 제자리를 찾습니다. 팀의 기술 제안을 들을 때, 벤더 미팅을 할 때, 아키텍처를 논의할 때 이 맵으로 돌아와서 현재 논의가 전체 그림에서 어디에 있는지 확인하세요.

## 기존 경험과의 연결 (Anchor Points)

당신은 "인프라를 모르는 초보"가 아닙니다. 8년간 모델을 만들면서 MLOps가 해결하려는 문제를 직접 겪어왔습니다.

- **모델 개발 라이프사이클** → MLOps의 모든 자동화는 당신이 수동으로 해왔던 그 과정을 체계화한 것입니다. 데이터 준비 → 실험 → 평가 → 배포의 흐름을 이미 알고 있으므로, 각 자동화 도구가 "왜" 필요한지 직관적으로 이해할 수 있습니다.
- **MLflow 경험** → 실험 추적과 모델 레지스트리를 실무에서 써봤다는 것은 MLOps 도구 생태계의 핵심 조각 하나를 이미 경험한 것입니다. 다른 도구(Weights & Biases, Kubeflow, Vertex AI 등)를 평가할 때 MLflow와 비교하는 프레임이 생깁니다.
- **ML 워크플로우의 고통** → "노트북에서는 되는데 프로덕션에서 안 돼요", "데이터가 바뀌었는데 모델은 그대로예요", "어떤 실험이 어떤 결과를 냈는지 모르겠어요" — 이런 문제를 직접 겪어본 사람은 해결책의 가치를 즉시 판단할 수 있습니다.
- **모델 메트릭 이해** → accuracy, precision, recall, AUC 등 모델 성능 지표를 깊이 아는 것은 모니터링 시스템 설계에서 핵심 자산입니다. 무엇을 측정해야 하는지 모르는 인프라 엔지니어보다 훨씬 유리한 출발점입니다.

## 도메인 전경 (Domain Landscape)

### 영역 1: ML 파이프라인 오케스트레이션

**실무에서 하는 일**: 데이터 수집 → 전처리 → 학습 → 평가 → 배포의 전체 워크플로우를 코드로 정의하고, 자동으로 실행되게 만듭니다. 파이프라인이 실패했을 때 어디서 왜 실패했는지 추적하고, 재실행을 관리합니다.

**왜 중요한가**: 파이프라인 없이는 모든 것이 수동이고, 수동이면 재현이 안 되고, 재현이 안 되면 프로덕션에서 신뢰할 수 없습니다. 팀이 커질수록 이 문제는 기하급수적으로 악화됩니다.

**핵심 용어**:
- **DAG (Directed Acyclic Graph)**: 파이프라인의 단계 간 의존관계를 표현하는 구조. Airflow, Kubeflow Pipelines 등이 이 개념을 씁니다
- **Orchestrator**: DAG를 실행·스케줄링·모니터링하는 시스템 (Airflow, Prefect, Dagster, Kubeflow Pipelines)
- **Idempotency**: 같은 파이프라인을 여러 번 실행해도 같은 결과가 나오는 성질. 재실행 안전성의 핵심

**다른 영역과의 관계**: 피처 스토어, 모델 학습, 배포의 상위 조율자. CI/CD와 결합되어 자동 재학습 트리거를 구성합니다.

### 영역 2: 모델 서빙 & 배포 전략

**실무에서 하는 일**: 학습된 모델을 실제 트래픽이 들어오는 환경에 올려서 예측을 제공합니다. 실시간(REST API) 서빙, 배치 추론, 스트리밍 추론 중 어떤 패턴이 맞는지 결정하고, 배포 방식(canary, blue-green, shadow)을 선택합니다.

**왜 중요한가**: 아무리 좋은 모델도 프로덕션에 올라가지 못하면 비즈니스 가치가 0입니다. 그리고 "올리는 것" 자체보다 "안전하게 올리고, 문제가 생기면 빠르게 롤백하는 것"이 진짜 과제입니다.

**핵심 용어**:
- **Model Serving**: 모델을 API로 감싸서 추론 요청을 받는 인프라 (TFServing, Triton, Seldon, BentoML)
- **Canary Deployment**: 새 모델을 트래픽의 일부(예: 5%)에만 먼저 노출해서 문제가 없는지 확인 후 점진적 확대
- **Shadow Mode**: 새 모델이 실제 트래픽을 받지만 결과를 반환하지 않고 로그만 남김. 기존 모델과 비교 분석용
- **Latency SLA**: 추론 응답 시간 보장 수준. p50, p95, p99로 측정

**다른 영역과의 관계**: 모니터링과 직접 연결(배포 후 성능 추적), CI/CD의 최종 목적지, 인프라 아키텍처의 핵심 결정 사항.

### 영역 3: ML 모니터링 & 옵저버빌리티

**실무에서 하는 일**: 프로덕션에서 돌아가는 모델이 여전히 잘 작동하는지 감시합니다. 단순 시스템 메트릭(CPU, 메모리, 레이턴시)뿐 아니라 ML 고유의 문제 — 데이터 드리프트, 모델 성능 저하, 피처 분포 변화 — 를 탐지합니다.

**왜 중요한가**: 모델은 소프트웨어와 다르게 "조용히 망가집니다". 에러가 나지 않아도 예측 품질이 서서히 저하될 수 있고, 이걸 탐지 못 하면 비즈니스 손실이 누적됩니다. 당신의 모델 메트릭 이해가 여기서 결정적 강점이 됩니다.

**핵심 용어**:
- **Data Drift**: 프로덕션 입력 데이터의 분포가 학습 데이터와 달라지는 현상
- **Concept Drift**: 입력과 출력 사이의 관계 자체가 변하는 현상 (예: 코로나 이후 소비 패턴 변화)
- **Feature Attribution**: 각 피처가 개별 예측에 얼마나 기여했는지 추적 (SHAP, LIME)
- **Ground Truth Lag**: 실제 정답을 얻기까지 걸리는 시간. 이 지연 동안 모니터링은 proxy 지표에 의존

**다른 영역과의 관계**: 모델 서빙의 피드백 루프, 파이프라인 재학습 트리거의 입력, 거버넌스의 증거 기반.

### 영역 4: 피처 스토어 & 데이터 관리

**실무에서 하는 일**: ML에 사용되는 피처를 중앙에서 관리하고, 학습과 서빙에서 동일한 피처 로직을 사용하도록 보장합니다. 피처의 버전 관리, 접근 제어, 재사용을 체계화합니다.

**왜 중요한가**: Training-serving skew(학습 시와 서빙 시 피처 계산이 달라지는 문제)는 가장 잡기 어려운 ML 버그 중 하나입니다. 피처 스토어는 이 문제를 구조적으로 해결합니다. 또한 팀 간 피처 재사용은 개발 속도에 직접 영향을 줍니다.

**핵심 용어**:
- **Training-Serving Skew**: 학습과 서빙에서 피처 계산 로직이 달라 모델 성능이 저하되는 문제
- **Online Store / Offline Store**: 실시간 서빙용 저지연 저장소 vs 학습용 대용량 저장소
- **Feature Freshness**: 피처가 얼마나 최신 데이터를 반영하는지. 비즈니스 요구에 따라 달라짐
- **Point-in-Time Correctness**: 학습 시 특정 시점의 피처 값만 사용해 데이터 누수를 방지

**다른 영역과의 관계**: 파이프라인의 핵심 입력원, 모델 서빙과 직결(온라인 스토어), 데이터 거버넌스의 관리 대상.

### 영역 5: CI/CD for ML

**실무에서 하는 일**: 모델 코드, 학습 파이프라인, 인프라 설정의 변경을 자동으로 테스트하고 배포합니다. 일반 소프트웨어 CI/CD와 유사하지만, 데이터 검증, 모델 성능 게이트, A/B 테스트 연동 등 ML 고유의 단계가 추가됩니다.

**왜 중요한가**: 수동 배포는 느리고 실수가 많고 감사 추적이 안 됩니다. 팀 규모가 커지고 모델 수가 늘면 자동화 없이는 운영이 불가능해집니다.

**핵심 용어**:
- **Model Validation Gate**: 새 모델이 기존 모델보다 성능이 좋은지 자동 확인하는 단계. 통과해야 배포 진행
- **Infrastructure as Code (IaC)**: Terraform, Pulumi 등으로 인프라를 코드로 정의·관리
- **GitOps**: Git 저장소를 인프라·배포 상태의 단일 진실 소스로 사용하는 방식
- **CT (Continuous Training)**: 새 데이터가 들어오면 자동으로 모델을 재학습하는 파이프라인

**다른 영역과의 관계**: 파이프라인 오케스트레이션을 트리거하고, 모델 서빙으로 결과를 전달하며, 모니터링의 결과가 CT를 트리거.

### 영역 6: 인프라 & 플랫폼 아키텍처

**실무에서 하는 일**: GPU 클러스터, 쿠버네티스, 클라우드 서비스(AWS SageMaker, GCP Vertex AI, Azure ML)를 조합해서 팀이 사용할 ML 플랫폼을 설계합니다. 컴퓨팅 자원 관리, 비용 최적화, 멀티테넌시, 보안을 다룹니다.

**왜 중요한가**: 잘못된 인프라 선택은 되돌리기 매우 어렵고 비용이 큽니다. "이걸 직접 만들 것인가, 매니지드 서비스를 쓸 것인가"는 팀 리더가 내려야 할 가장 중요한 전략적 결정 중 하나입니다.

**핵심 용어**:
- **Managed ML Platform**: SageMaker, Vertex AI 등 클라우드 벤더가 제공하는 통합 ML 플랫폼
- **Kubernetes (K8s)**: 컨테이너 오케스트레이션 시스템. MLOps에서 학습/서빙 워크로드 실행의 기반
- **GPU Scheduling**: 고가의 GPU 자원을 팀원/작업 간 효율적으로 배분하는 문제
- **Build vs Buy**: 자체 구축 vs 매니지드 서비스. 팀 규모, 커스터마이징 필요, 벤더 의존도의 트레이드오프

**다른 영역과의 관계**: 모든 다른 영역의 실행 기반. 비용과 직접 연결되며, 팀 구성과 채용 전략에도 영향.

### 영역 7: ML 거버넌스 & 규정 준수

**실무에서 하는 일**: 모델의 전체 생애주기를 추적·기록하고, 누가 어떤 데이터로 어떤 모델을 만들어 어디에 배포했는지 감사할 수 있게 합니다. 모델의 공정성, 편향, 해석 가능성을 관리합니다.

**왜 중요한가**: 규제(EU AI Act, 금융 규제 등)가 강화되고 있고, 규정 미준수 시 법적·재무적 리스크가 큽니다. 경영진에게 "우리 모델이 어떻게 작동하고 왜 이런 결정을 내렸는지" 설명할 수 있어야 합니다.

**핵심 용어**:
- **Model Lineage**: 모델이 어떤 데이터, 코드, 파라미터로 만들어졌는지 추적하는 계보
- **Model Card**: 모델의 용도, 한계, 성능, 편향 분석 등을 문서화한 표준 형식
- **Reproducibility**: 동일 조건에서 동일 결과를 재현할 수 있는 능력
- **Audit Trail**: 모델 변경 이력, 승인 프로세스, 배포 기록의 감사 가능한 로그

**다른 영역과의 관계**: 모든 영역에 횡단적으로 적용. 파이프라인에서 리니지를, 서빙에서 감사 로그를, 모니터링에서 공정성 지표를 수집.

## 학습 우선순위 맵

당신의 목적(팀의 기술 방향 설정, 아키텍처 의사결정, 기술 제안 평가)에 기반한 우선순위:

**여기서 시작** (당신의 목표에 가장 높은 레버리지):
- **영역 6: 인프라 & 플랫폼 아키텍처** — 팀 리더로서 가장 먼저 내려야 할 결정이 "우리 플랫폼을 어떻게 구성할 것인가"이고, 이 결정이 다른 모든 선택을 제약합니다. Build vs buy 판단 프레임워크가 급선무입니다
- **영역 5: CI/CD for ML** — 팀의 개발 속도와 운영 안정성을 동시에 결정하는 영역. 팀원의 기술 제안이 가장 빈번하게 나올 곳이기도 합니다

**그 다음** (중요하지만 즉시는 아닌):
- **영역 2: 모델 서빙 & 배포 전략** — 프로덕션 가치 전달의 핵심이지만, 플랫폼 결정 이후에 구체화됨
- **영역 3: ML 모니터링** — 당신의 모델 메트릭 지식이 큰 강점이 되는 영역. 다만 배포가 먼저 있어야 모니터링이 의미 있음
- **영역 1: 파이프라인 오케스트레이션** — 팀의 일상 생산성에 직접 영향. 도구 선택이 필요

**나중에 깊이 들어가도 됨** (가치 있지만 기다릴 수 있음):
- **영역 4: 피처 스토어** — 팀과 모델 수가 충분히 커져야 ROI가 나오는 영역
- **영역 7: ML 거버넌스** — 규제 산업이 아니라면 초기에는 기본적인 리니지 추적만으로 충분

## 현재 상태와 목표 사이의 간극

- **인프라 아키텍처 판단력 부재**: 모델 개발은 잘 하지만, "이 모델을 어떤 인프라 위에서 어떻게 서빙할 것인가"에 대한 선택지와 트레이드오프를 모릅니다. Kubernetes, 클라우드 매니지드 서비스, 서버리스 등 각각의 적합 조건을 파악해야 합니다.
- **Build vs Buy 의사결정 프레임워크 없음**: 팀이 "Kubeflow vs Vertex AI vs 자체 구축"을 논의할 때, 판단 기준(팀 규모, 커스터마이징 필요도, 벤더 락인 리스크, TCO)을 아직 갖추지 못했습니다.
- **프로덕션 운영 감각 부족**: 모델을 "만드는" 것과 "운영하는" 것의 차이 — SLA, 장애 대응, 롤백, 온콜 — 에 대한 실무 경험이 없습니다. 팀의 운영 프로세스를 설계하려면 이 감각이 필요합니다.
- **기술 전략 커뮤니케이션**: 경영진에게 "왜 이 도구에 투자해야 하는지", "자체 구축의 기술 부채 리스크"를 비즈니스 언어로 설명하는 능력이 아직 없습니다. 트레이드오프를 기술적으로 이해하는 것과 경영진 언어로 번역하는 것은 다른 스킬입니다.
- **팀의 기술 제안 평가 기준 부재**: 팀원이 "Seldon Core 도입하자"고 제안했을 때, 대안(BentoML, KServe, 매니지드 서비스)과 비교해서 왜 그것이 좋은지/나쁜지 평가할 기준이 아직 없습니다.
