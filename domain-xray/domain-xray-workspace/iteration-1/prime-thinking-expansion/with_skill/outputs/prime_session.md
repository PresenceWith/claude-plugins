## Q1 — Prediction

데이터 파이프라인 설계에서 전문가가 초보 데이터 엔지니어의 가장 흔한 실수로 꼽을 것은 무엇일까요?

**A.** ETL 변환 로직에서 엣지 케이스를 처리하지 않아 데이터가 누락되는 것

**B.** 파이프라인의 idempotency를 고려하지 않아 재실행 시 데이터가 중복되는 것

**C.** DAG 의존성을 잘못 설정하여 순서가 꼬이는 것

---

### [사용자 선택: B]

### 3-Layer Thinking Expansion

**Layer 1 — Surface the reasoning**

B를 선택하셨군요 — 그렇다면 파이프라인의 가장 위험한 실패는 "데이터가 잘못 들어가는 것"이고, 특히 재실행 상황에서의 중복이 가장 흔한 문제라고 가정하고 계신 셈입니다. 이것은 백엔드 API 설계에서 POST 요청의 멱등성을 고민했던 경험에서 오는 자연스러운 직관입니다 — API에서도 같은 요청이 두 번 들어오면 중복 생성이 문제가 되니까요.

**Layer 2 — Shift the perspective**

한편, 이 영역의 전문가들은 이 상황을 "코드의 문제"가 아니라 "시스템 설계의 문제"로 보는 경향이 있습니다. 예를 들어, idempotency 자체는 이미 잘 알려진 패턴이라 대부분의 프레임워크가 지원하지만, 초보가 놓치는 것은 그보다 더 근본적인 수준에 있을 수 있습니다. 전문가들이 말하는 "파이프라인은 코드가 아니라 시스템이다"라는 관점에서 보면, 가장 흔한 실수의 성격 자체가 달라질 수 있습니다.

**Layer 3 — Open a deeper question**

그렇다면, "파이프라인이 실패하는 것은 정상"이라는 전문가의 전제에서 출발할 때, 초보가 간과하는 것은 중복 방지 같은 개별 기법이 아니라 복구 설계(recovery design) 자체의 부재일 수 있을까요?

## Q2 — Connection

당신의 백엔드 API 설계 경험(요청/응답 흐름)이 데이터 파이프라인 설계의 입력/변환/출력 흐름과 가장 닮은 점은 무엇일까요?

**A.** 입력 검증 — API에서 request validation을 하듯, 파이프라인에서도 원시 데이터의 스키마와 값을 검증하는 것

**B.** 에러 핸들링 — API에서 try-catch와 에러 응답을 설계하듯, 파이프라인에서도 실패 시 재시도와 알림을 설계하는 것

**C.** 멱등성 — API에서 같은 요청이 같은 결과를 보장하듯, 파이프라인에서도 같은 입력이 같은 출력을 보장하는 것

---

### [사용자 선택: A]

### 3-Layer Thinking Expansion

**Layer 1 — Surface the reasoning**

A를 선택하셨군요 — 그렇다면 API의 request validation과 파이프라인의 data validation이 가장 직접적인 대응 관계라고 가정하고 계신 셈입니다. "잘못된 입력을 거르는 것이 시스템 안정성의 첫걸음"이라는 프레임으로 두 세계를 연결하고 계신 것입니다. 이것은 매우 논리적인 매핑입니다 — 둘 다 경계(boundary)에서 나쁜 데이터를 차단한다는 공통점이 있으니까요.

**Layer 2 — Shift the perspective**

한편, 이 영역의 전문가들은 API 검증과 파이프라인 검증의 결정적 차이에 주목합니다. API에서는 잘못된 요청을 400 에러로 거부하면 끝이지만, 파이프라인에서는 "잘못된 데이터를 거부하면 그 데이터는 어디로 가는가?"라는 질문이 생깁니다. 예를 들어, upstream 시스템이 스키마를 갑자기 바꾸었을 때(schema drift), 모든 데이터를 거부하는 것이 맞을까요, 아니면 일단 받아서 따로 저장해야 할까요? API의 "거부하면 끝" 모델과는 근본적으로 다른 판단이 필요합니다.

**Layer 3 — Open a deeper question**

그렇다면, 데이터 품질 관리에서 말하는 "data contract"이라는 개념은 API 스펙(OpenAPI/Swagger)과 어떻게 다르고, 왜 데이터 엔지니어링에서는 별도의 용어와 도구가 필요하게 되었을까요?

## Q3 — Relationship

데이터 파이프라인 설계(Area 1)에서의 실수가 데이터 모델링(Area 6)에 미치는 가장 큰 영향은 무엇일까요?

**A.** 파이프라인이 느리면 fact table 업데이트가 지연되어 분석가가 오래된 데이터를 보게 된다

**B.** 파이프라인에서 중복 데이터가 들어오면 dimension table의 grain이 깨져서 집계가 틀어진다

**C.** 파이프라인의 변환 로직이 비즈니스 로직과 불일치하면 star schema의 설계 자체가 무의미해진다

---

### [사용자 선택: C]

### 3-Layer Thinking Expansion

**Layer 1 — Surface the reasoning**

C를 선택하셨군요 — 그렇다면 파이프라인과 모델링의 관계에서 가장 치명적인 문제는 "기술적 오류"가 아니라 "의미적 불일치"라고 가정하고 계신 셈입니다. 변환 로직이 비즈니스의 실제 의미와 어긋나면 아무리 정교한 star schema도 쓸모없다는 판단인데, 이것은 백엔드에서 "API 스펙은 완벽한데 비즈니스 로직이 잘못 구현되어 있으면 의미가 없다"라는 경험에서 오는 통찰입니다.

**Layer 2 — Shift the perspective**

한편, 이 영역의 전문가들은 파이프라인과 모델링의 관계를 "한 방향의 영향"이 아니라 "양방향 피드백 루프"로 보는 경향이 있습니다. 예를 들어, 파이프라인의 변환 로직이 모델링에 영향을 주는 것만이 아니라, 데이터 모델의 grain 정의가 파이프라인이 어떤 수준으로 데이터를 변환해야 하는지를 결정합니다. 실무에서 "파이프라인 먼저 만들고 나중에 모델링"하는 팀과 "모델링 먼저 설계하고 파이프라인이 따르게 하는" 팀의 결과가 크게 다르다고 합니다.

**Layer 3 — Open a deeper question**

그렇다면, 데이터 엔지니어링에서 "grain을 먼저 정의한다"는 것이 구체적으로 파이프라인의 어떤 설계 결정을 선제적으로 결정짓게 되고, 이것이 왜 백엔드의 "DB 스키마 먼저 vs API 먼저" 논쟁과는 본질적으로 다른 문제일까요?
