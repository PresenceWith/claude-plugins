# Prime Thinking Session: 데이터 엔지니어링

> Orient Map 기반 학습 준비 질문 세션
> 본격적인 학습에 들어가기 전, 기존 지식을 활성화하고 새로운 개념과의 연결고리를 미리 만들어두는 사전 사고 세션입니다.

---

## Q1: API 요청 흐름과 데이터 파이프라인은 어떻게 다를까?

**질문**: 백엔드에서 API 요청이 들어오면 요청 -> 처리 -> 응답의 흐름을 따릅니다. 데이터 파이프라인의 입력 -> 변환 -> 출력도 비슷해 보이는데, 실제로 어떤 점이 근본적으로 다를까요? 실패했을 때의 상황을 생각해보세요.

**가정된 답변**: "API는 실시간으로 한 건씩 처리하고, 파이프라인은 대량의 데이터를 배치로 처리하는 차이가 있지 않을까요? 실패하면 API는 에러 응답을 주면 되지만, 파이프라인은... 재시도?"

**확장 피드백**: 좋은 출발점입니다. 핵심적인 차이는 **실패의 의미**에 있습니다. API 요청이 실패하면 클라이언트가 다시 시도하면 됩니다. 하지만 파이프라인이 실패하면 그 뒤에 의존하는 모든 후속 작업이 멈춥니다. 그래서 데이터 엔지니어링에서는 **idempotency**(멱등성)라는 개념이 매우 중요합니다 -- 같은 작업을 여러 번 실행해도 결과가 동일해야 합니다. API에서도 멱등성을 고려하지만, 파이프라인에서는 이것이 설계의 핵심 원칙입니다. 또한 파이프라인은 "시스템"이지 단순한 "코드"가 아닙니다. 실패는 예외가 아니라 정상 상태의 일부이고, 복구 설계가 핵심입니다.

---

## Q2: 데이터를 저장할 때 "어디에"보다 "어떤 형식으로"가 더 중요할까?

**질문**: 백엔드에서는 보통 PostgreSQL 같은 RDB에 데이터를 넣고, 스키마를 먼저 정의한 뒤 데이터를 삽입합니다. 데이터 엔지니어링에서는 data warehouse, data lake, lakehouse라는 선택지가 있다고 합니다. 왜 이렇게 여러 종류가 필요할까요? 하나로 통일하면 안 될까요?

**가정된 답변**: "데이터 양이 너무 많아서 RDB로 감당이 안 되니까 다른 저장소가 필요한 것 아닐까요? Data lake는 뭐든 다 넣을 수 있는 큰 저장소 같고, warehouse는 정리된 저장소 같은 느낌입니다."

**확장 피드백**: 방향은 맞습니다. 하지만 단순히 "크기" 문제만은 아닙니다. 핵심은 **schema-on-write vs schema-on-read**라는 철학적 차이입니다. Warehouse는 데이터를 넣을 때 구조를 강제합니다(백엔드 RDB와 비슷). Lake는 일단 원본 그대로 저장하고, 읽을 때 구조를 부여합니다. 왜 이런 차이가 생겼을까요? 데이터의 용도를 처음부터 알 수 없는 경우가 많기 때문입니다. 로그 데이터를 수집할 때, 3개월 후에 어떤 분석에 쓸지 모릅니다. 그래서 일단 다 저장해두고 나중에 필요한 형태로 변환하는 전략이 나온 것입니다. Lakehouse는 이 두 장점을 결합하려는 시도입니다. 또한 Parquet 같은 columnar format은 "전체 행"이 아니라 "특정 열"만 빠르게 읽을 수 있게 해줍니다 -- 분석 쿼리에서 성능 차이가 10-100배까지 납니다.

## Q3: CI/CD와 워크플로우 오케스트레이션은 정말 같은 개념일까?

**질문**: Orient map에서 "Airflow는 데이터 버전의 CI/CD"라고 했습니다. 이미 CI/CD 경험이 있다면 Airflow를 금방 이해할 수 있을까요? 아니면 근본적으로 다른 점이 있을까요?

**가정된 답변**: "CI/CD는 코드 변경 -> 빌드 -> 테스트 -> 배포의 흐름인데, Airflow도 비슷하게 작업A -> 작업B -> 작업C 순서로 실행하는 거 아닌가요? DAG이라는 게 파이프라인 그래프 같은 건데, 의존 관계 설정하는 건 CI/CD랑 비슷할 것 같습니다."

**확장 피드백**: CI/CD와의 유사성은 "의존 관계를 가진 작업들의 순차 실행"이라는 점에서 맞습니다. 하지만 결정적인 차이가 있습니다. CI/CD는 **이벤트 트리거**(코드 푸시)로 실행되지만, 데이터 오케스트레이션은 **시간 기반 스케줄링**이 주입니다. 매일 새벽 2시에 어제의 데이터를 처리하는 식입니다. 여기서 **backfill**이라는 고유한 문제가 생깁니다 -- "지난 3일간 파이프라인이 실패했으면, 3일치 데이터를 다시 처리해야 한다." CI/CD에는 없는 개념입니다. 또한 **sensor**라는 개념이 있어서, "S3에 파일이 도착할 때까지 기다렸다가 실행"처럼 외부 상태를 감시하는 기능이 필요합니다. CI/CD보다 훨씬 복잡한 실행 조건과 실패 복구 시나리오를 다룹니다.

## Q4: "이 데이터 믿을 수 있어?"라는 질문에 어떻게 답할 수 있을까?

**질문**: 데이터팀과 협업할 때 가장 흔한 질문이 "이 데이터 믿을 수 있어?"라고 합니다. 백엔드에서는 유닛 테스트와 통합 테스트로 코드의 정확성을 검증하는데, 데이터의 정확성은 어떻게 검증할까요?

**가정된 답변**: "데이터도 테스트 코드를 짜서 검증하면 되지 않나요? 예를 들어 null이 없는지, 값의 범위가 올바른지 체크하는 식으로."

**확장 피드백**: 기본적인 validation은 그렇습니다. 하지만 데이터 품질은 코드 테스트보다 훨씬 미묘합니다. 코드는 같은 입력에 같은 출력을 내지만, 데이터는 매일 다릅니다. "어제 주문이 1만 건이었는데 오늘 100건"이면 버그일까요, 아니면 공휴일이라 그런 걸까요? 이것이 **anomaly detection**의 영역입니다. 또한 **schema drift**라는 문제가 있습니다 -- upstream 시스템이 컬럼명을 바꾸거나 데이터 타입을 변경하면 파이프라인 전체가 깨집니다. Orient map에서 "스키마 변경은 데이터 엔지니어의 가장 큰 적"이라고 한 이유입니다. 최근에는 **data contract**라는 개념으로, 데이터 제공자와 소비자 간에 스키마와 품질 기준을 명시적으로 합의하는 방식이 등장했습니다. API 스펙을 문서화하듯이요.

## Q5: 배치 처리와 실시간 처리, 정말 둘 다 필요할까?

**질문**: 실시간 처리가 가능하다면 배치 처리는 필요 없는 것 아닌가요? 모든 것을 실시간으로 처리하면 더 좋은 것 아닐까요?

**가정된 답변**: "실시간이 항상 더 좋은 건 아닌가요? 배치는 옛날 방식 같고... 하지만 비용이 더 많이 드는 걸까요?"

**확장 피드백**: 직관적으로는 실시간이 항상 우월해 보이지만, 현실은 다릅니다. 세 가지 이유가 있습니다. 첫째, **비용** -- 실시간 인프라(Kafka 클러스터 등)는 배치보다 운영 비용과 복잡성이 몇 배 높습니다. 둘째, **정확성** -- 실시간 처리에서 "exactly-once processing"(정확히 한 번 처리)은 분산 시스템에서 가장 어려운 문제 중 하나입니다. 배치는 이 문제가 훨씬 단순합니다. 셋째, **필요성** -- "어제의 매출 리포트"는 실시간일 필요가 없습니다. 대부분의 분석 워크로드는 하루나 한 시간 단위의 배치로 충분합니다. 실시간이 필수인 경우는 이상 탐지, 추천 시스템, 실시간 대시보드처럼 **지연 시간 자체가 비즈니스 가치**인 경우뿐입니다. 실무에서는 배치와 스트리밍을 함께 사용하는 **Lambda/Kappa 아키텍처**가 일반적입니다.

## Q6: "언제 기준의 데이터인지"가 왜 "정확한 데이터"보다 더 중요할까?

**질문**: Orient map의 전문가 인사이트에서 "정확한 데이터보다 언제 기준의 데이터인지가 더 중요한 질문"이라고 했습니다. 이게 무슨 의미일까요? 정확하지 않은 데이터가 어떻게 유용할 수 있죠?

**가정된 답변**: "음... 시점에 따라 데이터 값이 달라질 수 있어서? 예를 들어 오늘 조회한 매출과 내일 조회한 어제 매출이 다를 수도 있는 건가요?"

**확장 피드백**: 정확히 그 지점입니다. 데이터는 시간에 따라 변합니다. 고객이 주문을 취소하면, "3시에 본 매출"과 "5시에 본 매출"이 다릅니다. 둘 다 해당 시점에서는 "정확한" 데이터입니다. 문제는 의사결정자가 "매출이 얼마야?"라고 물었을 때, 답이 "언제 기준이냐에 따라 다릅니다"라는 것입니다. 이것이 **data freshness**와 **point-in-time correctness**의 영역입니다. 데이터 모델링에서 **Slowly Changing Dimension(SCD)**이라는 개념이 있는 이유도 이것입니다 -- 고객의 주소가 바뀌었을 때, "현재 주소"만 저장할지, "변경 이력"까지 저장할지 결정해야 합니다. 이 결정이 "3개월 전 주문은 당시 주소 기준으로 배송되었는가?"라는 질문에 답할 수 있는지를 결정합니다.

## Q7: 내가 만드는 백엔드 API가 데이터 파이프라인에 어떤 영향을 줄까?

**질문**: 데이터팀과 협업한다고 했는데, 백엔드 개발자로서 내가 하는 작업이 데이터 파이프라인에 직접적인 영향을 주는 경우가 있을까요?

**가정된 답변**: "API에서 DB에 쓰는 데이터를 데이터팀이 가져다 쓰는 건가요? 그러면 내가 테이블 스키마를 바꾸면 문제가 될 수 있겠네요."

**확장 피드백**: 바로 그것이 데이터 엔지니어링 현업에서 가장 큰 마찰 지점입니다. 백엔드 개발자가 별 생각 없이 컬럼명을 바꾸거나, enum 값을 추가하거나, JSON 필드의 구조를 변경하면 -- 그 테이블을 소스로 사용하는 파이프라인 전체가 깨집니다. 이것이 **data lineage**(데이터 계보)가 중요한 이유입니다. "이 테이블을 누가 읽고 있는가?"를 추적해야 합니다. 최근의 **data contract** 패턴은 이 문제를 해결하려는 시도입니다 -- API 스펙처럼, 데이터 스키마도 제공자(백엔드)와 소비자(데이터팀) 간에 명시적으로 합의하고, 변경 시 사전 통보하는 프로세스를 만드는 것입니다. 이 맥락을 이해하면 데이터팀과의 협업에서 즉시 신뢰를 얻을 수 있습니다.
