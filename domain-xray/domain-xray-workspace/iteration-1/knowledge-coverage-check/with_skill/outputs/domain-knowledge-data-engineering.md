# Data Engineering: 실무 전문가의 암묵지 해부

이 문서는 Data Engineering(데이터 엔지니어링) 분야에서 20년 이상의 경력을 가진 실무자가 "당연하다"고 여기지만 결코 초보자에게 설명하지 않는 암묵지를 체계적으로 정리한 것이다. 백엔드 개발자로서 Python과 SQL 경험이 있는 독자를 위해, 기존 지식과의 접점을 의식하면서 데이터 엔지니어링의 실제 작동 원리를 해부한다.

---

# Stage 1: Data Pipeline Design (ETL/ELT 패턴, 실패 복구, 멱등성)

데이터 파이프라인은 데이터 엔지니어링의 핵심 산출물이다. 백엔드 개발에서 API endpoint가 요청을 받아 처리하고 응답하는 것처럼, 파이프라인은 데이터를 받아 변환하고 적재한다. 그러나 이 유사성이 오히려 심각한 오해를 낳는다. API는 단건 요청을 실시간으로 처리하지만, 파이프라인은 수십억 건의 레코드를 배치로 처리하며, 실패 시 "재시도"의 의미가 완전히 다르다.

## Practical Terminology and Jargon (실무 용어와 뉘앙스)

**ETL vs ELT**: 교과서에서는 Extract-Transform-Load와 Extract-Load-Transform의 순서 차이로 설명하지만, 실무에서 이 구분은 "변환 책임이 어디에 있는가"를 둘러싼 조직적, 아키텍처적 결정이다. ETL은 데이터가 웨어하우스에 도착하기 전에 정제되어야 하는 패턴으로, 전통적인 on-premise 환경에서 컴퓨팅 리소스가 제한적일 때 선호되었다. ELT는 "일단 원본 그대로 넣고 나중에 변환하자"는 접근으로, 클라우드 웨어하우스(BigQuery, Snowflake)의 거의 무한한 컴퓨팅 파워가 이를 가능하게 만들었다. 팀 미팅에서 누군가 "우리는 ELT로 가자"고 말하면, 그것은 단순히 순서를 바꾸자는 의미가 아니라 "raw data(원시 데이터)를 보존하고, 변환 로직을 SQL 기반으로 웨어하우스 안에서 관리하자"는 아키텍처 철학의 선언이다.

**Idempotency(멱등성)**: 백엔드 개발자에게 멱등성은 "같은 API 요청을 여러 번 보내도 결과가 같다"는 의미다. 데이터 파이프라인에서의 멱등성은 훨씬 더 구체적이고 중요한 개념이다. "같은 파이프라인을 같은 날짜 파라미터로 세 번 실행해도 결과 테이블에 데이터가 세 배로 늘어나지 않는다"는 것을 의미한다. 이것을 보장하기 위해 실무에서는 INSERT 대신 MERGE(upsert) 패턴을 쓰거나, 파티션 단위로 DELETE-INSERT(해당 날짜 파티션을 통째로 지우고 다시 넣기)를 수행한다. 팀에서 "이 파이프라인 멱등한가?"라고 물으면, "재실행해도 안전한가?"를 묻는 것이다.

**Backfill(백필)**: 과거 데이터를 소급 처리하는 작업이다. 새로운 파이프라인을 만들었을 때 "지난 6개월치를 채워야 한다"는 상황, 또는 버그를 발견해서 "3월 데이터를 다시 돌려야 한다"는 상황에서 발생한다. 교과서에는 거의 나오지 않지만 실무에서는 파이프라인 개발 시간의 30% 이상이 backfill 가능한 설계에 투입된다. 누군가 "backfill 돌려줘"라고 말하면, 특정 날짜 범위에 대해 파이프라인을 과거 시점으로 소급 실행하라는 뜻이다.

**DAG (Directed Acyclic Graph)**: Airflow 등 오케스트레이션 도구에서 파이프라인의 작업 의존성을 표현하는 방향성 비순환 그래프다. 실무에서 "DAG"는 거의 "파이프라인" 자체를 가리키는 동의어로 쓰인다. "이 DAG 언제 돌아?" = "이 파이프라인은 몇 시에 실행돼?"

**SLA (Service Level Agreement)**: 데이터 파이프라인 맥락에서 SLA는 "이 데이터가 몇 시까지 준비되어야 하는가"를 의미한다. 경영진의 아침 대시보드가 오전 8시에 업데이트되어야 한다면, 파이프라인의 SLA는 오전 7시 30분(버퍼 포함)이 된다. SLA를 지키지 못하면 downstream(하류)의 모든 의사결정이 지연된다.

## Hidden Preconditions (숨겨진 전제 조건)

전문가가 파이프라인을 설계할 때 자연스럽게 가정하지만 초보자에게는 보이지 않는 전제 조건들이 있다.

첫째, **모든 파이프라인은 반드시 실패한다**는 전제다. 백엔드에서 API가 다운되면 즉시 알림이 오고 수분 내에 복구하지만, 배치 파이프라인은 새벽 3시에 조용히 실패하고 아침 8시에 대시보드가 빈 것을 보고 알게 되는 경우가 부지기수다. 따라서 전문가는 파이프라인 설계 시 "정상 경로"보다 "실패 경로"를 먼저 생각한다. "이 단계가 실패하면 어디서부터 재실행할 수 있는가?", "부분 실패 시 이미 적재된 데이터는 어떻게 되는가?"가 설계의 핵심 질문이다.

둘째, **데이터 소스는 당신이 통제하지 못한다**는 전제다. 백엔드 개발에서는 자신이 만든 DB 스키마를 자신이 관리하지만, 데이터 엔지니어링에서 소스 데이터는 대부분 외부 팀이나 외부 시스템에서 온다. 마케팅 팀이 갑자기 CRM의 컬럼 이름을 바꾸거나, 외부 API가 응답 포맷을 변경하는 일이 예고 없이 발생한다. 이것이 schema drift(스키마 드리프트)이며, 모든 파이프라인은 이에 대한 방어 로직을 갖추고 있어야 한다.

셋째, **시간(time)은 데이터 파이프라인에서 가장 복잡한 차원**이라는 전제다. "오늘 데이터"라고 하면 간단해 보이지만, UTC 기준인가 KST 기준인가, 이벤트 발생 시각 기준인가 데이터 도착 시각 기준인가에 따라 결과가 완전히 달라진다. 전문가는 항상 "event time(이벤트 시간) vs processing time(처리 시간)"을 구분하며, 이 구분을 놓치면 데이터 중복이나 누락이 발생한다. Late-arriving data(지연 도착 데이터)는 어제의 이벤트가 오늘에야 시스템에 도착하는 현상으로, 이를 처리하지 않으면 어제의 집계가 영원히 부정확해진다.

## Beginner's Traps (초보자의 함정)

**함정 1: "파이프라인을 한 번 만들면 끝이다"라는 착각**

시나리오: 주니어 엔지니어가 외부 API에서 데이터를 가져오는 파이프라인을 만들었다. 잘 동작한다. 2주 후 API 제공자가 pagination(페이지네이션) 로직을 변경했다. 파이프라인은 첫 페이지만 가져오고 성공으로 표시된다. 아무도 모르는 사이에 2주간 데이터의 90%가 누락된다. 이것이 "silent failure(조용한 실패)"이며, 데이터 파이프라인에서 가장 무서운 실패 유형이다. 에러가 나서 멈추는 것은 차라리 낫다. 에러 없이 잘못된 데이터를 계속 적재하는 것이 진짜 재앙이다.

교훈: 파이프라인에는 반드시 data quality check(데이터 품질 검증)이 포함되어야 한다. 최소한 "예상 레코드 수 대비 실제 레코드 수" 확인, "null 비율이 갑자기 급증했는지" 확인이 필요하다. 전문가는 파이프라인 코드보다 검증 코드에 더 많은 시간을 투자하기도 한다.

**함정 2: 멱등하지 않은 파이프라인으로 재실행하기**

시나리오: 파이프라인이 실패해서 재실행했더니 데이터가 두 배로 늘어났다. INSERT만 사용하고 중복 방지 로직이 없었기 때문이다. 이를 발견한 것은 3일 후 매출 리포트 숫자가 갑자기 두 배가 된 것을 본 재무팀의 문의 메일이었다. 파이프라인을 고치는 것은 30분이지만, 이미 적재된 중복 데이터를 정리하고 downstream 리포트를 모두 재생성하는 데는 이틀이 걸렸다.

교훈: 파이프라인의 write 연산은 항상 멱등하게 설계해야 한다. 가장 간단한 패턴은 파티션 단위 DELETE-then-INSERT(또는 REPLACE)이며, 이것을 "파티션 오버라이트"라 부른다. BigQuery에서는 `WRITE_TRUNCATE` 옵션, Spark에서는 `mode("overwrite")`와 파티션 지정의 조합으로 구현한다.

**함정 3: 로컬에서 잘 되는 코드를 그대로 프로덕션에 배포하기**

시나리오: 로컬에서 1만 건 샘플로 테스트한 pandas 코드가 프로덕션에서 5억 건을 처리할 때 OOM(Out of Memory)으로 죽는다. 백엔드에서 API를 개발할 때는 개별 요청의 데이터 크기가 작아서 메모리 문제가 드물지만, 데이터 파이프라인은 전체 데이터셋을 한꺼번에 처리해야 할 때가 많다. pandas의 `read_csv()`는 전체 파일을 메모리에 올리며, 10GB 파일은 메모리에서 30GB 이상을 차지할 수 있다(인덱스와 내부 표현 때문). 전문가는 처음부터 chunk 처리, 또는 분산 프레임워크(Spark, Dask)를 염두에 두고 설계한다.

## Rules of Thumb (실무 판단 기준)

**"파이프라인의 복잡도는 소스의 신뢰도에 반비례한다."** 소스 시스템이 안정적이고 스키마가 잘 관리되면 파이프라인은 단순해도 된다. 그러나 외부 API, 스크래핑, 사용자 업로드 파일처럼 신뢰할 수 없는 소스일수록 validation(검증), retry(재시도), dead letter queue(실패 데이터 격리) 등 방어 로직이 많아진다. 이 규칙이 깨지는 경우: 내부 시스템이라도 레거시 시스템은 외부 소스만큼이나 불안정할 수 있다. "우리 시스템이니까 안전하겠지"는 위험한 가정이다.

**"ETL vs ELT 선택은 팀의 SQL 역량으로 결정된다."** 팀에 SQL에 능숙한 분석가나 analytics engineer가 있다면 ELT가 유리하다. 변환 로직을 dbt 같은 도구로 SQL화할 수 있기 때문이다. 반대로 팀이 Python 중심이고 복잡한 비즈니스 로직(ML 전처리 등)이 많다면 ETL이 적합할 수 있다. 이 규칙이 깨지는 경우: 데이터 거버넌스가 엄격한 조직(금융, 의료)에서는 raw data를 그대로 웨어하우스에 넣는 ELT가 규정 위반일 수 있다. PII(Personally Identifiable Information, 개인식별정보) 마스킹이 적재 전에 이루어져야 한다면 ETL의 T 단계가 필수다.

**"Backfill 가능 여부가 파이프라인 성숙도의 가장 확실한 지표다."** 파이프라인에 날짜 파라미터를 넘겨서 과거 특정 날짜의 데이터를 재생성할 수 있으면 성숙한 파이프라인이다. 하드코딩된 `datetime.now()`로 "오늘" 데이터만 처리할 수 있는 파이프라인은 미성숙하다. 이 규칙이 깨지는 경우: 소스 시스템이 히스토리를 보관하지 않는 경우(예: 현재 재고 수량만 제공하는 API) backfill 자체가 불가능하다. 이런 소스에 대해서는 snapshot 패턴(매일 전체 상태를 스냅샷으로 저장)이 대안이다.

**"작은 파이프라인 여러 개가 큰 파이프라인 하나보다 낫다."** 하나의 거대한 DAG보다 작고 독립적인 파이프라인을 느슨하게 연결하는 것이 운영에 유리하다. 하나가 실패해도 나머지는 영향받지 않고, 각각 독립적으로 backfill할 수 있다. 이 규칙이 깨지는 경우: 파이프라인 간 데이터 정합성이 critical한 경우(예: 주문 데이터와 결제 데이터가 반드시 동일 시점 기준이어야 할 때) 분리하면 consistency(정합성) 문제가 생길 수 있다.

# Stage 2: Data Quality Management (데이터 품질 관리)

데이터 품질은 데이터 엔지니어링에서 가장 과소평가되면서 가장 큰 사고를 일으키는 영역이다. 코드 버그는 테스트로 잡지만, 데이터 품질 문제는 "데이터가 맞는 것처럼 보이지만 틀린" 상태로 오래 잠복한다.

## Practical Terminology and Jargon

**Schema Drift(스키마 드리프트)**: 소스 시스템의 테이블이나 API 응답의 구조가 시간에 따라 변하는 현상이다. 새 컬럼이 추가되거나, 기존 컬럼의 타입이 변경되거나, 컬럼이 제거된다. 실무에서 "드리프트 났다"고 하면 소스 스키마가 파이프라인의 기대와 달라져서 문제가 생겼다는 의미다. 이것은 버그가 아니라 자연 현상이며, 방어해야 할 대상이다.

**Data Contract(데이터 계약)**: 데이터 생산자(producer)와 소비자(consumer) 간의 명시적 합의다. "이 테이블은 이러한 컬럼을 이러한 타입으로 제공하며, 매일 오전 6시까지 업데이트된다"는 약속이다. API의 OpenAPI spec과 유사한 개념이다. 데이터 계약이 없으면 소스 팀이 마음대로 스키마를 바꿔도 할 말이 없다. 계약이 있으면 "계약 위반"이므로 소스 팀에 수정을 요구할 수 있다. 아직 업계 표준이 완전히 확립되지 않은 영역이지만, 성숙한 조직일수록 이를 도입하고 있다.

**Data Freshness(데이터 신선도)**: 데이터가 얼마나 최신인지를 나타낸다. "이 테이블의 freshness는 1시간"이라면 최대 1시간 전 데이터까지 반영되어 있다는 뜻이다. Freshness SLA를 어기면 의사결정자가 오래된 데이터로 결정을 내리게 된다.

**Observability(관측 가능성)**: 파이프라인과 데이터의 상태를 실시간으로 모니터링할 수 있는 능력이다. 단순히 "에러 났다"를 감지하는 것이 아니라, "데이터 볼륨이 평소의 40%밖에 안 된다", "null 비율이 갑자기 15%에서 80%로 올랐다"는 이상 징후를 잡아내는 것이다. Monte Carlo, Great Expectations, Elementary 같은 도구가 이 영역에 속한다.

**Lineage(리니지, 데이터 계보)**: 특정 데이터가 어디서 왔고, 어떤 변환을 거쳤고, 어디서 소비되는지를 추적하는 것이다. "이 매출 숫자가 왜 이상하지?"라는 질문에 답하려면, 최종 리포트에서 원본 소스까지 역추적할 수 있어야 한다. 이것이 lineage이며, dbt의 가장 강력한 기능 중 하나가 자동 lineage 생성이다.

## Hidden Preconditions

전문가가 당연하게 여기는 전제 중 가장 중요한 것은 **"데이터 품질은 파이프라인의 일부이지, 별도의 후속 작업이 아니다"**라는 인식이다. 초보자는 파이프라인을 먼저 만들고 나서 "품질 검증도 추가해야겠다"고 생각하지만, 전문가는 파이프라인의 각 단계에 검증 로직을 내장한다. 데이터 적재 후 row count 확인, 핵심 컬럼의 null 비율 확인, 비즈니스 규칙 위반 확인(예: 주문 금액이 음수인 레코드)이 파이프라인 코드의 일부로 존재한다.

또 하나의 숨겨진 전제는 **"데이터 품질 문제의 대부분은 기술적 문제가 아니라 커뮤니케이션 문제"**라는 것이다. 마케팅 팀이 CRM에 새 필드를 추가하면서 데이터 엔지니어링 팀에 알리지 않았고, 그 필드가 기존 필드의 의미를 미묘하게 바꿨다면, 이것은 코드로 잡을 수 없는 품질 문제다. 따라서 전문가는 데이터 계약, 정기적인 소스 팀과의 미팅, 변경 알림 체계를 기술적 솔루션만큼 중요하게 여긴다.

## Beginner's Traps

**함정 1: "테스트가 통과했으니 데이터가 정확하다"는 착각**

시나리오: 파이프라인의 단위 테스트가 모두 통과한다. 그러나 소스 시스템에서 특정 고객 세그먼트의 데이터가 12시간 지연 도착하고 있다는 사실은 단위 테스트로 잡을 수 없다. 결과적으로 일일 집계에서 해당 세그먼트의 매출이 항상 과소 보고된다. 이것은 코드의 정확성과 데이터의 정확성이 별개라는 것을 보여준다. 파이프라인 코드는 완벽하게 동작하지만, 입력 데이터 자체가 불완전하면 출력도 불완전하다. 전문가는 코드 테스트와 별도로 데이터 테스트(data tests)를 운영한다.

**함정 2: 스키마 변경을 에러로만 처리하기**

시나리오: 소스 테이블에 새 컬럼이 추가되었다. 파이프라인이 `SELECT *`를 사용하고 있어서 새 컬럼이 그대로 들어왔고, downstream 테이블의 스키마와 불일치로 에러가 발생했다. 급히 파이프라인을 고치느라 반나절을 허비했다. 반대 사례로, 명시적 컬럼 지정(`SELECT col_a, col_b, col_c`)을 사용하면 새 컬럼은 무시되어 에러가 나지 않지만, 중요한 새 컬럼을 놓칠 수 있다. 전문가의 접근: 명시적 컬럼 지정을 기본으로 하되, 소스 스키마를 주기적으로 감지(monitoring)해서 변경이 생기면 알림을 보내는 체계를 갖춘다. `SELECT *`는 탐색용 쿼리에서만 쓰고 프로덕션 파이프라인에서는 절대 쓰지 않는다.

**함정 3: 데이터 품질 규칙을 너무 엄격하게 설정하기**

시나리오: "null은 하나도 있으면 안 된다"는 규칙을 설정했더니, 정당하게 null인 경우(예: 아직 배송이 시작되지 않은 주문의 배송일)까지 알림이 울려서 팀 전체가 alert fatigue(알림 피로)에 빠졌다. 결국 모든 알림을 무시하게 되었고, 진짜 문제가 발생했을 때도 대응하지 못했다. 전문가는 "이 컬럼은 null이 0%여야 한다"와 "이 컬럼은 null이 30% 이하여야 한다"를 구분하고, 절대 규칙과 통계적 이상 탐지를 조합한다.

## Rules of Thumb

**"데이터 품질 검증의 80%는 세 가지로 커버된다: row count, null rate, freshness."** 정교한 비즈니스 규칙 검증보다 이 세 가지 기본 검증이 대부분의 심각한 문제를 잡는다. Row count가 전일 대비 50% 이상 변동했으면 뭔가 잘못된 것이다. 핵심 컬럼의 null 비율이 갑자기 급증했으면 소스에 문제가 있다. Freshness가 SLA를 넘겼으면 파이프라인이 멈춰있다. 이 규칙이 깨지는 경우: 데이터 볼륨이 정상이고 null도 없지만 값 자체가 틀린 "semantic error(의미적 오류)"는 이 세 가지로 잡을 수 없다. 예를 들어 환율 변환 오류로 매출 숫자가 두 배가 된 경우, 레코드 수도 null 비율도 정상이지만 값이 틀리다.

**"Data contract는 신뢰의 문제가 아니라 규모의 문제다."** 팀이 작을 때는 "옆자리에 물어보면 된다"로 충분하지만, 조직이 커지면 데이터 생산자와 소비자가 서로 모르는 상황이 된다. 데이터 계약은 팀 규모가 20명을 넘어갈 때 투자 대비 효과가 확실해진다. 이 규칙이 깨지는 경우: 작은 팀이라도 외부 데이터 소스를 많이 사용한다면 계약적 사고방식이 필요하다. 외부 소스는 팀 크기와 무관하게 통제 불가하기 때문이다.

# Stage 3: Data Storage Architecture (저장소 아키텍처)

데이터를 어디에, 어떤 형태로 저장하는가는 비용, 성능, 유연성의 균형이다. 백엔드 개발에서 PostgreSQL을 선택하는 것과는 차원이 다른 고민이 존재한다.

## Practical Terminology and Jargon

**Data Warehouse(데이터 웨어하우스)**: 분석 목적에 최적화된 저장소다. 행(row) 단위가 아닌 열(column) 단위로 데이터를 저장하는 columnar storage(컬럼형 저장)가 핵심 특성이다. `SELECT SUM(revenue) FROM sales WHERE year = 2024`처럼 특정 컬럼만 집계하는 쿼리가 압도적으로 빠르다. 그러나 `SELECT * FROM users WHERE id = 12345`처럼 특정 행을 찾는 OLTP(Online Transaction Processing) 쿼리에는 오히려 느리다. 실무에서 "웨어하우스에 넣자"고 하면, "이 데이터를 분석 가능한 형태로 정리해서 분석가들이 쓸 수 있게 하자"는 의미다. Snowflake, BigQuery, Redshift가 대표적이다.

**Data Lake(데이터 레이크)**: 구조화되지 않은 raw data를 원본 그대로 저장하는 저장소다. 주로 S3, GCS, ADLS 같은 object storage(오브젝트 스토리지) 위에 Parquet이나 ORC 같은 파일 포맷으로 저장한다. "일단 다 넣어두고 나중에 필요할 때 쓰자"는 철학이다. 비용이 웨어하우스보다 10~100배 저렴하지만, 관리하지 않으면 "data swamp(데이터 늪)"으로 전락한다. 메타데이터 없이, 소유자 없이, 문서 없이 쌓여만 가는 파일들의 묘지가 되는 것이다.

**Lakehouse(레이크하우스)**: Data Lake의 저렴함과 Data Warehouse의 관리 기능을 결합한 아키텍처다. Delta Lake, Apache Iceberg, Apache Hudi가 이를 가능하게 하는 open table format(오픈 테이블 포맷)이다. 이 포맷들은 object storage 위의 파일들에 ACID transaction(원자성 트랜잭션), schema enforcement(스키마 강제), time travel(시간 여행 — 과거 시점의 데이터 조회) 같은 웨어하우스 기능을 부여한다. 실무에서 "우리 lakehouse 도입하자"는 말은 "S3에 Parquet 파일 쌓는 대신 Iceberg나 Delta 테이블로 관리하자"는 의미다.

**Partitioning(파티셔닝)**: 대용량 테이블을 특정 컬럼 값(주로 날짜)에 따라 물리적으로 분할하는 것이다. `sales` 테이블을 `date` 컬럼으로 파티셔닝하면, `WHERE date = '2024-03-15'` 쿼리가 해당 날짜의 파티션만 읽으므로 비용과 속도가 극적으로 개선된다. 파티셔닝이 안 된 테이블에서 특정 날짜 데이터를 조회하면 전체 테이블을 스캔(full table scan)해야 하며, 이는 BigQuery에서 수백 달러의 비용을 발생시킬 수 있다.

**Parquet**: 컬럼형 파일 포맷으로, 데이터 레이크의 사실상 표준이다. CSV 대비 파일 크기가 1/5~1/10로 줄어들고, 특정 컬럼만 읽을 수 있어 쿼리 성능이 극적으로 좋아진다. 실무에서 "파케이로 저장해"는 거의 상식 수준의 지시다.

## Hidden Preconditions

전문가가 저장소를 선택할 때 가장 먼저 고려하는 것은 **"누가 이 데이터를 쓰는가"**이다. 분석가가 SQL로 ad-hoc 쿼리를 날린다면 웨어하우스가 적합하고, ML 엔지니어가 대용량 데이터를 Spark로 처리한다면 데이터 레이크가 적합하다. 기술적 성능보다 소비자의 도구와 워크플로가 결정을 좌우한다.

또한 **비용 모델의 근본적 차이**를 이해해야 한다. 웨어하우스는 컴퓨팅에 과금된다(쿼리할 때 비용). 데이터 레이크는 스토리지에 과금된다(저장할 때 비용). 따라서 "데이터를 많이 저장하지만 자주 쿼리하지 않는" 경우(로그 데이터, 감사 기록) 데이터 레이크가 압도적으로 저렴하고, "데이터양은 적지만 수백 명이 동시에 쿼리하는" 경우(비즈니스 KPI 테이블) 웨어하우스가 적합하다.

## Beginner's Traps

**함정 1: 모든 데이터를 웨어하우스에 넣으려 하기**

시나리오: 로그 데이터, 이미지 메타데이터, IoT 센서 데이터를 전부 BigQuery에 넣었다. 한 달 후 스토리지 비용 청구서가 예상의 10배다. BigQuery 스토리지 비용은 GB당 $0.02/월이고, S3는 GB당 $0.023/월로 비슷해 보이지만, BigQuery는 active storage 기준이며 90일 미접근 데이터도 여전히 비싸다. 또한 데이터 레이크에 저장하면 동일한 데이터를 여러 컴퓨팅 엔진(Spark, Presto, Athena)으로 처리할 수 있지만, 웨어하우스에 넣으면 해당 웨어하우스의 컴퓨팅에 종속된다.

**함정 2: 파티셔닝 없이 대용량 테이블을 만들기**

시나리오: 3년치 이벤트 로그 5TB를 파티셔닝 없이 적재했다. 분석가가 "어제 이벤트 수"를 세려고 쿼리를 날렸는데 5TB 전체를 스캔했다. BigQuery 기준으로 한 번의 쿼리에 $25 이상이 과금되었고, 이런 쿼리가 하루에 수십 번 실행된다. 파티셔닝을 했다면 하루치 데이터(약 5GB)만 스캔해서 $0.025만 들었을 것이다. 비용 차이가 1000배다.

## Rules of Thumb

**"뜨거운 데이터(hot data)는 웨어하우스, 차가운 데이터(cold data)는 레이크."** 최근 90일 이내 데이터처럼 빈번하게 쿼리되는 "뜨거운" 데이터는 웨어하우스의 빠른 성능이 필요하다. 90일 이상 된 과거 데이터나 원본 로그처럼 가끔만 접근하는 "차가운" 데이터는 레이크에 저렴하게 보관한다. 이 규칙이 깨지는 경우: 규제 요건으로 7년치 거래 데이터에 즉시 접근해야 하는 금융 도메인에서는 차가운 데이터도 웨어하우스에 유지해야 할 수 있다.

**"확신이 없으면 Parquet + object storage로 시작하라."** Parquet으로 S3/GCS에 저장하는 것은 거의 모든 시나리오에서 안전한 출발점이다. 나중에 웨어하우스에 넣든, Spark로 처리하든, Iceberg 테이블로 전환하든 가능하다. CSV로 시작하면 나중에 모든 것을 변환해야 한다. 이 규칙이 깨지는 경우: 실시간 쿼리 응답이 필요한 대시보드용 데이터는 처음부터 웨어하우스에 넣어야 한다.

**"Lakehouse는 데이터 레이크가 이미 있고 웨어하우스 기능이 필요할 때 도입하라."** 처음부터 lakehouse를 목표로 설계하는 것은 과잉 엔지니어링일 수 있다. 데이터 레이크에 관리 문제(중복 파일, 불일치 스키마, time travel 필요)가 체감될 때 Iceberg나 Delta를 도입하는 것이 자연스럽다. 이 규칙이 깨지는 경우: 그린필드(완전히 새로운) 프로젝트라면 처음부터 Iceberg로 시작하는 것이 이후 마이그레이션 비용을 줄일 수 있다. 특히 2024년 이후로는 Iceberg 생태계가 충분히 성숙했다.

# Stage 4: Data Modeling (차원 모델링과 분석 데이터 구조)

데이터 모델링은 데이터를 분석하기 좋은 구조로 조직하는 작업이다. 백엔드 개발에서의 DB 정규화와는 목적이 정반대다. OLTP에서는 중복을 최소화하지만, 분석용 모델에서는 의도적으로 중복을 도입해서 쿼리를 단순하고 빠르게 만든다.

## Practical Terminology and Jargon

**Dimensional Modeling(차원 모델링)**: Ralph Kimball이 정립한 분석 데이터 모델링 방법론으로, 비즈니스 프로세스를 Fact(팩트)와 Dimension(디멘션) 테이블로 분해한다. 30년이 지난 지금도 분석 데이터 모델링의 핵심 패러다임이다. 실무에서 "킴볼 모델"이라고 하면 이 방법론을 가리킨다.

**Fact Table(팩트 테이블)**: 비즈니스 이벤트의 측정값(measure)을 저장하는 테이블이다. "주문이 발생했다(event)"에서 "주문 금액, 주문 수량"이 측정값이다. 팩트 테이블의 한 행은 하나의 비즈니스 이벤트를 나타낸다. `fact_orders`, `fact_page_views`, `fact_payments`처럼 명명한다. 팩트 테이블은 대부분 매우 크다(수억~수십억 행). 핵심 특성: 팩트 테이블은 주로 숫자값(금액, 수량, 횟수)과 디멘션 테이블의 foreign key(외래 키)로 구성된다. 텍스트 설명이나 이름 같은 속성은 팩트에 넣지 않는다.

**Dimension Table(디멘션 테이블)**: 팩트의 "맥락"을 제공하는 테이블이다. "누가(고객), 무엇을(상품), 어디서(매장), 언제(시간)" 주문했는지를 설명한다. `dim_customers`, `dim_products`, `dim_stores`, `dim_date`가 전형적이다. 디멘션 테이블은 상대적으로 작고(수천~수백만 행), 텍스트 속성이 풍부하다.

**Star Schema(스타 스키마)**: 중앙에 팩트 테이블이 있고, 주변에 디멘션 테이블들이 별 모양으로 연결된 구조다. 실무에서 가장 널리 쓰이는 분석 모델이며, "스타 스키마로 만들자"는 말은 "킴볼 방식으로 팩트와 디멘션을 분리하자"와 거의 같은 의미다.

**SCD (Slowly Changing Dimension)**: 디멘션 데이터가 시간에 따라 변하는 것을 처리하는 패턴이다. 고객의 주소가 바뀌었을 때, Type 1은 그냥 덮어쓰고(과거 기록 소실), Type 2는 새 행을 추가하고 이전 행에 종료일을 기록하며(과거 기록 보존), Type 3는 이전 값과 현재 값을 별도 컬럼에 저장한다. 실무에서 SCD Type 2가 가장 많이 쓰이며, "SCD2로 처리해"라고 하면 히스토리를 보존하는 방식을 의미한다.

**Grain(그레인)**: 팩트 테이블에서 한 행이 나타내는 비즈니스 이벤트의 세부 수준이다. "주문 단위 grain"은 행 하나가 주문 하나이고, "주문-상품 단위 grain"은 행 하나가 주문 내 상품 하나이다. Grain을 명확히 정의하지 않으면 집계 결과가 완전히 달라진다. "매출 얼마야?"라는 질문에 주문 단위에서 합산하면 맞지만, 주문-상품 단위에서 주문 금액을 합산하면 상품 수만큼 중복 계산된다.

**Surrogate Key(대리 키)**: 비즈니스 의미가 없는 자동 생성 키(보통 정수 시퀀스)다. 소스 시스템의 natural key(자연 키, 예: 고객 번호)가 변경되거나 시스템 간 충돌할 수 있으므로, 웨어하우스에서는 별도의 surrogate key를 생성해서 사용한다.

## Hidden Preconditions

전문가가 당연하게 여기는 가장 중요한 전제는 **"분석 모델은 쿼리 성능과 이해 용이성을 위해 의도적으로 비정규화한다"**는 것이다. 3정규형에 익숙한 백엔드 개발자에게 이것은 직관에 반한다. OLTP에서 중복은 곧 불일치 위험이지만, 분석 환경에서는 파이프라인이 주기적으로 모델을 재생성하므로 불일치 위험이 낮고, JOIN을 줄이는 것이 쿼리 성능과 사용자 편의에 압도적으로 유리하다. 분석가가 10개 테이블을 JOIN해야 기본 질문에 답할 수 있다면, 그 모델은 실패한 것이다.

또 하나의 전제: **"모델링은 기술적 작업이 아니라 비즈니스 이해 작업이다."** 가장 어려운 부분은 SQL을 작성하는 것이 아니라 "우리 비즈니스에서 '주문'이란 정확히 무엇인가?"를 정의하는 것이다. 취소된 주문은 포함하는가? 부분 환불된 주문의 금액은 원래 금액인가, 환불 후 금액인가? 이런 비즈니스 정의를 명확히 하지 않으면 같은 "매출" 숫자를 두 팀이 다르게 계산하는 사태가 발생한다.

## Beginner's Traps

**함정 1: OLTP 모델을 그대로 분석에 쓰기**

시나리오: "이미 PostgreSQL에 잘 정규화된 DB가 있으니 그걸 그대로 분석에 쓰면 되지 않나?"라고 생각한다. 분석가가 "지난 달 카테고리별 매출 추이"를 보려고 orders, order_items, products, categories, payments, refunds 6개 테이블을 JOIN하는 쿼리를 작성해야 한다. 쿼리가 200줄이 되고, 작은 실수 하나로 매출이 3배로 집계된다(흔한 fan-out 문제). 분석가는 쿼리를 포기하고 Excel로 돌아간다. Dimensional model이었다면 `fact_orders`와 `dim_products` 두 테이블의 간단한 JOIN으로 충분했을 것이다.

**함정 2: Grain을 혼합한 팩트 테이블**

시나리오: `fact_orders` 테이블에 주문 수준 데이터(주문 금액, 주문일)와 주문-상품 수준 데이터(상품별 수량, 상품별 금액)를 한 테이블에 섞었다. 주문 금액을 합산하면 상품 수만큼 중복 계산되어 실제 매출의 3배가 나온다. 이 오류는 검증 없이는 발견하기 어렵다. 숫자가 "합리적"으로 보이기 때문이다(3배가 아니라 "좀 높다" 정도로 느껴진다). 전문가의 원칙: "하나의 팩트 테이블, 하나의 grain." 이것은 절대 규칙이며, 예외를 두는 것은 거의 항상 실수다.

**함정 3: SCD를 무시하고 덮어쓰기만 하기**

시나리오: 고객이 서울에서 부산으로 이사했다. 고객 테이블에서 주소를 덮어썼다. 이제 과거 분석에서 "서울 지역 매출"이 갑자기 줄어들었다. 서울에서 발생한 과거 주문이 부산 고객의 주문으로 바뀌었기 때문이다. SCD Type 2를 사용했다면 과거 주문은 서울 주소 레코드에, 최근 주문은 부산 주소 레코드에 연결되어 정확한 분석이 가능했을 것이다.

## Rules of Thumb

**"먼저 Grain을 정하고, 그 다음에 모든 것을 결정하라."** Grain이 결정되면 어떤 디멘션이 필요한지, 어떤 measure가 가능한지가 자연스럽게 따라온다. Grain이 모호한 상태에서 컬럼을 추가하기 시작하면 반드시 혼란이 온다. 이 규칙이 깨지는 경우: 비즈니스 요구사항이 급격히 변해서 grain을 바꿔야 할 때. 이 경우 새 팩트 테이블을 만드는 것이 기존 것을 수정하는 것보다 안전하다.

**"분석가가 2번 이상의 JOIN 없이 80%의 질문에 답할 수 있어야 한다."** 모델의 성공 기준은 쿼리 복잡도다. 분석가가 기본적인 비즈니스 질문에 답하기 위해 복잡한 JOIN을 작성해야 한다면, 모델을 더 비정규화해야 한다. 이 규칙이 깨지는 경우: 극도로 복잡한 다대다(many-to-many) 관계(예: 하나의 주문에 여러 프로모션이 적용된 경우)는 추가 JOIN이 불가피하다. 이 경우 bridge table(브릿지 테이블)이라는 중간 테이블을 활용한다.

**"SCD Type 2는 기본값이다. 덮어쓰기는 의식적 선택이어야 한다."** 새로운 디멘션 테이블을 설계할 때 "이 속성이 변할 수 있는가?"를 묻고, 변할 수 있다면 SCD Type 2를 기본으로 적용한다. 덮어쓰기(Type 1)는 "과거 분석이 불필요한 속성"에만 의식적으로 선택해야 한다. 이 규칙이 깨지는 경우: 변경 빈도가 극히 높은 속성(예: 실시간 재고 수량)에 SCD Type 2를 적용하면 디멘션 테이블이 팩트 테이블만큼 커져서 성능이 저하된다.

# Stage 5: Workflow Orchestration (워크플로 오케스트레이션)

오케스트레이션은 "무엇을, 언제, 어떤 순서로 실행할 것인가"를 관리하는 것이다. crontab으로 스크립트를 돌리는 것과는 차원이 다른 영역이다.

## Practical Terminology and Jargon

**Orchestrator(오케스트레이터)**: 파이프라인의 실행을 관리하는 시스템이다. Apache Airflow가 사실상의 업계 표준이며, Dagster와 Prefect가 차세대 대안으로 성장하고 있다. "오케스트레이터 뭐 쓰세요?"는 데이터 엔지니어끼리의 흔한 대화 시작 방식이다.

**Task/Operator**: Airflow에서 DAG를 구성하는 개별 작업 단위. `PythonOperator`는 Python 함수를 실행하고, `BigQueryOperator`는 BigQuery 쿼리를 실행한다. 실무에서 "태스크"는 파이프라인의 한 단계를 가리킨다.

**Schedule vs Trigger**: Schedule은 "매일 새벽 2시에 실행"처럼 시간 기반이고, Trigger는 "새 파일이 S3에 도착하면 실행"처럼 이벤트 기반이다. 실무에서 대부분의 배치 파이프라인은 schedule 기반이지만, 점점 더 많은 파이프라인이 event-driven(이벤트 기반)으로 전환되고 있다.

**Sensor**: Airflow에서 특정 조건을 기다리는 operator다. "S3에 파일이 도착할 때까지 기다린다", "상위 DAG가 완료될 때까지 기다린다" 등. Sensor를 남용하면 worker slot을 점유하면서 아무 일도 하지 않는 상태가 되어 전체 시스템 성능이 저하된다.

## Hidden Preconditions

전문가가 당연하게 아는 것: **"오케스트레이터는 실행 엔진이 아니라 지휘자"**라는 사실이다. Airflow는 데이터를 처리하지 않는다. "Spark를 실행해라", "BigQuery에 쿼리를 날려라"라고 지시할 뿐이다. 따라서 Airflow 자체의 성능을 튜닝하는 것은 큰 의미가 없고, 실제 처리를 수행하는 Spark, BigQuery 등의 성능이 중요하다. 초보자는 종종 Airflow worker의 리소스를 늘려서 파이프라인을 빠르게 하려 하지만, 이것은 지휘자에게 더 좋은 지휘봉을 주는 것과 같다.

또한 **"execution date(실행 날짜)는 실제 실행 시각이 아니다"**는 Airflow의 유명한 혼란 포인트다. Airflow에서 execution date가 `2024-03-15`인 DAG는 3월 15일에 실행되는 것이 아니라 3월 16일에 실행되어 3월 15일의 데이터를 처리한다. 이것은 "interval의 시작 시점"이라는 개념인데, 거의 모든 Airflow 입문자가 이 부분에서 하루치 데이터를 잘못 처리하는 실수를 한다. Airflow 2.2 이후 `data_interval_start`/`data_interval_end`라는 더 직관적인 변수가 도입되었다.

## Beginner's Traps

**함정 1: crontab으로 충분하다고 생각하기**

시나리오: "5개 스크립트를 순서대로 실행하면 되니까 crontab으로 시차를 두면 되지"라고 설계했다. 첫 번째 스크립트가 평소 10분 걸리는데 어느 날 45분 걸렸다. 두 번째 스크립트는 예정 시각에 실행되었지만 첫 번째의 결과가 아직 없어서 빈 데이터를 처리했다. 오케스트레이터를 사용했다면 "Task A가 성공한 후에만 Task B를 실행"이라는 의존성을 명시적으로 관리했을 것이다.

**함정 2: DAG에 비즈니스 로직을 넣기**

시나리오: Airflow DAG 파일 안에 데이터 변환 로직, SQL 쿼리, 데이터 검증 로직을 전부 작성했다. 500줄짜리 DAG 파일이 됐고, 테스트가 불가능하고, 한 줄 수정에도 전체 DAG를 재배포해야 한다. 전문가의 원칙: DAG 파일은 오케스트레이션 로직(무엇을 어떤 순서로 실행할지)만 담고, 실제 비즈니스 로직은 별도의 Python 패키지나 SQL 파일로 분리한다.

## Rules of Thumb

**"Airflow는 검증된 선택이지만, 새 프로젝트라면 Dagster를 진지하게 고려하라."** Airflow는 생태계와 인력풀이 압도적이지만, 로컬 개발 경험과 테스트 용이성에서 Dagster가 월등하다. 기존에 Airflow 인프라가 있으면 마이그레이션 비용이 크므로 유지하되, 처음 시작한다면 Dagster의 "software-defined assets" 패러다임이 더 현대적이다. 이 규칙이 깨지는 경우: 팀에 Airflow 경험자가 많고 Dagster 경험자가 없다면, 도구보다 팀 역량이 중요하다.

**"DAG의 task 수가 50개를 넘으면 분할을 고려하라."** 하나의 DAG가 너무 커지면 가시성이 떨어지고, 부분 재실행이 어려워진다. 논리적으로 독립적인 단위로 DAG를 분할하고, DAG 간 의존성은 sensor나 dataset trigger로 관리한다.

# Stage 6: Real-time Stream Processing (실시간 스트림 처리)

이 영역은 Scope에서 경량 수준으로 다루도록 지정되었으므로, 핵심 개념과 배치와의 차이점에 집중한다.

## Practical Terminology and Jargon

**Streaming(스트리밍)**: 데이터를 배치(한꺼번에)가 아닌 이벤트 단위로 실시간 처리하는 패턴이다. Kafka는 메시지 브로커(이벤트를 받아서 전달하는 중간자), Flink와 Spark Streaming은 스트림 처리 엔진(전달받은 이벤트를 실시간으로 변환/집계)이다.

**Event-driven Architecture**: 시스템 간 통신이 직접 호출(API call)이 아닌 이벤트 발행/구독(pub/sub)으로 이루어지는 아키텍처. 백엔드의 message queue(RabbitMQ, SQS)와 개념적으로 유사하지만, Kafka는 이벤트를 영구 저장하고 재처리할 수 있다는 점이 다르다.

**Exactly-once Semantics(정확히 한 번 처리)**: 메시지가 정확히 한 번만 처리되는 것을 보장하는 것이 스트림 처리에서 가장 어려운 문제 중 하나다. 네트워크 장애로 메시지 전송이 실패하면 재전송하는데, 원본이 이미 처리되었다면 중복 처리가 발생한다.

## Rules of Thumb

**"진짜 실시간이 필요한지 먼저 확인하라."** 대부분의 분석 유스케이스는 5분~1시간 지연을 허용한다. "실시간"을 요구하는 이해관계자에게 "10분 지연이면 괜찮은가?"를 물으면 대부분 "괜찮다"고 한다. 이 경우 micro-batch(수분 간격의 배치)가 스트리밍보다 훨씬 단순하고 안정적이다. 진짜 실시간이 필요한 경우: 사기 탐지, 실시간 추천, 알림 시스템. 이런 경우에만 스트리밍의 복잡도를 감수할 가치가 있다.

# Stage 7: Data Governance and Lineage (데이터 거버넌스와 계보)

## Practical Terminology and Jargon

**Data Governance(데이터 거버넌스)**: 데이터의 가용성, 무결성, 보안을 관리하는 전사적 체계다. "누가 어떤 데이터에 접근할 수 있는가", "개인정보는 어떻게 처리하는가", "데이터 정의의 소유자는 누구인가"를 정하는 것이다.

**Data Catalog(데이터 카탈로그)**: 조직의 모든 데이터 자산을 검색 가능한 형태로 정리한 메타데이터 저장소. "매출 데이터가 어디 있지?"라는 질문에 카탈로그에서 검색하면 해당 테이블, 소유자, 업데이트 주기, 컬럼 설명을 한눈에 볼 수 있다.

**PII (Personally Identifiable Information)**: 개인을 식별할 수 있는 정보(이름, 이메일, 전화번호 등). 파이프라인에서 PII 처리는 기술적 문제이자 법적 문제다. GDPR, 개인정보보호법 등에 의해 PII의 저장, 처리, 접근이 엄격하게 규제된다.

## Rules of Thumb

**"거버넌스는 처음에는 과잉으로 느껴지지만, 없으면 6개월 후에 반드시 후회한다."** 초기에 테이블 명명 규칙, 소유자 지정, 접근 권한 체계를 잡아두면 조직이 커져도 관리 가능하다. "나중에 정리하자"는 결코 오지 않는다.

# 종합: 백엔드 개발자를 위한 마인드셋 전환

백엔드에서 데이터 엔지니어링으로의 마인드셋 전환에서 가장 중요한 세 가지:

첫째, **"요청-응답"에서 "배치-파이프라인"으로.** API는 밀리초 단위로 응답하지만, 파이프라인은 분~시간 단위로 실행된다. 실패 시 복구 전략이 완전히 다르다.

둘째, **"내가 통제하는 데이터"에서 "남이 주는 데이터"로.** 백엔드에서는 자신이 스키마를 정의하지만, 데이터 엔지니어링에서는 소스 데이터의 변화에 방어적으로 대응해야 한다.

셋째, **"정확한 코드"에서 "정확한 데이터"로.** 코드가 버그 없이 실행되더라도 데이터가 틀릴 수 있다. 데이터의 정확성은 코드 테스트와 별개의 검증이 필요하다.
