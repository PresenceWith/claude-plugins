# 데이터 엔지니어링: 전문가의 암묵지 해부

## 서론: 데이터 엔지니어링의 본질

데이터 엔지니어링은 단순히 "데이터를 A에서 B로 옮기는 일"이 아니다. 그것은 조직 전체의 의사결정 기반을 설계하고 운영하는 일이다. 백엔드 개발자로서 3년간 Python과 SQL을 다뤄온 경험은 분명 도움이 되지만, 데이터 엔지니어링에는 근본적으로 다른 사고방식이 필요하다. API 서버에서는 요청 하나가 실패하면 재시도하거나 에러를 반환하면 그만이지만, 데이터 파이프라인에서는 하나의 실패가 하루 전체의 분석 데이터를 오염시킬 수 있다. 이 차이를 체화하는 것이 첫걸음이다.

이 문서는 데이터 엔지니어링의 전체 lifecycle을 따라가면서, 각 단계에서 전문가들이 "당연히 알겠지"라고 생각하며 설명하지 않는 지식을 발굴한다.

# 1단계: Data Pipeline Design (ETL/ELT 패턴, 실패 복구, 멱등성)

## 1.1 ETL과 ELT: 용어 뒤에 숨겨진 아키텍처 철학

### 실무 용어와 뉘앙스

ETL(Extract, Transform, Load)과 ELT(Extract, Load, Transform)은 단순히 Transform의 위치가 다른 것이 아니다. 이 두 패턴은 "누가 연산 비용을 부담하는가"와 "데이터의 진실(source of truth)을 어디에 둘 것인가"에 대한 근본적으로 다른 철학을 반영한다.

ETL은 전통적 패턴으로, 데이터를 추출한 뒤 별도의 처리 서버(혹은 클러스터)에서 변환하고, 깨끗하게 정리된 결과만 목적지에 적재한다. 이 패턴이 지배적이었던 이유는 과거에 storage(저장소)가 비쌌기 때문이다. Data warehouse(데이터 웨어하우스, 분석용 중앙 저장소) 라이선스 비용이 TB당 수천 달러였던 시절, 원본 데이터를 그대로 넣는 것은 낭비였다.

ELT는 클라우드 시대의 패러다임이다. Storage 비용이 극적으로 낮아지면서, "일단 원본 데이터를 모두 적재하고, 필요할 때 warehouse의 연산 능력으로 변환하자"는 접근이 가능해졌다. Snowflake, BigQuery, Databricks 같은 현대적 warehouse는 연산과 저장을 분리(compute-storage separation)했기 때문에, 데이터를 많이 저장해도 연산 비용은 실제 쿼리할 때만 발생한다.

실무에서 "우리는 ELT를 한다"라고 말할 때, 이것은 단순한 기술 선택이 아니라 "원본 데이터를 보존한다(raw data preservation)"는 데이터 거버넌스 원칙을 내포하고 있다. 팀 미팅에서 이 맥락을 이해하지 못하면, 왜 "이미 변환된 데이터만 저장하면 되지 않느냐"는 질문이 어색하게 들리는지 이해할 수 없다.

### 숨겨진 전제 조건

전문가들이 ETL/ELT를 논의할 때 언급하지 않는 전제가 있다. 바로 "추출 시점의 데이터 상태를 알아야 한다"는 것이다. 이것을 source system understanding(소스 시스템 이해)이라 하는데, 파이프라인을 설계하기 전에 반드시 다음을 파악해야 한다: 소스 데이터베이스의 트랜잭션 격리 수준은 무엇인가? 추출 중에 소스 데이터가 변경될 수 있는가? CDC(Change Data Capture, 변경 데이터 캡처)를 지원하는가, 아니면 full scan(전체 스캔)만 가능한가?

백엔드 개발자라면 API를 통해 데이터를 받아본 경험이 있을 것이다. 그러나 데이터 파이프라인에서의 "추출"은 API 호출과 근본적으로 다르다. API는 보통 현재 상태의 스냅샷을 반환하지만, 파이프라인은 "어제 자정부터 오늘 자정까지 변경된 데이터"처럼 시간 범위 기반으로 추출해야 하는 경우가 대부분이다. 이때 소스 시스템에 `updated_at` 컬럼이 있는지, 그 컬럼이 인덱싱되어 있는지, soft delete(논리 삭제)를 쓰는지 hard delete(물리 삭제)를 쓰는지에 따라 파이프라인 설계가 완전히 달라진다.

### 초보자의 함정

**함정 1: "Transform을 한 곳에서만 하면 된다"는 착각**

실무에서 순수한 ETL이나 순수한 ELT는 거의 존재하지 않는다. 대부분의 파이프라인은 hybrid(혼합형)다. 예를 들어, 소스에서 추출한 JSON 데이터의 nested structure(중첩 구조)를 평탄화(flatten)하는 것은 Load 전에 해야 할 수도 있고(warehouse가 중첩 JSON을 효율적으로 처리하지 못하는 경우), PII(Personally Identifiable Information, 개인식별정보) 마스킹은 반드시 Load 전에 해야 한다(원본 데이터에 개인정보가 포함된 상태로 warehouse에 들어가면 규정 위반). 한편 비즈니스 로직 기반 집계는 Load 후에 하는 것이 자연스럽다.

구체적 시나리오: 신입 데이터 엔지니어가 e-commerce 주문 데이터 파이프라인을 설계하면서 "ELT니까 변환 없이 모두 raw로 적재하자"고 결정했다. 그런데 주문 데이터에 고객 전화번호와 주소가 포함되어 있었다. GDPR이나 개인정보보호법 대상이라면, 마스킹 없이 적재하는 순간 컴플라이언스 위반이 된다. "ELT = 변환 없이 적재"라는 단순화가 만든 실수다.

**함정 2: 파이프라인을 "코드"로만 생각하는 것**

백엔드 개발자는 파이프라인을 Python 스크립트 하나로 생각하기 쉽다. 하지만 프로덕션 파이프라인은 코드 자체보다 그것을 둘러싼 인프라가 더 중요하다. 스케줄링, 모니터링, 알림, 재실행 메커니즘, 로깅, 메타데이터 관리가 파이프라인의 본체다. 코드는 그 안에서 실행되는 한 부분일 뿐이다.

### 판단 기준(Rules of Thumb)

**"데이터가 10GB 미만이고 변환 로직이 SQL로 표현 가능하면 ELT, 복잡한 비즈니스 로직이나 ML 전처리가 필요하면 ETL 요소를 추가하라."** 이 기준은 대부분의 상황에서 유효하지만, 레거시 시스템과의 통합이 필요한 경우에는 깨진다. 예를 들어 on-premise(온프레미스, 자체 서버) Oracle Data Warehouse를 쓰고 있고 클라우드 마이그레이션 계획이 없다면, storage 비용이 여전히 핵심 제약이므로 ETL이 합리적이다.

**"파이프라인의 첫 번째 버전은 가능한 한 단순하게 만들어라. Batch(일괄 처리)로 시작하고, 진짜로 실시간이 필요하다는 것이 증명된 후에만 streaming(스트리밍)으로 전환하라."** 이 원칙은 거의 항상 맞지만, fraud detection(사기 탐지)이나 실시간 추천처럼 latency(지연 시간) 자체가 비즈니스 요구사항인 경우에는 처음부터 streaming을 고려해야 한다.

## 1.2 Idempotency (멱등성): 파이프라인 신뢰성의 핵심

### 실무 용어와 뉘앙스

Idempotency(멱등성)는 "같은 파이프라인을 같은 입력으로 여러 번 실행해도 결과가 동일한 것"을 의미한다. 백엔드 개발에서 PUT 요청의 멱등성과 개념은 같지만, 데이터 파이프라인에서의 멱등성은 훨씬 더 미묘하고 달성하기 어렵다.

실무에서 이 개념은 주로 "rerunnable(재실행 가능한)"이라는 단어로 표현된다. "이 파이프라인 rerunnable해?"라고 물으면, "어제 실패한 파이프라인을 오늘 다시 돌려도 데이터가 중복되거나 누락되지 않느냐"는 질문이다.

### 숨겨진 전제 조건

멱등성을 구현하는 가장 보편적인 패턴은 "delete-and-reinsert"(삭제 후 재삽입) 또는 "MERGE/UPSERT"다. 그런데 이 패턴들이 제대로 동작하려면 반드시 partition key(파티션 키) 또는 natural key(자연 키)가 명확해야 한다. "이 데이터의 고유 식별자가 무엇인가"라는 질문에 답할 수 없으면 멱등성 구현은 불가능하다.

전문가들이 당연하게 전제하지만 초보자가 놓치는 것: 파이프라인의 시간 경계(time boundary)를 어떻게 정의하느냐가 멱등성의 핵심이다. "2024년 1월 15일 데이터를 처리한다"고 할 때, 이것이 UTC 기준인지 KST 기준인지, 데이터 생성 시각 기준인지 수신 시각 기준인지에 따라 결과가 완전히 달라진다. 이 "시간 경계"를 실무에서는 logical date 또는 execution date라 부르며, Airflow 같은 오케스트레이터에서 이 개념이 명시적으로 관리된다.

### 초보자의 함정

**함정: INSERT 기반 파이프라인의 재실행 재앙**

가장 흔한 실수는 파이프라인이 단순히 INSERT만 수행하도록 만드는 것이다. 예를 들어, 매일 주문 데이터를 소스에서 추출해 warehouse에 INSERT하는 파이프라인을 만들었다고 하자. 월요일 실행이 네트워크 문제로 절반만 성공했다. 화요일에 월요일 것을 다시 실행하면? 이미 들어간 절반이 중복 삽입된다. 이제 월요일 매출이 실제보다 50% 이상 높게 집계된다. 분석팀은 "매출이 갑자기 뛰었다"고 경영진에 보고하고, 나중에 데이터 오류임이 밝혀지면 데이터팀 전체의 신뢰가 무너진다.

올바른 접근은 "해당 날짜의 파티션을 통째로 교체(overwrite)"하거나, MERGE 문으로 "이미 존재하면 업데이트, 없으면 삽입"을 수행하는 것이다. 이것이 왜 처음부터 강조되지 않느냐면, 전문가들에게는 "파이프라인은 당연히 멱등하게 만든다"가 호흡처럼 자연스러운 전제이기 때문이다.

### 판단 기준

**"파이프라인의 target(대상) 테이블이 append-only(추가 전용)인지 overwrite(덮어쓰기)인지를 먼저 결정하라. Append-only는 이벤트 로그처럼 자연적으로 멱등한 데이터에만 사용하고, 그 외에는 partition overwrite나 MERGE를 기본으로 하라."** 이 규칙은 데이터 양이 적을 때는 잘 동작하지만, 하루 수십억 건의 이벤트 로그를 처리할 때는 MERGE의 비용이 너무 커서 "deduplication(중복 제거) 후 append"라는 중간 전략이 필요해진다.

## 1.3 Failure Recovery (실패 복구): 실패를 전제로 한 설계

### 실무 용어와 뉘앙스

데이터 파이프라인에서는 "실패하지 않는 파이프라인"이 아니라 "실패해도 안전한 파이프라인"을 목표로 한다. 이것을 fault tolerance(내결함성)라 하며, 구체적으로는 retry(재시도), backfill(과거 데이터 재처리), circuit breaker(회로 차단기) 패턴 등으로 구현된다.

Backfill은 데이터 엔지니어링에서 가장 빈번하게 사용되면서도 가장 과소평가되는 개념이다. "지난 3개월간의 데이터를 새로운 로직으로 다시 처리해야 합니다"라는 요청은 월 1회 이상 발생한다. 비즈니스 로직이 변경되거나, 버그가 발견되거나, 새로운 데이터 소스가 추가될 때마다 backfill이 필요하다. 파이프라인이 backfill을 지원하도록 설계되지 않았다면, 매번 임시방편(ad-hoc) 스크립트를 작성해야 하고, 이것은 데이터 정합성 사고의 주요 원인이 된다.

### 숨겨진 전제 조건

실패 복구 설계의 핵심 전제는 "파이프라인의 각 단계가 독립적으로 재실행 가능해야 한다"는 것이다. 이것은 곧 각 단계가 자신의 입력과 출력을 명확하게 정의하고 있어야 한다는 뜻이다. 전문가들은 이것을 "contract(계약)"이라 부른다. 단계 A의 출력 스키마가 단계 B의 입력 스키마와 일치해야 하며, 이 계약이 깨지면 파이프라인 전체가 실패한다.

또 하나의 숨겨진 전제: 실패 시 partial state(부분 상태)를 어떻게 처리할 것인가. 1000개의 파일 중 500개를 처리한 시점에 실패했다면, 재실행 시 500개부터 이어서 할 것인가(checkpoint/resume 패턴), 아니면 1000개를 처음부터 다시 할 것인가(full reprocessing 패턴). 전자는 효율적이지만 구현이 복잡하고, 후자는 단순하지만 멱등성이 보장되어야 한다.

### 초보자의 함정

**함정: retry만으로 충분하다는 착각**

네트워크 일시 장애 같은 transient error(일시적 오류)에는 retry가 효과적이다. 그러나 데이터 파이프라인의 실패 원인 대부분은 transient하지 않다. 소스 시스템의 스키마가 변경되었거나(schema drift), 데이터 볼륨이 예상을 초과했거나(OOM - Out of Memory), 비즈니스 로직의 edge case가 발생한 것이다. 이런 경우 retry는 같은 실패를 반복할 뿐이다.

구체적 시나리오: 매일 외부 API에서 거래 데이터를 가져오는 파이프라인을 만들었다. retry를 3회로 설정했다. 어느 날 API 응답의 JSON 구조가 바뀌었다(필드명이 `transaction_id`에서 `txn_id`로 변경). 파이프라인은 3번 retry한 뒤 최종 실패하고 알림을 보냈다. 그런데 알림을 확인하고 수정하기까지 3일이 걸렸다면, 3일치 데이터를 backfill해야 한다. 이 파이프라인이 backfill을 지원하도록 설계되어 있었는가? 대부분 그렇지 않다.

### 판단 기준

**"모든 파이프라인은 날짜 파라미터를 받아 특정 날짜의 데이터만 처리할 수 있도록 설계하라. 이것이 backfill의 기본 단위가 된다."** 이 원칙은 일별 batch 파이프라인에는 완벽하게 적용되지만, 시간별(hourly) 파이프라인이나 near-real-time 파이프라인에서는 시간 단위를 더 세밀하게 정의해야 하며, streaming 파이프라인에는 offset이나 checkpoint 기반의 다른 접근이 필요하다.

# 2단계: Data Quality Management (데이터 품질 관리)

## 2.1 데이터 품질: 보이지 않는 파이프라인의 진짜 가치

### 실무 용어와 뉘앙스

Data quality(데이터 품질)는 데이터 엔지니어링에서 가장 많은 시간을 소비하면서도 가장 인정받지 못하는 영역이다. 새로운 파이프라인을 만드는 것은 화려하지만, 기존 파이프라인의 데이터 품질을 유지하는 것은 눈에 보이지 않는 노동이다.

실무에서 data quality는 크게 네 가지 차원으로 구분된다: completeness(완전성 - 데이터가 빠짐없이 있는가), accuracy(정확성 - 값이 실제와 일치하는가), timeliness(적시성 - 데이터가 제때 도착하는가), consistency(일관성 - 서로 다른 시스템의 데이터가 모순 없는가).

이 중 초보자가 가장 놓치는 것은 timeliness다. 데이터가 정확하고 완전해도, 경영진이 아침 9시 회의에 필요한 리포트를 9시 30분에야 쓸 수 있다면, 그 데이터는 쓸모없다. SLA(Service Level Agreement, 서비스 수준 협약)를 데이터 파이프라인에 적용하는 것은 이 맥락에서 이해해야 한다.

### 숨겨진 전제 조건

데이터 품질 검증은 파이프라인의 "안에서" 이루어져야 한다. 파이프라인 완료 후 별도로 품질을 검사하는 것은 이미 늦은 것이다. 왜냐하면 품질이 나쁜 데이터가 이미 downstream(하류) 테이블들에 전파되었기 때문이다.

전문가들이 말하지 않는 전제: 데이터 품질 규칙은 "비즈니스와 함께" 정의해야 한다. 기술팀이 독자적으로 "NULL이면 안 된다", "음수면 안 된다" 같은 규칙을 만드는 것은 필요조건이지 충분조건이 아니다. "주문 금액이 100만 원을 초과하면 이상값인가?"라는 질문은 비즈니스 도메인 지식 없이는 답할 수 없다. 명품 쇼핑몰이라면 정상이고, 편의점 POS 데이터라면 이상값이다.

### Schema Drift (스키마 변경)

Schema drift는 소스 시스템의 데이터 구조가 예고 없이 변경되는 현상이다. 이것은 데이터 엔지니어의 일상적 고통이다. 소스 팀이 새 컬럼을 추가하거나, 기존 컬럼의 데이터 타입을 변경하거나, 컬럼명을 바꾸는 일이 빈번하다. 대부분의 경우 소스 팀은 "내부 변경이니 알릴 필요 없다"고 생각하지만, downstream 파이프라인은 그 변경으로 인해 깨진다.

이에 대한 대응으로 등장한 것이 Data Contract(데이터 계약)이다. 소스 시스템과 소비자(consumer) 사이에 "이 테이블의 스키마는 이렇게 유지된다"는 명시적 합의를 문서화하고, 스키마 변경 시 사전 통보와 마이그레이션 기간을 보장하는 것이다. 이것은 기술적 해결책이라기보다 조직적 프로세스에 가깝다.

### 초보자의 함정

**함정 1: 데이터 품질 검증을 "나중에" 추가하려는 것**

파이프라인을 먼저 만들고, 안정화된 후에 품질 검증을 추가하겠다는 계획은 거의 실현되지 않는다. 왜냐하면 "안정화"라는 상태가 오지 않기 때문이다. 새로운 요구사항, 소스 시스템 변경, 데이터 볼륨 증가가 끊임없이 발생하고, 품질 검증은 항상 "다음 스프린트"로 밀린다.

구체적 시나리오: 마케팅 데이터 파이프라인을 만들었다. 광고 플랫폼 API에서 일별 캠페인 성과를 가져온다. 품질 검증 없이 운영하던 중, 어느 날 API가 반환하는 금액 단위가 달러에서 센트로 바뀌었다(이런 일은 실제로 일어난다). 파이프라인은 에러 없이 정상 완료되었지만, 모든 광고비가 100배로 뛰어올랐다. 이것을 분석팀이 2주 뒤에 발견했다면, 2주간의 마케팅 의사결정이 잘못된 데이터에 기반한 것이 된다. "전일 대비 광고비 변동이 50%를 초과하면 알림"이라는 단순한 규칙 하나만 있었어도 당일에 발견할 수 있었다.

**함정 2: 과도한 품질 규칙으로 파이프라인을 마비시키는 것**

반대의 극단도 위험하다. 모든 가능한 이상 케이스에 대해 검증 규칙을 만들면, 사소한 데이터 변동에도 파이프라인이 멈춘다. 금요일 저녁에 "주문 건수가 평소보다 30% 낮다"는 알림이 왔는데, 실제로는 공휴일이어서 그런 것이었다면, 불필요한 대응에 시간을 쏟게 된다. Alert fatigue(알림 피로)는 실제로 중요한 문제를 놓치게 만드는 주요 원인이다.

### 판단 기준

**"모든 파이프라인에 최소 세 가지 품질 검증을 넣어라: row count(행 수) 변동, NULL 비율 변동, 그리고 핵심 비즈니스 메트릭의 합계 변동."** 이 규칙은 시작점으로 훌륭하지만, 이벤트 기반 데이터(예: 실시간 로그)처럼 자연적 변동이 큰 데이터에는 정적 임계값이 아닌 통계적 이상 탐지가 필요하다. 또한 주말/공휴일 효과를 반영하지 않으면 false positive(오탐)가 빈번해진다.

# 3단계: Data Storage Architecture (데이터 저장소 아키텍처)

## 3.1 Warehouse vs Lake vs Lakehouse: 저장소 선택의 본질

### 실무 용어와 뉘앙스

Data Warehouse(데이터 웨어하우스), Data Lake(데이터 레이크), Lakehouse(레이크하우스)는 단순히 "데이터를 어디에 저장하느냐"의 문제가 아니다. 각각은 데이터의 구조화 정도, 사용자 유형, 쿼리 패턴에 대한 서로 다른 가정을 반영한다.

Data Warehouse는 structured data(구조화된 데이터)를 위한 것이다. 스키마가 사전에 정의되고(schema-on-write), SQL로 질의하며, 비즈니스 분석가와 BI 도구가 주요 소비자다. Snowflake, BigQuery, Redshift가 대표적이다. Warehouse의 핵심 가치는 "쿼리 성능의 예측 가능성"이다. 잘 모델링된 warehouse에서는 대부분의 분석 쿼리가 수 초 내에 완료된다.

Data Lake는 원래 "모든 종류의 데이터를 원본 그대로 저장하는 곳"으로 시작했다. Schema-on-read(읽을 때 스키마를 적용)라는 철학이며, JSON, Parquet, Avro, 이미지, 비디오까지 무엇이든 저장할 수 있다. S3, GCS, ADLS 같은 object storage(오브젝트 스토리지)가 기반이다. 그러나 실무에서 Data Lake는 종종 "Data Swamp(데이터 늪)"이 된다. 누가 어떤 데이터를 어디에 넣었는지 아무도 모르고, 스키마 문서도 없고, 같은 데이터의 서로 다른 버전이 여기저기 흩어져 있는 상태. 이것은 기술의 문제가 아니라 거버넌스의 문제다.

Lakehouse는 이 두 세계를 통합하려는 시도다. Lake의 저장 비용 효율성과 유연성 위에 Warehouse의 ACID 트랜잭션과 스키마 관리를 올려놓는다. Delta Lake, Apache Iceberg, Apache Hudi가 이 패러다임의 핵심 기술이다. 이들은 object storage 위에 "테이블" 추상화를 제공하여, Parquet 파일의 집합을 마치 데이터베이스 테이블처럼 다룰 수 있게 한다.

### 숨겨진 전제 조건

저장소 선택의 숨겨진 변수는 "팀의 기술 역량"이다. Lakehouse를 운영하려면 Spark, Delta Lake/Iceberg, object storage 최적화에 대한 깊은 이해가 필요하다. 5명 이하의 소규모 데이터팀이라면, managed warehouse(Snowflake, BigQuery)가 거의 항상 더 나은 선택이다. 인프라 관리를 클라우드 벤더에게 맡기고, 팀은 비즈니스 가치 창출에 집중할 수 있기 때문이다.

또 하나의 숨겨진 전제: 저장소의 비용 모델을 이해해야 한다. BigQuery는 쿼리한 데이터 양에 따라 비용이 책정되고(scan-based pricing), Snowflake는 가동한 연산 자원의 시간에 따라 과금된다(compute-time pricing). 이 차이는 사용 패턴에 따라 10배 이상의 비용 차이를 만들 수 있다. ad-hoc 쿼리가 빈번한 환경에서 BigQuery의 scan-based pricing은 비용 폭탄의 원인이 된다(분석가가 `SELECT *`를 습관적으로 쓰면 매번 전체 테이블을 스캔).

### 초보자의 함정

**함정: "최신 기술이 최선이다"라는 사고**

Lakehouse가 최신 트렌드라고 해서 무조건 좋은 것은 아니다. 팀이 10명 미만이고, 데이터 볼륨이 TB 단위이며, 주요 사용자가 SQL을 쓰는 분석가라면, Snowflake 하나로 충분하다. Lakehouse를 도입하면서 Delta Lake 운영, Spark 클러스터 관리, 파일 compaction(압축), small file problem(소파일 문제) 해결 등 새로운 운영 부담을 떠안는 것은 과잉 엔지니어링(over-engineering)이다.

구체적 시나리오: 스타트업 데이터팀이 "확장성을 위해" 처음부터 Delta Lake 기반 Lakehouse를 구축했다. 그러나 전체 데이터가 500GB에 불과하고, 팀원 3명 중 Spark를 깊이 이해하는 사람이 없었다. small file problem으로 쿼리 성능이 나빠지고, Z-ordering이나 OPTIMIZE 같은 유지보수 작업을 제때 수행하지 못해, 오히려 단순한 PostgreSQL + dbt 조합보다 느리고 불안정한 시스템이 되었다.

### 판단 기준

**"데이터 볼륨이 10TB 미만이고 비정형 데이터(이미지, 비디오, 로그 원문) 처리 요구가 없다면, managed warehouse에서 시작하라. 10TB를 넘기거나 ML 워크로드가 중요해지면 Lakehouse 전환을 고려하라."** 이 기준은 비용 최적화 관점에서는 유효하지만, 규제 환경(금융, 의료)에서 데이터 소유권(data sovereignty)이 중요한 경우에는 managed service 대신 self-hosted를 선택해야 할 수 있으며, 이때는 Lakehouse가 오히려 합리적인 출발점이 될 수 있다.

## 3.2 File Format과 Partitioning: 눈에 보이지 않는 성능의 핵심

### 실무 용어와 뉘앙스

파일 포맷 선택은 "사소한 결정"처럼 보이지만, 쿼리 성능과 저장 비용에 10배 이상의 차이를 만든다. 실무에서 사용되는 주요 포맷은 Parquet(파케이), ORC, Avro, JSON이다.

Parquet은 columnar format(열 기반 포맷)으로, 분석 쿼리에 최적화되어 있다. "특정 열만 읽기(column pruning)"가 가능하기 때문에, 100개 컬럼 중 3개만 필요한 쿼리에서 나머지 97개 컬럼의 데이터를 아예 읽지 않는다. 또한 같은 열의 데이터는 유사한 값이 모여 있으므로 압축률이 높다. 실무에서 "분석용 데이터는 Parquet"은 거의 불문율이다.

Partitioning(파티셔닝)은 데이터를 논리적 단위로 나누어 저장하는 것이다. 가장 흔한 패턴은 날짜 기반 파티셔닝(예: `year=2024/month=01/day=15/`). 이렇게 하면 "2024년 1월 데이터만 조회"하는 쿼리가 해당 파티션의 파일만 읽으면 되므로, 전체 데이터를 스캔하는 것보다 수십~수백 배 빠르다. 이것을 partition pruning이라 한다.

### 숨겨진 전제 조건

파티셔닝 키 선택의 핵심 전제: "대부분의 쿼리가 어떤 조건으로 필터링하는가"를 알아야 한다. 날짜로 파티셔닝했는데 대부분의 쿼리가 고객 ID로 필터링한다면, 파티셔닝의 이점을 전혀 누리지 못한다. 전문가들은 query pattern(쿼리 패턴) 분석을 파티셔닝 결정의 전제로 당연하게 수행하지만, 이것을 명시적으로 언급하는 경우는 드물다.

### 초보자의 함정

**함정: 과도한 파티셔닝(over-partitioning)**

파티셔닝이 좋다고 해서 여러 컬럼으로 중첩 파티셔닝하면 오히려 성능이 나빠진다. 예를 들어 `year/month/day/hour/country/category`로 파티셔닝하면, 각 파티션에 들어가는 파일이 매우 작아진다(small file problem). 분산 시스템에서 작은 파일 수천 개를 읽는 것은 큰 파일 수 개를 읽는 것보다 훨씬 느리다. 파일 목록을 가져오는 metadata operation(메타데이터 연산) 자체가 병목이 되기 때문이다.

구체적 시나리오: IoT 센서 데이터를 `date/device_id`로 파티셔닝했다. 디바이스가 10만 개이므로 하루에 10만 개의 파티션 디렉토리가 생기고, 각 파티션에는 수 KB짜리 파일이 하나씩 들어간다. 한 달이면 300만 개의 디렉토리와 파일이 생겨, 단순한 `SELECT COUNT(*)`조차 몇 분이 걸리게 된다.

### 판단 기준

**"파티션 하나에 최소 128MB~1GB의 데이터가 들어가도록 파티셔닝 키를 선택하라."** 이 규칙은 object storage + Spark/Presto 환경에서 잘 적용되지만, BigQuery처럼 내부적으로 파일 관리를 자동화하는 warehouse에서는 덜 중요하다. BigQuery는 사용자가 파일 크기를 직접 관리할 필요가 없기 때문이다.

# 4단계: Data Modeling (분석용 데이터 모델링)

## 4.1 Dimensional Modeling: 분석의 뼈대

### 실무 용어와 뉘앙스

Dimensional modeling(차원 모델링)은 Ralph Kimball이 체계화한 분석용 데이터 모델링 방법론이다. 핵심 개념은 두 가지: Fact table(팩트 테이블)과 Dimension table(디멘션 테이블).

Fact table은 "무엇이 일어났는가"를 기록한다. 주문이 발생했다, 페이지가 조회되었다, 결제가 완료되었다 -- 이런 비즈니스 이벤트를 숫자(measure, 측정값)와 함께 기록하는 테이블이다. 일반적으로 매우 크다(수억~수십억 행).

Dimension table은 "그 이벤트의 맥락은 무엇인가"를 기록한다. 어떤 고객이, 어떤 제품을, 어떤 매장에서 -- 이런 "누가, 무엇을, 어디서"에 대한 설명 정보를 담는다. 일반적으로 상대적으로 작다(수천~수백만 행).

Star schema(스타 스키마)는 하나의 fact table을 중심에 놓고, 여러 dimension table이 이를 둘러싸는 구조다. 이름 그대로 별 모양이다. 이 구조가 분석에 최적인 이유는, JOIN이 항상 fact-dimension 간에만 발생하고(dimension-dimension JOIN은 거의 없다), 쿼리가 직관적이기 때문이다.

실무에서 "이 테이블은 fact야 dimension이야?"라는 질문은 매우 중요하다. 이것을 잘못 판단하면 모델 전체의 구조가 뒤틀린다. 판단 기준은 "이 테이블에 measure(합산/평균/카운트 가능한 숫자)가 있는가"이다. 주문 금액, 클릭 수, 체류 시간처럼 집계할 수 있는 숫자가 핵심이면 fact, 고객명, 제품 카테고리, 매장 위치처럼 "그룹 기준"이 핵심이면 dimension이다.

### 숨겨진 전제 조건

Dimensional modeling에 들어가기 전에 반드시 해야 하지만 아무도 말하지 않는 것: "비즈니스 프로세스를 먼저 정의하라." "주문 분석"과 "배송 분석"은 서로 다른 비즈니스 프로세스이며, 각각 별도의 fact table이 필요하다. 이것을 하나의 "주문-배송 통합 테이블"로 만들려는 것은 가장 흔한 모델링 실수다.

또 하나의 숨겨진 전제: grain(세분도, 분석의 최소 단위)을 먼저 결정해야 한다. "하나의 행(row)이 무엇을 나타내는가"를 명확하게 정의하는 것이다. 주문 fact table의 grain이 "주문 건"인지 "주문 내 상품 라인"인지에 따라 모든 것이 달라진다. grain이 "주문 건"이면 `order_amount`는 주문 총액이지만, grain이 "주문 라인"이면 `line_amount`는 개별 상품 금액이다. 이 구분이 모호하면 집계 시 이중 카운팅(double counting)이 발생한다.

### Slowly Changing Dimension (SCD): 차원의 시간 변화 처리

Dimension 데이터는 시간이 지나면 변한다. 고객이 주소를 바꾸거나, 제품이 카테고리를 이동하거나, 직원이 부서를 옮긴다. 이 변화를 어떻게 추적하느냐에 따라 SCD Type 1, 2, 3 등으로 분류한다.

SCD Type 1은 단순히 덮어쓴다. 고객 주소가 바뀌면 기존 레코드를 업데이트한다. 이전 주소의 기록은 사라진다. 과거 분석을 할 때 "이 고객이 주문 당시 어디에 살았는지"를 알 수 없다는 단점이 있지만, 구현이 가장 단순하다.

SCD Type 2는 새로운 행을 추가한다. 주소가 바뀌면 기존 행의 `valid_to` 날짜를 채우고, 새 주소로 새 행을 추가한다. 이렇게 하면 "2023년 6월 시점의 고객 주소"를 정확히 조회할 수 있다. 대부분의 프로덕션 환경에서 핵심 dimension은 SCD Type 2로 관리된다.

### 초보자의 함정

**함정 1: OLTP(운영 시스템)의 정규화를 분석 모델에 그대로 적용하는 것**

백엔드 개발자에게 데이터베이스 정규화(normalization)는 자연스러운 습관이다. 3NF(제3정규형)로 테이블을 분리하고, 중복을 제거하며, 외래 키로 연결한다. 그러나 분석용 모델에서 과도한 정규화는 해악이다.

구체적 시나리오: 제품 dimension을 설계하면서, "카테고리" 정보를 별도 테이블로 분리했다(product -> sub_category -> category -> department). OLTP에서는 올바른 설계이지만, 분석가가 "부서별 매출"을 보려면 3번의 JOIN이 필요하다. Warehouse에서 JOIN은 비용이 크며, 분석가는 SQL 전문가가 아닌 경우가 많다. 차라리 product dimension에 category_name, department_name을 비정규화(denormalization)하여 포함시키는 것이 분석 환경에서는 훨씬 실용적이다.

**함정 2: Grain을 혼합하는 것**

하나의 fact table에 서로 다른 grain의 데이터를 넣는 것은 가장 위험한 모델링 실수다. 예를 들어, 일별 매출 집계와 건별 주문 데이터를 같은 테이블에 넣으면, `SUM(amount)`을 할 때 일별 집계와 건별 금액이 섞여서 의미 없는 숫자가 나온다.

구체적 시나리오: "주문 요약" 테이블을 만들면서, 일부 행은 개별 주문(grain = 주문 1건)이고 일부 행은 일별 총계(grain = 하루)인 데이터를 넣었다. 분석가가 `SELECT SUM(amount) FROM order_summary WHERE date = '2024-01-15'`를 실행하면, 개별 주문 금액들의 합과 일별 총계 행의 금액이 모두 더해져서 실제 매출의 약 2배가 나온다. 이런 오류는 데이터가 "틀려 보이지 않기" 때문에 오랫동안 발견되지 않는다.

### 판단 기준

**"모든 fact table의 첫 번째 설계 결정은 grain이다. grain이 결정되기 전에는 컬럼을 추가하지 마라."** 이 원칙은 거의 예외 없이 적용된다. 다만, accumulating snapshot fact(누적 스냅샷 팩트)처럼 하나의 비즈니스 프로세스가 여러 이정표(milestones)를 거치는 경우(예: 주문 접수 -> 결제 완료 -> 배송 시작 -> 배송 완료), grain은 "하나의 프로세스 인스턴스"가 되며, 이 경우 같은 행이 시간에 따라 업데이트되는 것은 정상이다.

**"dimension에서 비정규화를 두려워하지 마라. 분석 환경에서 JOIN 비용은 storage 비용보다 비싸다."** 이 규칙은 현대 warehouse에서는 거의 항상 맞지만, dimension이 매우 자주 변경되는 경우(SCD Type 2)에는 비정규화된 필드의 동기화가 복잡해질 수 있다. 이때는 view(뷰)를 활용하여 비정규화를 물리적으로 하지 않고 논리적으로만 구현하는 것이 대안이다.

# 5단계: Workflow Orchestration (워크플로우 오케스트레이션)

## 5.1 오케스트레이터의 역할: 파이프라인의 두뇌

### 실무 용어와 뉘앙스

Orchestrator(오케스트레이터)는 "어떤 작업을, 어떤 순서로, 언제 실행할 것인가"를 관리하는 시스템이다. Airflow가 사실상 표준(de facto standard)이며, Dagster와 Prefect이 차세대 대안으로 부상하고 있다.

실무에서 가장 중요한 개념은 DAG(Directed Acyclic Graph, 방향성 비순환 그래프)이다. 파이프라인의 각 단계를 task(태스크)로 정의하고, task 간의 의존성을 방향성 그래프로 표현한다. "A가 완료된 후에 B를 실행하고, B와 C가 모두 완료된 후에 D를 실행한다"는 식이다.

Airflow에서 혼란을 주는 개념 중 하나가 execution_date(실행 날짜)다. 이것은 "파이프라인이 실제로 실행된 시각"이 아니라 "처리 대상 데이터의 논리적 날짜"다. 예를 들어, 매일 자정에 실행되는 파이프라인의 1월 15일 execution_date는 1월 15일 00:00이지만, 실제 실행은 1월 16일 00:00에 시작된다(전일 데이터가 모두 쌓인 후 처리해야 하므로). 이 "하루 지연" 개념을 이해하지 못하면 Airflow의 동작이 도무지 논리적으로 맞지 않는 것처럼 보인다.

### 숨겨진 전제 조건

오케스트레이터를 사용하기 전에 전문가들이 당연하게 수행하는 것: 파이프라인 간의 의존성 맵(dependency map)을 그리는 것이다. "A 파이프라인의 출력이 B 파이프라인의 입력"이라는 관계를 시각화해야 한다. 이 맵 없이 파이프라인을 하나씩 만들면, 어느 순간 순환 의존성(circular dependency)이 생기거나, 특정 파이프라인의 실패가 예상치 못한 곳까지 연쇄적으로 영향을 미치게 된다.

### 판단 기준

**"10개 이하의 간단한 cron job이면 오케스트레이터 없이 시작해도 된다. 하지만 task 간 의존성이 존재하거나 backfill이 필요한 순간 오케스트레이터를 도입하라."** 이 시점을 놓치면 "cron job 지옥"에 빠지게 된다 -- 수십 개의 cron job이 서로를 sleep으로 기다리고, 하나가 지연되면 연쇄적으로 실패하며, backfill은 수동으로만 가능한 상태.

# 6단계: Real-time Stream Processing (실시간 스트림 처리)

## 6.1 Batch와 Streaming의 경계

### 실무 용어와 뉘앙스

Batch processing(일괄 처리)은 "모아서 한꺼번에" 처리하고, stream processing(스트림 처리)은 "발생하는 즉시" 처리한다. 실무에서는 이 둘 사이에 micro-batch(마이크로 배치)라는 중간 영역이 있다. Spark Streaming이 대표적인 micro-batch 시스템으로, 수 초~수 분 단위로 작은 batch를 반복 실행한다.

Kafka(카프카)는 "메시지 브로커"로 소개되지만, 데이터 엔지니어링 맥락에서는 "분산 이벤트 로그"로 이해하는 것이 더 정확하다. 카프카의 핵심은 메시지가 소비된 후에도 사라지지 않는다는 것이다(retention 설정에 따라). 이것이 전통적 메시지 큐(RabbitMQ 등)와의 결정적 차이이며, 이 덕분에 카프카는 "데이터의 진실의 원천(source of truth)"으로도 활용될 수 있다.

### 숨겨진 전제 조건

Streaming을 도입하기 전의 핵심 질문: "정말 실시간이어야 하는가?" 실무에서 "실시간"이라고 요청하는 것의 90%는 "15분 이내"면 충분하다. 그리고 "15분 이내"는 잘 최적화된 micro-batch로 달성할 수 있다. 진정한 sub-second latency(1초 미만 지연)가 필요한 use case(사기 탐지, 실시간 입찰 등)는 전체의 극히 일부다.

### 판단 기준

**"Streaming은 복잡성을 10배 늘린다. 운영 부담(장애 대응, 모니터링, 스케일링)도 10배다. 분 단위 지연이 허용되면 micro-batch, 초 단위 지연이 허용되면 간단한 Kafka consumer, sub-second가 필요할 때만 Flink 같은 전용 프레임워크를 고려하라."** 이 원칙은 기술적으로 유효하지만, 조직 정치의 맥락에서는 깨질 수 있다. "실시간 대시보드"가 경영진의 핵심 요구사항이라면, 기술적 합리성보다 비즈니스 요구가 우선하는 경우가 있다.

# 7단계: Data Governance and Lineage (데이터 거버넌스와 계보 추적)

## 7.1 거버넌스: 기술이 아닌 문화

### 실무 용어와 뉘앙스

Data governance(데이터 거버넌스)는 "데이터의 소유권, 품질, 접근 권한, 생명주기를 조직적으로 관리하는 체계"다. 기술적 도구보다 프로세스와 문화가 더 중요하다.

Data lineage(데이터 계보)는 "이 데이터가 어디에서 왔고, 어떤 변환을 거쳤으며, 어디로 갔는가"를 추적하는 것이다. 분석 리포트의 숫자가 이상할 때, lineage를 따라가면 어느 단계에서 문제가 발생했는지 추적할 수 있다. dbt의 lineage graph, Apache Atlas, DataHub 같은 도구가 이 영역을 지원한다.

Data catalog(데이터 카탈로그)는 "어떤 데이터가 어디에 있고, 무엇을 의미하는지"를 검색할 수 있는 시스템이다. 이것이 없으면 분석가가 "고객별 월 매출 데이터가 어느 테이블에 있나요?"라는 질문을 매번 데이터 엔지니어에게 해야 하며, 이것은 팀의 스케일을 심각하게 제한한다.

### 숨겨진 전제 조건

거버넌스의 전제는 data ownership(데이터 소유권)이 명확해야 한다는 것이다. 모든 테이블에는 "이 테이블의 정의, 품질, 접근 권한에 대해 책임지는 사람/팀"이 있어야 한다. 소유권이 불명확한 테이블은 아무도 유지보수하지 않고, 결국 신뢰할 수 없는 데이터가 된다.

### 판단 기준

**"거버넌스 도구를 도입하기 전에, 데이터 관련 의사결정 프로세스를 먼저 정의하라. 도구는 프로세스를 자동화할 뿐이지, 프로세스를 만들어주지 않는다."** 이 원칙은 거의 항상 맞지만, 반대로 도구 없이 프로세스만으로는 일정 규모 이상에서 지속 불가능하다. 테이블이 1000개를 넘으면 스프레드시트 기반 카탈로그는 유지보수가 불가능해진다.

# 8단계: End-to-End Workflow (전체 흐름의 통합)

## 8.1 실무에서의 데이터 파이프라인 전체 흐름

백엔드 개발자로서 이해하기 쉬운 비유를 쓰자면, 데이터 파이프라인의 전체 흐름은 CI/CD 파이프라인과 구조적으로 유사하다. 소스 코드가 변경되면(데이터가 생성되면), 빌드가 트리거되고(추출, 변환이 실행되고), 테스트를 통과하면(품질 검증을 통과하면), 배포된다(분석 가능 상태가 된다).

전체 흐름은 다음과 같다:

**Ingestion(수집) 단계**: 소스 시스템(OLTP 데이터베이스, API, 로그 파일, SaaS 도구)에서 데이터를 추출한다. 이때 CDC, API polling, file drop 등 다양한 메커니즘이 사용된다. 이 단계의 핵심은 "소스 시스템에 최소한의 부하만 줄 것"이다. 프로덕션 DB에 무거운 쿼리를 날려서 서비스에 영향을 주는 것은 절대 금기다. 이를 위해 read replica(읽기 전용 복제본)를 사용하거나, CDC로 변경 이벤트만 캡처한다.

**Staging(스테이징) 단계**: 추출한 원본 데이터를 그대로 저장한다. 이것을 raw layer 또는 landing zone이라 부른다. 이 단계에서는 어떤 변환도 하지 않는다(PII 마스킹 제외). 원본 보존이 목적이며, 나중에 비즈니스 로직이 바뀌어도 원본에서 다시 시작할 수 있어야 한다.

**Transformation(변환) 단계**: dbt(data build tool)가 이 영역의 표준으로 자리잡았다. SQL 기반으로 데이터를 변환하며, version control, testing, documentation이 내장되어 있다. 변환은 보통 여러 레이어로 구성된다: staging(소스별 최소 정리), intermediate(비즈니스 로직 적용), mart(최종 분석용 테이블).

**Serving(서빙) 단계**: 최종 테이블을 BI 도구(Tableau, Looker, Metabase), 분석가의 SQL 쿼리, ML 모델의 feature store 등에 제공한다. 이 단계에서는 쿼리 성능 최적화(인덱싱, materialized view, 캐싱)가 핵심이다.

### 숨겨진 전제 조건

전문가들이 당연하게 수행하지만 초보자가 놓치는 것: 각 레이어 간의 명명 규칙(naming convention)이다. `raw_`, `stg_`, `int_`, `fct_`, `dim_` 같은 접두사로 테이블의 레이어를 명시한다. 이것이 없으면 수백 개의 테이블 중 어떤 것이 최종 분석용이고 어떤 것이 중간 단계인지 구분할 수 없다.

또 하나: 환경 분리(environment separation). 개발(dev), 스테이징(staging), 프로덕션(prod) 환경을 분리하는 것은 백엔드에서는 당연하지만, 데이터 파이프라인에서는 종종 간과된다. 개발 중인 변환 로직이 프로덕션 테이블을 덮어쓰는 사고는 놀라울 정도로 자주 발생한다.

### 초보자의 함정

**함정: "한 방에" 전체 파이프라인을 만들려는 것**

소스에서 최종 mart 테이블까지 하나의 거대한 쿼리나 스크립트로 만들려는 시도는 실패한다. 중간 단계가 없으면 디버깅이 불가능하고, 일부만 재실행할 수 없으며, 다른 파이프라인과 중간 결과를 공유할 수 없다.

구체적 시나리오: 주문 분석 파이프라인을 설계하면서, 소스 DB에서 직접 최종 집계 테이블을 만드는 200줄짜리 SQL을 작성했다. 몇 주 후, 배송 분석 파이프라인에서도 같은 주문 데이터가 필요해졌다. 그런데 기존 SQL의 중간 단계(주문 정리, 주소 정규화 등)를 재사용할 수 없어서, 같은 로직을 복사-붙여넣기했다. 이제 주문 데이터 정리 로직이 두 곳에 존재하고, 하나를 수정하면 다른 하나와 불일치가 발생한다. 이것이 레이어 분리(raw -> staging -> intermediate -> mart)의 실질적 이유다.

# 백엔드 개발자를 위한 브릿지: 기존 지식과의 연결

백엔드 개발 경험에서 데이터 엔지니어링으로 자연스럽게 연결되는 개념들이 있다:

**API Rate Limiting과 Source System Protection**: API를 설계할 때 rate limiting을 적용하듯, 소스 DB에서 데이터를 추출할 때도 부하 제한이 필수다. 백엔드에서 써본 backoff 전략이 여기서도 동일하게 적용된다.

**CI/CD와 Data Pipeline Orchestration**: Jenkins/GitHub Actions에서 stage를 정의하고 의존성을 설정하듯, Airflow에서 task를 정의하고 의존성을 설정한다. "빌드 실패 시 다음 단계 실행하지 않음"은 "task 실패 시 downstream task 스킵"과 같은 패턴이다.

**Database Migration과 Schema Evolution**: Django/Rails의 마이그레이션 도구가 스키마 변경을 버전 관리하듯, dbt가 분석 모델의 변경을 버전 관리한다. ALTER TABLE 대신 dbt의 모델 파일을 수정하고, git으로 관리한다.

**Monitoring과 Data Observability**: APM(Application Performance Monitoring)이 API 응답 시간과 에러율을 추적하듯, Data Observability 도구가 데이터 freshness(최신성), volume(볼륨), schema(스키마) 변화를 추적한다. 개념은 같고 관찰 대상만 다르다.

# 디자인 리뷰 참여를 위한 핵심 질문들

데이터 인프라 디자인 리뷰에서 유의미한 질문을 하기 위해 알아야 할 프레임워크:

**파이프라인 설계 리뷰 시**: "이 파이프라인의 grain은 무엇인가?", "멱등하게 재실행 가능한가?", "backfill은 어떻게 지원되는가?", "실패 시 partial state는 어떻게 처리되는가?"

**저장소 선택 리뷰 시**: "예상 데이터 볼륨과 성장률은?", "주요 쿼리 패턴은 무엇인가?", "파티셔닝 전략은?", "비용 모델(compute vs storage)의 균형은?"

**데이터 품질 리뷰 시**: "핵심 품질 메트릭은 무엇인가?", "이상 탐지 임계값은 어떻게 설정했는가?", "data contract은 소스 팀과 합의되었는가?"

**모델링 리뷰 시**: "fact과 dimension의 구분이 명확한가?", "grain이 혼합되어 있지 않은가?", "SCD 전략은 무엇인가?", "비정규화의 근거는?"

이 질문들을 적절한 맥락에서 던질 수 있다면, 데이터 엔지니어링 팀과의 협업에서 "이 사람은 기본을 이해하고 있다"는 인상을 줄 수 있다. 기술적 깊이보다 올바른 질문을 하는 능력이 협업에서는 더 가치 있다.
